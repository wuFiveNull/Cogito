# 部署、配置、测试与架构治理

> **文档编号**：OPS-GOVERNANCE  
> **适用范围**：进程部署、启动关闭、配置、Profile、测试、运维、恢复、架构约束和验收  
> **权威边界**：本文是其范围内的规范性来源；总体架构文档只负责概念与边界。  
> **细化规范**：配置、数据库、本地运维和测试发布分别以 `CONFIG-PROFILES`、`DATABASE-SCHEMA`、`LOCAL-OPERATIONS`、`TEST-EVALUATION` 为准。  
> **关联文档**：RUNTIME-FLOWS, STORAGE-DATA, SECURITY-OBS, CAPABILITY-PLUGINS

## 阅读说明

**目的**：定义运行组件、代码组织、配置、测试、运维、恢复和架构验收。

**边界**：业务字段和模块算法由其他专题文档定义。

**建议读取方式**：部署、发布、重构、配置变更或架构审查时读取。

## 1. 运行时与部署架构

### 1.1 推荐运行组件

```text
channel-gateway
agent-api
agent-worker
interaction-web
```

#### 1.1.1 channel-gateway

职责：

- 维护 Channel 连接；
- 接收平台消息和事件；
- 进行平台对象与内部契约转换；
- 调用 Agent API；
- 执行 Delivery 的平台发送；
- 上报平台发送结果和平台消息 ID。

#### 1.1.2 agent-api

职责：

- 接收入站消息、Command 和 Query；
- 执行低延迟 Turn；
- 管理 Context Partition Lane；
- 将持久工作写入 Task/Outbox；
- 提供 Interaction API。

#### 1.1.3 agent-worker

职责：

- 获取 Task Lease；
- 执行 Connector、Event Consumer、Delivery、Memory Maintenance 等后台工作；
- 执行重试、恢复和延迟任务；
- 清理过期数据和回收 Payload。

#### 1.1.4 interaction-web

职责：

- Dashboard 和 Web Chat UI；
- 使用 Query/Command/Stream API；
- 不直接连接 SQLite 进行业务写入。

### 1.2 可合并部署

在资源有限时可以合并进程：

```text
personal-agent process
├── Agent API
├── Worker
├── Scheduler
└── Interaction Web

channel-gateway process
└── LangBot / adapters
```

也可以全部在一个进程内运行，但必须保留以下逻辑边界：

- Channel Adapter 不调用 Agent 内部类；
- Worker 仍通过 Task Repository 和 Lease 获取工作；
- Dashboard 仍通过 Command Handler 改状态；
- Event Handler 仍采用 Outbox 消费语义。

### 1.3 进程通信

#### 1.3.1 Gateway → Agent Core

推荐使用只监听 loopback 的本机 HTTP 或受本机文件权限保护的 Unix Domain Socket：

```text
POST /internal/v1/inbound-messages
POST /internal/v1/delivery-results
GET  /internal/v1/health
```

需要流式回复时使用 WebSocket、SSE 或双向本地 RPC，但所有平台 placeholder、编辑和最终发送仍必须持久化为 Delivery Attempt，详见 `STREAMING-DELIVERY`。

#### 1.3.2 Agent Core → Gateway

推荐两种方式：

```text
方式 A：Gateway 拉取 Delivery Queue
方式 B：Agent Worker 调 Gateway Send API
```

无论采用哪种方式，Delivery 表都是状态事实源，Gateway 调用只是 Attempt。

#### 1.3.3 Interaction → Agent Core

```text
Query API    读取 Projection
Command API  创建持久 Command/执行应用服务
Stream API   推送 Trace、Task、Delivery 和状态变化
```

### 1.4 进程所有权

必须避免两个进程同时“拥有”同一工作：

```text
Context Partition Lane   agent-api 的 Lane Coordinator
Task                      持有有效 Lease 的 worker
Delivery                  持有有效 Delivery Lease 的 worker/gateway
Connector Poll            持有 Connector Lease 的 worker
Scheduled Fire            Scheduler 通过唯一触发键创建 Task
```

### 1.5 启动顺序

建议顺序：

```text
1. 加载最小配置和 Secret Store
2. 打开 SQLite，验证版本和 WAL
3. 执行或验证 Migration
4. 初始化 Repository 与 Payload Store
5. 加载内置 Policy 和 Provider
6. 验证插件并加载允许的插件
7. 启动 Recovery Scan
8. 启动 Worker、Scheduler 和 Event Consumer
9. 启动 Agent API
10. 启动 Channel Gateway
11. 启动 Interaction Web
```

Gateway 在 Agent API 未就绪时应缓存或拒绝接收，不应静默丢弃平台消息。

### 1.6 优雅关闭

关闭流程：

```text
1. 停止接收新的普通工作
2. 允许取消/控制命令继续进入短时间窗口
3. 停止 Scheduler 创建新 Task
4. Worker 停止获取新 Lease
5. 当前 RunAttempt 到达安全点并写 Checkpoint
6. 刷新 Outbox 状态和 Trace Buffer
7. 释放 Lease 或缩短过期时间
8. 关闭数据库和网络连接
```

不可中断的外部副作用必须在关闭前记录 `unknown` 或 Receipt，恢复时执行 reconcile。

### 1.7 健康检查

每个进程提供：

```text
liveness   进程是否运行
readiness  是否可以接收新工作
dependency SQLite、Payload、Provider、Gateway 是否可用
backlog    Queue、Outbox、Delivery 是否堆积
```

健康状态不能只返回 HTTP 200；必须包含降级原因。

### 1.8 故障隔离

```text
Channel Gateway 失败   不影响数据库中 Task 和 Memory
Interaction 失败       不影响 Agent 和 Worker
Model Provider 失败    Turn 降级或失败，不损坏状态
Plugin 失败            禁用插件或单次调用失败
Payload Export 失败    Outbox 重试，不回滚业务事务
Delivery 失败          Delivery 重试，不重新运行整个 Agent
Connector 失败         更新错误状态，不重复消费成功项
```

### 2.2 依赖检查

建议使用静态架构测试验证：

```text
domain 不导入 infrastructure
application 不导入具体 Channel SDK
plugins 不导入 app 内部私有模块
agent_runtime 不导入数据库驱动
interaction 不直接导入 Repository 实现
```

### 2.3 模块公开面

每个模块只公开：

```text
contracts.py
ports.py
service.py 或 facade.py
commands.py
queries.py
```

`internal/`、ORM Model 和私有实现不作为跨模块依赖。

### 2.4 配置层级

```text
defaults
→ profile config
→ local override
→ environment override
→ runtime command override
```

每次有效配置变更生成版本和 Audit。影响运行语义的配置应关联到新 RunAttempt，而不是无记录地改变正在执行的 Attempt。

### 2.5 测试架构

#### 单元测试

验证领域规则、状态机、策略和纯函数。

#### 契约测试

验证 Gateway、Provider、Plugin、Tool 和 Event Schema。

#### 集成测试

验证 SQLite、Outbox、Task Lease、Delivery 和 Connector。

#### 恢复测试

在关键点模拟进程崩溃：

```text
Tool 执行前后
Receipt 写入前后
Delivery 发送前后
Task Checkpoint 前后
Cursor 推进前后
```

#### 安全测试

验证 Prompt Injection、路径逃逸、权限提升、Secret 泄漏和群聊隔离。

#### 回放测试

使用保存的契约和 Payload Reference 重建 Context，验证系统能解释历史决策。

### 2.6 架构变更规则

以下变更必须更新总体架构和 ADR：

- 新增顶层模块；
- 修改状态所有权；
- 引入新的跨进程通信；
- 引入新的事实源；
- 改变 Task/Delivery 语义；
- 扩大插件权限；
- 允许新的高风险副作用；
- 改变身份合并策略；
- 改变数据保留或加密策略。

### 2.7 总体架构验收标准

系统架构实现至少应满足：

1. Channel SDK 对象不会进入 Agent Core 领域层。
2. 同一平台消息重复投递不会创建重复逻辑 Turn。
3. 同一 Conversation 的普通用户消息有确定执行顺序。
4. 长期任务可在进程重启后恢复或进入明确失败状态。
5. 高风险 Tool 调用经过 Policy 和持久 Approval。
6. 外部副作用具有幂等键和 Receipt/Reconcile 机制。
7. Agent 完成与 Channel 发送失败相互解耦。
8. SQLite 是唯一业务事实源，JSONL 不决定状态。
9. Event、Delivery 和 Payload Export 使用 Outbox。
10. 外部内容始终携带 Trust Label，不能提升权限。
11. Secret 不出现在默认 Trace 和日志中。
12. Plugin 只能使用公开 API 和声明权限。
13. Dashboard 写操作全部通过 Command API。
14. 每个 Turn/RunAttempt/Task 都受 ResourceBudget 约束。
15. 核心链路可以通过 Trace、Audit 和状态记录解释。
16. 备份能够恢复数据库、Payload 引用和插件配置。
17. 降级状态不会静默产生未持久化副作用。
18. 模型、Tool、Channel 和 Plugin 失败不会破坏其他模块的事实状态。

### 2.8 总体设计结论

本系统的核心不是一个可无限插入 Hook 的 Pipeline，而是由明确状态、明确所有权和明确运行语义组成的长期 Agent Runtime：

```text
Channel Gateway 解决平台协议
Turn Orchestrator 解决同步交互
Durable Job Runner 解决跨时间工作
Event Bus 解决事实传播
Agent Runtime 解决推理和 Tool Loop
Cognition 解决个人长期认知
Proactive Decision 解决主动性边界
Delivery 解决可靠投递
Policy 解决权限和副作用
SQLite/Outbox 解决状态与恢复
Observability 解决解释与运维
```

后续所有模块设计都必须服从这些总体边界。

## 3. Configuration & Secrets

### 3.1 配置层级

```text
defaults
→ profile config
→ local override
→ environment override
→ runtime command override
```

高优先级不能覆盖标记为不可运行时修改的安全配置。

### 3.2 配置 Schema

每个模块声明：

```text
schema_version
defaults
validation
secret_fields
reload_mode
```

### 3.3 Secret 管理

Secret（API Key、Token、密码）**仅存放于 `.env` 文件**，不写入 `config.yaml`：

- `.env` 中的变量通过 `secret://` 前缀在 `config.yaml` 中引用；
- 行为配置（超时、阈值、特性开关、路径、展示偏好）全部放在 `config.yaml`；
- 不得要求用户在 `.env` 中设置行为开关——`.env` 是 secret-only 的文件；
- 新增 Secret 字段在 `OPTIONAL_ENV_VARS` 注册表中声明（含描述、提示文本、是否为密码字段、获取 URL）。

**示例**：

```yaml
# config.yaml — 行为配置
model:
  provider: deepseek
  main:
    model: "deepseek-v4-flash"
    api_key: secret://deepseek/api_key     # 引用 .env 中的值
    enable_thinking: true
```

```bash
# .env — 仅 Secret
DEEPSEEK_API_KEY=sk-xxx
TELEGRAM_BOT_TOKEN=123456:ABC
```

### 3.4 Profile

即使只有单用户，也建议保留 Profile 概念，用于隔离：

```text
configuration
database
payload store
plugins
prompts
memory
channel endpoints
scheduled jobs
```

Profile 操作：

```text
cogito profile list           # 列出所有 Profile
cogito profile create <name>  # 从默认 Profile clone 创建新 Profile
cogito profile delete <name>  # 删除 Profile（确认后删除目录）
cogito -p <name>              # 以指定 Profile 启动
```

Profile 之间是完全隔离的独立实例——各自拥有独立的 config.yaml、.env、数据库、插件、记忆和 cron 任务。默认 Profile 位于 `~/.cogito/`，命名 Profile 位于 `~/.cogito/profiles/<name>/`。

测试 Profile 与正式 Profile 不能共享写入目录。

### 3.5 示例

```yaml
profile: personal
storage:
  database: data/database/agent.db
  payload_root: data/payload
  payload_encryption: none
  trace_level: 1

conversation:
  default_session_policy: channel_conversation
  group_sessions_per_user: true
  thread_sessions_per_user: false
  per_context_partition_concurrency: 1

interaction:
  bind_host: 127.0.0.1
  allow_remote: false
  validate_origin: true

agent:
  max_iterations: 8
  max_tool_calls: 12
  model_provider: primary
  fallback_model_provider: local

proactive:
  quiet_hours: "23:00-08:00"
  daily_push_limit: 8
  dry_run: false

security:
  require_approval_for: [high, critical]
  sandbox_shell: true
```

- Policy；
- Memory 冲突；
- Delivery 目标选择；
- Idempotency Key；

针对：

- Channel Gateway ↔ Agent Core；

- 主动推送；

### 4.4 恢复测试

必须验证：

- Tool 执行后、Receipt 写入前崩溃；
- Delivery 成功后、本地状态更新前崩溃；
- Worker 持有 Lease 时被终止；
- SQLite Busy；
- Payload 写入失败；
- 重启后 Schedule 不重复触发；
- waiting_user Turn 恢复并创建新 RunAttempt。

- Prompt Injection；
- 路径穿越；
- 符号链接逃逸；
- SSRF；
- Secret 泄露到 Trace；
- 插件越权；

### 4.6 回放测试

从归档的 Channel/Event Payload 重放 Normalize、Decision 和 Rendering，确认结果可解释。Tool 副作用在回放模式下必须替换为 Stub。

## 5. 运维、恢复与数据治理

### 5.1 启动顺序

```text
1. validate config
2. open database and run integrity check
3. apply migrations
4. initialize payload store
5. load policy
6. validate plugins
7. start worker recovery
8. start API and interaction
9. start channel gateway/connectors
```

### 5.2 Shutdown

- 停止接受新 Turn；
- 停止领取新 Task；
- 等待短任务完成；
- 为长任务写 Checkpoint；
- 释放 Lease；
- Flush Trace 和 Outbox；
- 关闭 Channel 连接。

### 5.3 健康状态

```text
healthy
ready
degraded
unhealthy
```

单一 Connector 或非关键 Plugin 故障使系统 `degraded`，不应让整个 Agent 不可用。

### 5.4 数据导出与删除

系统应支持：

- 按 Conversation 导出；
- 按时间导出；
- 导出 Memory；
- 删除 Message 和相关 Payload；
- 删除或匿名化外部用户数据；
- 生成删除审计。

### 5.5 恢复原则

恢复优先级：

1. 数据一致性；
2. 避免重复副作用；
3. 恢复用户可见状态；
4. 恢复后台工作；
5. 恢复低优先级主动任务。

## 6. 全局实现约束

以下约束适用于所有模块：

1. 不在数据库事务内执行模型或网络调用；
2. 不由模型决定权限；
3. 不由插件修改受保护 Envelope 字段；
4. 不直接双写 SQLite 和 JSONL；
5. 不在内存中无限等待审批；
6. 不在重试前重复执行未知状态的副作用；
7. 不以显示名称作为身份唯一键；
8. 不将外部文本视为系统指令；
9. 不把完整 Secret 写入 Prompt、日志或 Trace；
10. 不让 Dashboard 直接执行 SQL 修改业务状态；
11. 不假设不同 Model Provider 完全等价；
12. 不混用不同 Embedding 空间；
13. 不保证平台 Exactly Once，必须实现本地幂等和对账；
14. 不把所有功能抽象成通用 Pipeline；
15. 不把所有插件默认放入核心进程。

## 7. 架构验收标准

系统设计和实现应能够明确回答以下问题：

- 一条消息来自哪个 Endpoint、Principal、Conversation 和 Session？
- 同一平台事件重复到达时会不会重复执行？
- 同一 Conversation 两条消息同时到达时如何排序？
- Agent 在哪一步调用了什么 Tool，权限为什么被允许？
- Worker 在任意步骤崩溃后如何恢复？
- Tool 已产生副作用但请求超时时如何对账？
- 回复已生成但 Channel 发送失败时如何重试？
- 主动消息为什么发送、延迟、加入摘要或丢弃？
- 一条 Memory 来源于哪条消息，是否经过确认，是否已过期？
- 哪个 Middleware 修改了数据，修改前后差异是什么？
- Trace 中是否包含 Secret、PII 或不应保存的完整内容？
- 插件故障是否会影响整个 Agent？
- 数据库和 Payload Store 如何一致备份和恢复？
- 当前资源消耗和每日主动任务预算是否可控？

如果任何问题只能依赖“查看代码猜测”或“模型应该不会这样做”，则相关设计仍不完整。

## 附录 B：参考项目与采纳边界

### LangBot

参考地址：`https://github.com/langbot-app/LangBot`

主要借鉴：

- 多平台 Channel Adapter；
- 消息对象转换；
- 平台连接和发送能力；
- 多 Channel 的运行经验。

采纳边界：

- 作为 Channel Gateway 或薄桥接层；
- Agent Core 不依赖 LangBot 内部 Pipeline 和消息实体；
- 通过版本化 ChannelEnvelope/AgentReply 契约集成。

### Akashic Agent

参考地址：`https://github.com/kachofugetsu09/akashic-agent`

主要借鉴：

- Agent 阶段化处理思路；
- 主动信息处理；
- Memory 和工具扩展理念；
- 可观察和可插入的运行节点。

采纳边界：

- 不复制多套语义相近的 Hook/Phase/Processor 抽象；
- 统一收敛为 Provider、Middleware、Task Handler 和 Event Handler；
- 将持久状态机、幂等和事务边界置于扩展机制之前。

### Hermes Agent

参考地址：`https://github.com/NousResearch/hermes-agent`

主要借鉴：

- 个人 Agent 的本地运行方式；
- Profile、Skill、Tool 和任务机制；
- 长期使用场景下的能力组织。

采纳边界：

- 保留本系统 Canonical Memory；
- 外部记忆能力作为检索或增强层；
- Profile 从配置、数据目录、任务和插件层面完整隔离。

## 附录 C：设计结论

本系统的核心不是“让所有东西都可插拔”，而是建立一套长期可运行、可恢复、可解释、可约束的个人 Agent Runtime。

最终架构遵循以下主线：

```text
Channel Gateway 负责协议
Turn Orchestrator 负责即时交互
Durable Job Runner 负责长期工作
Event Bus 负责事实通知
Agent Runtime 负责受限推理
Cognition 负责统一长期认知
Capability 负责受控外部能力
Delivery Outbox 负责可靠发送
SQLite 负责事务事实
Payload Store 负责大体积内容
Policy 负责权限和风险
Observability 负责可审计性
```

在此基础上，Provider 和 Plugin 才是实现替换与扩展的手段，而不是系统控制流本身。
