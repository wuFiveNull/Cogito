# 本地部署与运行手册

## 1. 目录

```text
config/       非敏感配置
data/database SQLite
data/payload  内容寻址 Payload
data/cache    可重建数据
data/exports  用户导出
data/backups  本地备份
logs/         脱敏日志
plugins/      已安装插件
```

数据目录仅允许当前本机用户访问。开发阶段 Payload 明文，禁止自动同步到公共云盘。

## 2. 进程

默认启动 `personal-agent`（API、Worker、Scheduler）和独立 `channel-gateway`。Interaction 监听 `127.0.0.1`。资源不足时可全部合并，但 Gateway Bridge 契约和 Worker Lease 仍保留。

## 3. 启动

```text
load config/secret refs
→ open SQLite and verify WAL
→ check/execute safe migration
→ verify Payload root
→ load policy/providers/plugins
→ recovery scan
→ start worker/scheduler/outbox
→ start API
→ start Gateway/UI
```

Recovery 未完成前不发送积压主动通知。

## 4. 关闭

停止新普通工作，保留短暂控制窗口；停止 Scheduler 和新 Lease；当前 Attempt 写 Checkpoint；标记 unknown 副作用；释放 Lease；刷新 Trace；关闭数据库。超过关闭期限时强制停止但不伪造 completed。

## 5. 健康检查

分别报告 liveness、readiness、SQLite/Payload、Provider、Gateway、队列 backlog 和降级原因。Provider 不可用不一定使只读 Query 不可用。

本地 Tool/MCP 诊断命令：

```powershell
python -m cogito config check --config config.toml
python -m cogito config profiles
python -m cogito config init --profile developer --output config.toml
python -m cogito doctor --config config.toml
python -m cogito tools list --config config.toml
python -m cogito tools audit --all --config config.toml
python -m cogito tools describe read_file --config config.toml
python -m cogito mcp list --config config.toml
python -m cogito mcp status --config config.toml
python -m cogito mcp tools --server aihot --config config.toml
```

`tools list` 使用与 reactive Agent 相同的 Toolset 和 Registry 组合规则，并实际连接已启用
MCP 以显示动态 Tool；不会启动 Worker、调用模型或执行 Tool。`doctor` 检查配置、数据库、
Workspace/Skill Root、模型配置和 MCP 初始化状态，任一必需项失败时退出码为 1。
运行中的本地控制面还提供只读 `/api/tools`、`/api/tools/{name}`、`/api/mcp/status`，便于
管理界面复用同一 Runtime 状态；这些端点不执行 Tool，也不修改 MCP 配置。

## 6. 升级

升级前检查磁盘、备份、插件兼容和 Migration；停止写入进程；执行升级；运行完整性检查和 smoke test；最后恢复 Gateway。失败时恢复数据库、Payload manifest、配置和插件版本。

## 7. 备份

先 SQLite online backup，再从快照生成 Payload manifest，复制 manifest 文件、配置版本和插件 manifest。每次备份记录 Hash、创建时间和应用版本。定期执行真实恢复演练。

## 8. 恢复

在隔离目录验证 SQLite integrity/foreign keys、Payload Hash 和配置；清除过期 Lease；扫描 unknown Tool/Delivery；重建索引；以 recovery profile 启动；确认后才允许真实副作用。

## 9. 容量维护

监控数据库、Payload、备份、Trace、附件和缓存增长。先清理缓存/过期 Trace，再归档旧 Payload；用户 Message 和确认 Memory 按保留策略处理，不因磁盘压力直接删除。

## 10. 常见故障

| 故障 | 首要动作 |
|---|---|
| SQLite busy | 检查长事务和写并发，不无限增加 timeout |
| Payload 缺失 | 停止相关任务，核对 manifest/备份 |
| Gateway 断开 | 保留 Delivery，恢复后重试/对账 |
| Provider 失败 | 启用降级，检查预算和认证 |
| unknown 副作用 | 禁止普通重试，执行 reconcile |
| 队列堆积 | 按优先级和错误分类处理，不盲目扩并发 |

## 11. 验收

新安装、正常重启、强制终止、升级失败、备份恢复、磁盘不足和 Provider/Gateway 故障都必须有可重复操作记录。
