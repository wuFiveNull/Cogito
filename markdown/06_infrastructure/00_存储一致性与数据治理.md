# 存储、一致性与数据治理

> **文档编号**：STORAGE-DATA  
> **适用范围**：SQLite、Payload Store、Repository、Unit of Work、Profile 隔离、Schema、索引、保留、备份和恢复  
> **权威边界**：本文是其范围内的规范性来源；总体架构文档只负责概念与边界。  
> **细化规范**：物理 Schema、配置和备份恢复操作分别以 `DATABASE-SCHEMA`、`CONFIG-PROFILES`、`LOCAL-OPERATIONS` 为准。  
> **关联文档**：PROACTIVE-TASKS, SECURITY-OBS, OPS-GOVERNANCE

## 阅读说明

**目的**：定义事务事实源、Payload Store、Repository、Profile 隔离、表结构、迁移、备份和数据生命周期。

**边界**：不定义业务模块的决策算法；数据库表建议必须服从领域聚合边界。

**建议读取方式**：设计持久化、Migration、备份、恢复或数据删除时读取。

## 1. 数据与存储总体架构

### 1.1 数据分类

```text
业务状态数据       Message、Turn、Task、Delivery、Memory
索引与关系数据     Conversation、Endpoint、Trace Index
原始 Payload        平台原始消息、邮件、网页、附件
执行 Payload        Prompt、模型响应、Tool 大结果
审计数据            Command、Approval、Policy Decision
派生数据            Dashboard Projection、Embedding、摘要缓存
临时数据            流式 Delta、临时下载、Sandbox 文件
```

### 1.2 Profile 与存储隔离

Profile 提供多个完全隔离的运行实例，每个 Profile 拥有独立的存储目录：

```text
~/.cogito/                      # 默认 Profile 的 HOME
├── config.yaml
├── .env                        # Secret 仅存放于 .env
├── data/
│   ├── database/agent.db
│   ├── payload/
│   └── cache/
├── plugins/
├── skills/
├── memory/
├── cron/
├── logs/
├── sessions/
└── backups/

~/.cogito/profiles/<name>/      # 命名 Profile 的 HOME（完全隔离）
├── config.yaml
├── .env
├── data/...
└── ...
```

关键规则：

- `get_home()` 始终返回当前活跃 Profile 的目录，不硬编码 `~/.cogito`
- `display_home()` 用于用户可见消息，展示简化的路径表示
- Profile 操作（list/create/delete）锚定在 `~/.cogito/profiles/`，不受 `get_home()` 影响——让 `cogito -p coder profile list` 能看到所有 Profile
- 测试必须隔离 `HOME` 到临时目录（autouse fixture），不得写入真实 `~/.cogito`
- Profile 之间不能共享写入目录，但可共享只读的参考数据（如通过 symlink 的共享 skills 目录）

### 1.3 SQLite 责任

SQLite 保存：

- Principal、Endpoint、Conversation、Session；
- Message、Turn、RunAttempt；
- Task、Lease、Checkpoint、Schedule；
- Event、Inbox、Outbox、Consumer Offset；
- Delivery、Attempt、Platform Receipt；
- MemoryItem、关系、候选和确认；
- ToolCall、Approval、Side Effect Receipt；
- Connector、Cursor 和源项索引；
- Trace/Span 元数据和 Audit；
- 配置版本、插件状态和资源使用量。

建议设置：

```text
journal_mode = WAL
foreign_keys = ON
busy_timeout = configured
synchronous = NORMAL 或按策略 FULL
```

### 1.4 Payload Store 责任

保存大对象：

- 原始 Channel Payload；
- Connector 原始内容；
- 完整 Prompt 与模型原始响应；
- Tool 大型输出；
- 图片、音频、文档和附件；
- 完整 Trace Payload；
- 数据导出和备份文件。

多模态 Asset、页面 derivative 和模型原始响应都只引用 Payload Store，不复制
文件路径成为新的事实源。视觉分析的结构化小字段可以进入 SQLite 以支持缓存查询，
但二进制、完整请求和完整原始响应仍保存为 Payload。

目录示例：

```text
data/payload/YYYY/MM/DD/<sha256>.<ext>
data/jsonl/YYYY/MM/DD/<stream>.jsonl.zst
data/attachments/<sha256>/<safe_filename>
data/tmp/<attempt_id>/
```

内存值对象 `PayloadObject`（PayloadStore 对外）含 `payload_id / storage_uri /
sha256 / content_type / size_bytes / compression / encryption / redaction_level /
retention_class / reference_count_hint / created_at`。实际持久化表 `payload_objects`
仅保留：

```text
payload_ref
sha256
content_type
size
storage_path
created_at
```

`compression / encryption / redaction_level / retention_class` 当前为内存侧默认值，未落列。

### 1.5 Payload 写入协议

```text
1. 写入临时文件
2. 计算 hash、size 和 content type
3. fsync/close
4. 原子 rename 到 hash 路径
5. 在同一 SQLite 事务中写 payload metadata 和业务对象引用
```

如果事务失败，孤儿文件由 Garbage Collector 清理。`reference_count` 只作为维护提示，实际可删除性以数据库引用扫描为准。GC 只删除超过安全期、未出现在有效数据库快照引用中的对象；个人单机部署不引入复杂分布式 GC。

当前开发阶段允许 Payload 明文保存，以便检查 Agent 行为。该策略必须在配置中显式为 `payload.encryption = none`，并要求数据目录仅对本机用户可读。启用远程访问、共享机器或外部备份前，应重新评估加密与脱敏；当前版本不实现密钥管理。

### 1.6 JSONL 的定位

JSONL 用于：

- 归档不可变原始流；
- 导出 Trace；
- 数据分析；
- 人工回放输入。

JSONL 不用于：

- 决定 Task 当前状态；
- 决定 Delivery 是否成功；
- 作为唯一 Message 历史；
- 与 SQLite 进行无事务双写。

### 1.7 Message 历史与 Context Snapshot

长期保存完整 Message，但 Agent 每次只读取受预算约束的 Context Snapshot。

Snapshot 包含：

```text
selected recent messages
summary references
selected memory items
preference/goal views
external context
selection policy version
token estimate
```

这样可以在回放时解释"当时模型看到了什么"。

### 1.8 向量和全文检索

Canonical 数据仍在 SQLite。检索索引是派生数据：

```text
SQLite FTS
sqlite-vec
外部向量库
本地 Embedding 文件索引
```

索引损坏可以重建，不得成为 Memory 唯一事实源。

### 1.9 缓存

可缓存：

- Provider capability；
- Embedding；
- Tool Manifest；
- 短期 Context Summary；
- Connector 条件请求状态；
- Dashboard Projection。

缓存必须有明确失效规则，不能保存不可恢复的唯一状态。

### 1.10 数据保留策略

```text
hot       当前状态、近期 Message、近期 Trace
warm      压缩历史 Payload、旧 Trace
archive   用户明确长期保存的数据
volatile  临时模型输出、下载、Sandbox 文件
secret    单独加密和访问审计
```

每种对象定义：

```text
retention period
delete condition
archive condition
legal/user hold
redaction requirement
```

### 1.11 删除与级联

删除 Principal 或 Conversation 不应立即物理删除所有证据。建议流程：

```text
mark deleted
→ revoke access
→ remove from normal query
→ create deletion job
→ delete/redact payload
→ garbage collect unreferenced data
→ retain minimal audit tombstone
```

### 1.12 备份与恢复

#### 1.12.1 备份内容

备份必须包含逻辑一致的：

```text
SQLite snapshot
Payload manifest
Payload files
Config without plaintext secrets
Plugin manifest and versions
Migration version
外部插件状态（见 §1.12.3）
```

低并发个人部署使用以下顺序即可保证正常恢复：先创建 SQLite online backup，再从该快照读取 Payload 引用形成固定 manifest，最后复制 manifest 中的内容寻址文件。备份期间 Payload 文件不可原地修改。

#### 1.12.2 恢复流程

恢复后执行：

- 校验 Payload Hash；
- 检查数据库外键；
- 重建派生索引；
- 将过期 Lease 清理；
- 扫描未知副作用；
- 禁止自动发送恢复前积压的过期主动通知，除非策略允许。

#### 1.12.3 外部插件状态

部分插件将状态存储在 `~/.cogito/` 之外的路径（例如某 Memory Provider 将数据存在 `~/.honcho/`）。备份命令需要支持声明式的外部路径捕获：

- 插件在 Manifest 中声明 `backup_paths: ["~/.honcho/"]`（路径列表，可为文件或目录）；
- 备份命令解析这些路径，将存在于 HOME 下的内容捕获到归档的 `_external/` 子目录；
- 恢复时，`_external/` 内容恢复到原始绝对路径；
- HOME 外的路径出于安全原因跳过（不备份系统文件）。

备份命令必须在没有网络访问、没有 `initialize()` 的情况下可调用——只从配置和环境变量解析路径。

### 1.13 Migration

数据库 Migration：

- 单调递增；
- 可重复检测；
- 在备份后执行；
- 记录开始、完成和失败；
- 插件 Migration 与 Core Migration 分开命名；
- 破坏性迁移必须提供数据导出或回滚方案。

## 2. Storage

### 2.1 Repository 边界

Repository 按 Aggregate 提供接口：

```text
ConversationRepository
TurnRepository
TaskRepository
EventRepository
DeliveryRepository
MemoryRepository
TraceRepository
PayloadRepository
```

不提供"万能 execute SQL"给业务模块或插件。

### 2.2 Unit of Work

Application Service 使用 UnitOfWork：

```python
async with uow:
    ...
    uow.events.add(...)
    uow.deliveries.add(...)
    await uow.commit()
```

Commit 将业务修改和 Outbox 一起写入。

### 2.3 Schema Migration

- 数据库 Migration 有单调版本号；
- Plugin Migration 使用独立命名空间；
- 启动前备份；
- 不允许插件修改核心表；
- 大型迁移支持离线执行和恢复点。

### 2.4 SQLite 写入策略

- 每个进程使用独立连接池，默认只有一个 Agent Core 写入者和有限 Worker；
- 写事务尽量短；
- Worker 批量领取任务时限制批次；
- 遇到 `database busy` 使用有限重试和抖动；
- Dashboard 大查询使用只读连接和分页；
- 向量检索或全文检索避免长时间锁表。

### 2.5 原子文件写入

所有持久化文件写入使用原子替换模式，防止写入中断导致文件损坏：

```text
1. 写入临时文件（同目录下 .tmp 或 .<random> 后缀）
2. fsync + close
3. 原子 rename 覆盖目标文件（同文件系统内是原子操作）
```

适用于：config.yaml、sessions.json、jobs.json、Skill 文件、Profile 配置等所有非 Payload 的持久化文件。JSON 写入先序列化到内存，验证 JSON 合法性，再写入临时文件。

### 2.6 文件锁

跨进程互斥使用文件锁：

- POSIX：`fcntl.flock()`（建议锁，进程退出自动释放）
- Windows：`msvcrt.locking()`（强制锁）
- 无锁可用时降级为进程内 `threading.Lock`（仅防御本进程并发）

典型使用场景：Scheduler tick 防重、备份过程防并发写入、配置迁移防多进程同时执行。
