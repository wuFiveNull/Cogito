# 可观察性与审计

## 1. 目标

系统必须回答：谁触发、使用什么上下文、调用什么模型/Tool、为何允许、状态怎样变化、外部结果是什么、消耗多少资源以及如何恢复。从入站消息到平台发送确认，每一步都可追溯。

## 2. Trace 树

### 2.1 端到端链路

```text
inbound message → platform ack（完整链路，同一个 trace_id）
├── channel.receive          入站到达 + 幂等检查
├── identity.resolve         身份解析（Principal、Endpoint）
├── conversation.enqueue     Context Partition Lane 排队
├── turn.run                Turn 执行
│   ├── context.build       上下文装配
│   │   ├── memory.retrieve 长期记忆召回
│   │   └── history.select  会话历史选择
│   ├── agent.iteration.1   第一轮 Agent Loop
│   │   ├── model.request   模型请求
│   │   ├── model.response  模型响应
│   │   └── tool.call       工具调用
│   │       ├── policy.evaluate
│   │       └── tool.call.completed
│   ├── agent.iteration.2
│   │   └── ...
│   └── reply.build         回复装配
├── storage.commit          业务状态 + Outbox 同一事务
├── delivery.enqueued       投递进入队列
├── delivery.attempt        投递尝试（可能多次）
└── delivery.result         平台发送结果 + 平台 message_id
```

### 2.2 异步链路

后台 Task 使用独立的 `trace_id`，但通过 `correlation_id` 和 `causation_id` 与上游关联：

```text
User Turn (trace_id=A)
  └── CreateTask Command
        └── TaskCreated Event (causation_id=command_id)
              └── Task Execution (trace_id=B, correlation_id=A)
                    ├── model.request
                    ├── tool.call
                    └── delivery.enqueued
                          └── Delivery Attempt (trace_id=C, correlation_id=B)
```

TraceContext 跨 Command、Event、Task 和 Delivery 传播。`causation_id` 记录直接触发者——例如 Task 的 `causation_id` 指向创建它的 Command 或 Event。

## 3. Span

Span 保存：名称、组件、开始/结束时间、状态、父子关系、资源使用、安全属性、Payload 引用。

禁止：把完整 Prompt、Secret 或任意 Tool 输出作为普通 Attribute。

过期 Span 由后台清理任务处理，不影响正常 Agent 运行。

## 4. Log

结构化字段：timestamp、level、component、event_code、trace_id、对象 ID、安全消息和有限 Metadata。

- 异常堆栈进入受限日志（`errors.log`，WARNING+）；
- 用户可见错误使用 ErrorEnvelope（safe_details 字段）；
- 禁止依赖字符串解析实现核心告警——使用稳定的 event_code。

## 5. Metric

核心指标：Turn 延迟/失败、模型 Token/成本、Tool 拒绝/unknown、Task backlog/Lease、Outbox age、Delivery 成功、Connector freshness、Memory 命中、SQLite busy、Payload 增长、Drift 资源。

指标标签禁止使用 principal_id、conversation_id、URL 全路径、文件名等高基数字段。

## 6. Audit

### 6.1 Audit 记录

```text
audit_id
actor_principal_id
action
target_type/id
before/after safe diff
policy/approval/config version
result
trace_id
created_at
```

### 6.2 必须审计的操作

| 操作类别 | 具体动作 |
|---------|---------|
| 审批 | 批准、拒绝、审批创建、审批过期 |
| Secret | 读取 Secret、Secret 引用创建 |
| 高风险 Tool | risk_level=high 或 critical 的 Tool 执行 |
| 配置 | 修改安全相关配置（Policy、权限、Sandbox） |
| 插件 | 安装、启用、禁用、卸载插件 |
| Memory | 确认 Memory、纠正 Memory、删除 Memory |
| 投递 | Delivery 重放 |
| 数据 | 数据导出、删除 Principal/Conversation |

### 6.3 约束

- Audit 记录不可由普通插件删除或修改；
- Audit 表受独立保留策略保护（不随普通 Log 清理）；
- 删除 Principal 时保留最小 Audit tombstone（操作人和时间），不保留参数细节。

## 7. Observer Hook 集成

### 7.1 与 Trace 的关系

Observer Hook（SECURITY-OBS §2）是 Trace 系统的**数据来源**。每个 Hook 触发时：

1. 构造 sanitized 负载；
2. 创建或更新 Span（Hook 名称映射为 Span 名称）；
3. 将负载中的关键字段写入 Span Attribute；
4. 将 `api_request_id`、`tool_call_id` 等关联 ID 写入 Span 属性供消费者 join。

Trace 系统是 Observer Hook 的默认消费者。如果用户安装了外部遥测插件（如 Langfuse 导出器），同一份 Hook 负载也发送给这些消费者。

### 7.2 Hook 字段映射

```
session_id, turn_id, task_id → Span attribute
api_request_id              → Span attribute + Span 名称后缀
tool_call_id                → Span attribute
correlation_id              → Span attribute（跨 Trace 关联）
duration_ms                 → Span duration
status                      → Span status (ok / error / cancelled)
usage (tokens)              → Span metric
error (type, message)       → Span event
```

## 8. 脱敏

写入前检测 Secret、Token、Cookie、认证头、PII 和大 Payload。

- 默认 Trace Level 只保存状态和摘要；
- 开发 Profile 可以保存完整 Prompt/Response Payload，但仍排除明确 Secret，并受本地目录权限和保留期控制；
- Observer Hook 负载在构造时已完成脱敏（sanitized 字段），不需要消费者二次处理。

## 9. 告警

告警基于持续异常和 backlog，不因单次可恢复失败骚扰用户。严重条件：

- 数据库不可写；
- 未知副作用积压（`unknown` 状态 ToolCall > 阈值）；
- 备份长期失败（> 7 天）；
- 插件逃逸（尝试访问 Manifest 未声明的资源）；
- 主动推送异常增加（> 2x 日预算）；
- 数据目录容量不足（< 10% 可用）。

告警通过系统健康指标触发，不依赖日志字符串匹配。

## 10. 保留

| 数据类型 | 保留策略 |
|---------|---------|
| Metric | 长期聚合（90 天细粒度 + 1 年粗粒度） |
| 普通 Log | 30 天 |
| Trace（Level 1） | 14 天 |
| Trace（Level 2 调试） | 3 天（或手动关闭） |
| Audit | 永久（不可自动清理） |
| 完整 Payload | 7 天（或按存储限额 LRU） |

清理通过后台 Task 执行并产生 Audit。

## 11. 测试

- Trace 因果链完整性（验证 corsor_id 跨 Command/Event/Task/Delivery 一致）；
- 跨进程 TraceContext 传播（Gateway → Agent API → Worker）；
- Secret 脱敏覆盖率（注入已知 Secret → 在所有 Redactor 执行点验证不存在）；
- 高基数标签限制（验证指标标签不包含 principal_id 等字段）；
- Audit 不可绕过（验证所有高风险 Tool 执行产生 Audit 记录）；
- Observer Hook fail-open（模拟插件异常 → Agent Loop 继续）；
- 告警去抖（同一条件短期内不重复告警）；
- 保留清理（验证过期数据被清理，Audit 不被清理）。
