# 安全策略、可观察性与资源治理

> **文档编号**：SECURITY-OBS  
> **适用范围**：Trust Label、Policy、Approval、Secret、Sandbox、Observer Hook、Trace、Metric、Audit、Budget、Degradation  
> **权威边界**：本文是其范围内的规范性来源；总体架构文档只负责概念与边界。  
> **细化规范**：Tool 隔离以 `TOOL-SANDBOX` 为准，Trace/Audit 以 `OBSERVABILITY-AUDIT` 为准，安全与恢复测试以 `TEST-EVALUATION` 为准。  
> **关联文档**：CAPABILITY-PLUGINS, ACCESS-DELIVERY, STORAGE-DATA, OPS-GOVERNANCE

## 阅读说明

**目的**：定义信任、权限、审批、安全隔离、Observer Hook 契约、Trace、Audit、资源预算和降级模式。

**边界**：不使用模型输出替代确定性授权；不把完整 Payload 作为默认日志。

**建议读取方式**：涉及副作用、外部内容、Secret、Shell、插件或运行告警时读取。

## 1. 安全、信任、权限与审批架构

### 1.1 威胁模型

主要威胁：

- 外部内容中的 Prompt Injection；
- 恶意或错误的 MCP/Plugin；
- Tool 参数被模型错误生成；
- Secret 泄漏到 Prompt、日志或 Trace；
- 群聊用户访问 Owner 数据；
- Shell 或文件 Tool 越权；
- 外部副作用重复执行；
- Dashboard 控制接口被误用；
- Payload 和备份泄漏；
- 供应链依赖被替换；
- 主动任务无限消耗资源。

### 1.2 Trust Label

```text
system_trusted
owner_trusted
local_untrusted
external_untrusted
secret
```

Trust Label 随数据来源附加，而不是由文本内容判断。

传播规则：

- 引用外部内容的摘要仍保留外部来源标记；
- 多来源合并使用最低信任或来源列表；
- Tool 输出默认不自动视为系统指令；
- Secret 与普通内容混合时整体按敏感处理。

### 1.3 权限 Scope

```text
filesystem.read:<path>
filesystem.write:<path>
network.connect:<host>
email.read
email.send
calendar.read
calendar.write
shell.execute
secret.read:<name>
channel.send:<endpoint>
plugin.manage
config.write
```

Scope 支持精确路径、Host、Endpoint 和资源限定。

### 1.4 授权决策输入

Policy Engine 根据：

```text
Principal
运行模式 reactive/proactive/scheduled/drift
Capability Manifest
requested permissions
Tool risk level
input trust label
target resource
resource budget
existing grants
approval policy
```

输出：

```text
allow
deny
require_approval
allow_with_constraints
```

### 1.5 Tool 风险等级

```text
low
  只读、无敏感数据、无外部副作用

medium
  本地受限写入、普通网络查询、可撤销操作

high
  外部写入、发送消息、Shell、敏感数据访问

critical
  不可逆、广泛权限、高价值资源或批量操作
```

### 1.6 审批模型

审批不是 UI 弹窗，而是持久领域对象：

```text
approval_id
subject_type: turn | task | tool_call | command
subject_id
requester_attempt_id
action_summary
requested_permissions
risk_level
arguments_snapshot
status
expires_at
responder_principal_id
response_reason
```

审批结果只对参数快照和运行上下文有效。参数改变后必须重新审批。
完整字段、消费事务和并发规则见 `APPROVAL-COMMANDS`。

### 1.7 Prompt Injection 防护

- Prompt 中区分 System Instruction、User Instruction 和 External Data；
- 外部内容放入带来源和边界标记的数据段；
- 外部文本不能声明 Tool 权限；
- Tool 参数必须通过结构化 Schema；
- 高风险操作使用确定性 Policy；
- 模型建议读取 Secret 时仍需 Policy 校验；
- Web 内容中的"执行以下命令"等视为数据；
- 图片、OCR、PDF 页面中的指令同样视为不可信外部数据；视觉描述注入 Context 时
  必须带来源、asset_id、trust label 与显式边界；
- Agent 输出中的工具名不能绕过 Registry。

### 1.8 Secret 管理

优先：

```text
OS Keyring
加密 Secret Store
受限环境变量
```

规则：

- Config 只保存 Secret Reference；
- Tool Executor 按需注入；
- Secret 不进入普通 Prompt；
- Trace Redactor 在写入前运行；
- Secret 读取产生 Audit；
- 插件只能读取 Manifest 声明并被允许的 Secret。

### 1.9 文件系统安全

文件 Tool 使用虚拟 Root 或允许路径列表：

```text
workspace/
data/exports/
用户显式授权目录
```

防护：

- 解析真实路径；
- 防止 `..` 和符号链接逃逸；
- 限制文件大小和类型；
- 写入使用临时文件；
- 删除和覆盖属于高风险操作；
- Sandbox 临时目录在任务后清理。

### 1.10 网络安全

- 按 Host/Port Allowlist；
- 阻止访问本地元数据地址和内网敏感服务；
- 限制重定向；
- 记录目标 Host 和数据量；
- 下载设大小和超时；
- 第三方 Plugin 不继承 Core 全部网络权限。

### 1.11 Shell 与代码执行

必须在 Sandbox 中执行：

- 独立工作目录；
- 无默认 Secret；
- 受限网络；
- CPU/内存/时间限制；
- 命令和输出审计；
- 禁止访问 Core 数据库文件；
- 输出通过文件 Manifest 返回。

### 1.12 群聊与身份隔离

- 群聊消息发送者单独解析 Principal；
- 非 Owner 默认不能访问个人 Memory；
- @mention、回复和 Thread 决定是否触发；
- 群聊回复不得泄露私聊历史；
- 群聊 Tool 权限使用单独策略；
- 主动推送默认不发送到群聊，除非明确允许。

## 2. 可观察性：Observer Hook 契约

### 2.1 设计原则

Observer Hook 是系统对外暴露的**只读**遥测契约。行为变更请使用 Middleware 或 Policy Gate。

```text
Observer Hook   只读：报告发生了什么，不改变执行流
Middleware      可写：修改请求/参数/结果，可能改变执行流
Policy Gate     可写：允许/拒绝/要求审批，强制执行
```

关键约束：

- Observer Hook 回调必须是 **fail-open** 的：插件异常被捕获、记录警告，Agent Loop 继续运行；
- 每个 Hook 负载携带 `telemetry_schema_version` 字段，确保消费者与生产者版本解耦；
- 插件应接受 `**kwargs`，使新增字段保持向后兼容；
- 负载构造受 `has_hook(name)` 门控：没有注册监听者时不构建，零开销。

### 2.2 关联 ID

Observer 负载使用以下稳定 ID，消费者无需依赖回调顺序即可关联事件：

```text
session_id              会话/对话标识
turn_id                 用户 Turn 标识
attempt_id              当前 RunAttempt
task_id                 后台 Task 标识（subagent 和隔离执行中重要）
api_request_id          模型请求标识（不透明，不要解析其字符串格式）
api_call_count          当前 Turn 内的 API 调用序号
tool_call_id            模型返回的 tool call ID
parent_session_id       委托子 Agent 的父 Session
child_session_id        委托子 Agent 的子 Session
parent_turn_id          产生委托工作的父 Turn
correlation_id          跨模块业务链路（Command → Event → Task → Delivery）
causation_id            直接触发当前对象的上游实体 ID
```

### 2.3 事件家族

#### 2.3.1 入站消息生命周期

```text
channel.receive         入站消息到达
  ├── payload: channel_type, sender_ref, conversation_ref, message_id
  ├── timing: received_at
  └── trust: trust_label

identity.resolve        身份解析完成
  ├── payload: principal_id, endpoint_id, identity_confidence
  └── decision: matched_by (pre_bind | verify_code | dashboard | whitelist | unknown)

conversation.enqueue    消息进入 Context Partition Lane
  ├── payload: conversation_id, session_id, receive_sequence
  └── state: lane_status, queue_depth
```

#### 2.3.2 Turn 生命周期

```text
turn.start              Turn 创建
  ├── payload: turn_id, session_id, principal_id, priority
  └── trigger: inbound_message | approval_response | resume | retry

run_attempt.start       RunAttempt 开始执行
  ├── payload: attempt_id, turn_id, checkpoint_ref
  └── budget: allocated tokens/calls/time

run_attempt.complete    RunAttempt 结束
  ├── result: succeeded | failed | cancelled | waiting_user
  ├── usage: model_calls, tool_calls, tokens, duration
  └── output: reply_mode, memory_candidates_count

turn.complete           Turn 完成
  ├── status: completed | cancelled | failed
  └── delivery: delivery_id, target
```

#### 2.3.3 Agent Loop 与模型调用

```text
context.build           上下文装配完成
  ├── memory: retrieved_count, selected_count, total_tokens
  ├── history: message_count, summary_version
  └── external: external_sources with trust_labels

model.request           模型请求发送前
  ├── identity: session_id, turn_id, attempt_id, api_request_id
  ├── config: provider, model, api_mode, iteration
  ├── sizing: message_count, tool_count, approx_input_tokens, max_output_tokens
  └── request: sanitized request payload（不含敏感字段）

model.response          模型响应返回后
  ├── result: finish_reason, response_model
  ├── usage: prompt_tokens, completion_tokens, cache_read/write_tokens
  ├── timing: started_at, ended_at, api_duration_ms
  ├── response: sanitized response payload
  └── cost: estimated_cost_usd

model.error             模型请求失败
  ├── error: type, message, status_code, retryable
  ├── retry: retry_count, max_retries, next_retry_at
  └── fallback: fallback_provider, degradation_reason
```

#### 2.3.4 Tool 生命周期

```text
tool.call.proposed      Agent 提议 Tool 调用
  ├── tool: tool_name, tool_version, call_id
  └── args: sanitized arguments

policy.evaluate         Policy 评估完成
  ├── decision: allow | deny | require_approval | allow_with_constraints
  ├── reason: reason_code, policy_version
  └── constraints: applied_constraints (if allow_with_constraints)

tool.call.approved      Tool 获得执行许可
  ├── approval: approval_id (if required), policy_decision_id
  └── tracking: idempotency_key

tool.call.executing     Tool 开始执行
  ├── sandbox: sandbox_profile, resource_limits
  └── context: working_dir, allowed_paths

tool.call.completed     Tool 执行完成
  ├── result: status (ok|error|timeout|unknown), duration_ms
  ├── receipt: receipt_id, external_operation_id, result_hash
  └── output: limited output summary + payload_ref

tool.call.blocked       Tool 被 Policy/Middleware 阻止
  ├── reason: blocked_by (policy_denied | middleware_block), message
  └── pre_execution: true（block 发生在执行之前）

tool.call.cancelled     Tool 被取消（用户取消或 Turn 超时）
  ├── reason: cancelled_by (user | timeout | turn_cancelled)
  └── partial: partial_result_ref（如果有部分输出）
```

#### 2.3.5 审批生命周期

```text
approval.created        审批请求创建
  ├── subject: approval_id, subject_type, subject_id
  ├── action: action_summary, risk_level, permissions
  └── expiry: expires_at

approval.responded      审批收到响应
  ├── decision: approved | rejected
  ├── responder: responder_principal_id, response_reason
  └── timing: created_at, responded_at

approval.expired        审批超时未响应
  └── consequence: auto_reject | keep_waiting | escalate（按策略）
```

#### 2.3.6 投递生命周期

```text
delivery.enqueued       投递进入 Outbox
  ├── delivery_id, target, content_mode, scheduled_at

delivery.attempt        投递尝试
  ├── attempt_no, channel_instance_id, started_at

delivery.result         投递结果
  ├── status: sent | partially_sent | interrupted | failed | unknown
  ├── platform_message_id, error_code
  └── timing: started_at, completed_at, duration_ms
```

#### 2.3.7 子 Agent 生命周期

```text
subagent.start          子 Agent 创建
  ├── parent_session_id, parent_turn_id, child_session_id
  ├── child_role (leaf | orchestrator), child_goal
  └── inheritance: inherited_toolsets, inherited_memory_scope

subagent.complete       子 Agent 返回
  ├── result: child_summary, status
  └── timing: duration_ms
```

### 2.4 负载安全

Observer 负载是给遥测消费者使用的，不是原始对象访问：

- 将 Provider 对象转换为 JSON 兼容结构；
- 限制大型负载大小（Prompt > 10KB 只保留头部 + 尾部摘要）；
- 脱敏 Secret Key、Token、认证头在序列化前完成；
- 二进制内容只记录引用（payload_ref + hash + size），不内联；
- 新消费者应使用 sanitized 字段（`request`、`response`、`error`），不依赖旧版兼容字段。

### 2.5 性能

默认的无 Observer 路径保持零开销。只有至少一个插件注册了对应 Hook 后，Core 才构造 sanitized 遥测负载：

```python
if has_hook("model.request") or has_hook("model.response"):
    payload = build_sanitized_model_payload(request, response)
```

插件作者应遵循：
- 只注册实际消费的 Hook；
- 不多次 deep-copy 或重复脱敏已 sanitized 的负载；
- 保持回调快速返回，将网络导出或批量写入交给后台线程。

## 3. Trace、Log、Metric 与 Audit

### 3.1 可观察性目标

系统必须能够回答：

- 一条消息从哪里进入；
- 为什么创建了这个 Turn/Task；
- 经过了哪些模块；
- 当时使用了哪些 Context 和 Memory；
- 模型和 Tool 调用了什么；
- 谁修改了数据；
- 为什么要求审批或拒绝；
- 外部副作用是否成功；
- 回复发到了哪里；
- 失败后是否重试、恢复或降级；
- 消耗了多少 Token、时间和存储。

### 3.2 Trace 层级

```text
Level 0  状态、耗时、错误码
Level 1  结构化摘要、Token、工具名、字段 Diff
Level 2  完整 Payload，临时调试开启
```

默认 Level 1。

### 3.3 Trace 树

端到端链路从入站消息贯穿到平台确认：

```text
trace: inbound message → platform ack
├── channel.receive
├── identity.resolve
├── conversation.enqueue
├── turn.run
│   ├── context.build
│   │   ├── memory.retrieve
│   │   └── history.select
│   ├── agent.iteration.1
│   │   ├── model.request
│   │   ├── model.response
│   │   └── tool.call
│   │       ├── policy.evaluate
│   │       └── tool.call.completed
│   ├── agent.iteration.2
│   │   └── ...
│   └── reply.build
├── storage.commit       ← 业务状态 + Outbox 同一事务
├── delivery.enqueued
├── delivery.attempt
└── delivery.result      ← 平台消息 ID + 发送状态
```

异步 Task 使用相同 `correlation_id`，但可以拥有新的 Trace。`causation_id` 记录直接上游——例如 `TaskCreated` Event 的 `causation_id` 指向创建它的 Command。

### 3.4 Log、Trace、Metric 与 Audit 四者分离

```text
Log     人类可读运行记录（结构化字段，不依赖字符串解析）
Trace   单次因果链和性能链路（Span 树 + TraceContext）
Metric  聚合趋势和告警（低基数标签，不携带 principal_id 等高基数字段）
Audit   权限、控制和敏感状态变更（不可删除，不可被普通插件修改）
```

不要用字符串日志替代状态和 Audit。

### 3.5 Redaction

写入前执行：

- Secret 检测；
- API Key、Token、Cookie 脱敏；
- 邮件、电话和身份信息按策略处理；
- Payload 大小限制；
- 二进制内容只记录引用；
- Prompt 中的敏感片段使用 Hash/占位符。

Redactor 在以下位置执行：入站原始日志前、模型请求 Trace 前、Tool Result Trace 前、异常序列化前、Dashboard 返回前、导出前。

### 3.6 核心指标

```text
Turn latency
Conversation queue wait
Model latency/token/cost/error
Tool success/timeout/approval rate
Task backlog/retry/failure
Delivery latency/failure/duplicate suspicion
Connector freshness/item rate
Proactive send/accept/dismiss rate
Memory candidate/confirm/retrieval hit
SQLite busy/transaction time
Payload storage growth
Plugin health
```

## 4. 资源治理与运行降级

### 4.1 ResourceBudget

每个 Turn/RunAttempt/Task 可配置：

```text
max_runtime_seconds
max_model_calls
max_input_tokens
max_output_tokens
max_tool_calls
max_network_requests
max_payload_bytes
max_cost
allowed_tools
allowed_paths
allowed_hosts
```

预算消耗必须累计持久化到 Turn/RunAttempt/Task，创建新 Attempt 或恢复后不能重置。

### 4.2 全局预算

```text
daily_model_cost
hourly_model_requests
daily_proactive_pushes
daily_drift_runs
storage_growth_limit
concurrent_tool_calls
concurrent_model_calls
```

### 4.3 主动系统资源边界

主动任务必须满足：

- 有明确来源；
- 有每日预算；
- 有静默时段；
- 有同主题冷却；
- 有最大执行时间；
- Drift 默认只能使用只读 Tool；
- 资源不足时主动任务优先降级，不影响用户即时消息。

### 4.4 降级模式

#### 4.4.1 Model Degraded

- 使用备用 Provider；
- 禁用复杂 Tool Loop；
- 返回简化结果；
- 将任务转后台；
- 明确标记降级原因。

#### 4.4.2 Storage Degraded

- SQLite 不可写时停止接受会产生状态的新工作；
- 不允许只在内存中继续完成副作用；
- 只读 Dashboard 可以继续；
- Channel Gateway 返回暂时不可用或缓存受限输入。

#### 4.4.3 Payload Store Degraded

- 小 Payload 可暂时内联到受限表；
- 大附件拒绝或延迟；
- 不得创建无效 Payload Reference。

#### 4.4.4 Channel Degraded

- Delivery 保留并重试；
- 可根据策略切换备用 Endpoint；
- 被动回复切换 Channel 必须获得用户明确配置，不能自动泄露到其他平台。

#### 4.4.5 Plugin Degraded

- 熔断插件；
- 移除其 Capability；
- 使用内置 Provider；
- 保留故障状态供 Dashboard 查看。

### 4.5 Dashboard 观察范围

Dashboard 展示：

- 当前进程和 Provider 健康；
- Channel/Connector 状态；
- Context Partition Lane 和排队；
- Turn、RunAttempt、Task、Approval；
- Event、Candidate、Decision；
- Delivery 和 Attempt；
- Memory 和来源；
- Trace Tree 和 Payload 权限视图；
- 插件、权限、预算和资源使用；
- 数据库、Payload、备份和 Migration 状态。

Dashboard 控制动作必须进入 Command API 并产生 Audit。

当前目标是单机个人使用：控制 API 默认只监听 loopback 或受权限控制的本地 Socket，不建设多租户认证系统。安全投入优先保障 Tool、Shell、文件、网络、MCP 和插件的 Policy/Sandbox 边界。未来开放局域网或远程访问前，必须补充认证和 TLS，不能直接复用本地信任假设。
