# 消息持久化与历史

## 1. 事实模型

Message 是不可变的标准内容事实。平台编辑通过 MessageRevision 表达；流式 Delta 不是 Message；Delivery 是发送生命周期，不替代 Message。

## 2. Message 字段

```text
message_id
conversation_id
session_id
sender_principal_id
sender_endpoint_id
role
direction: inbound|outbound|internal
platform_message_id
reply_to_message_id
current_revision_no
trust_label
raw_payload_id
receive_sequence
created_at
deleted_at
```

Identity Service 将 LangBot Bridge 的稳定 `sender_endpoint_ref` 解析/绑定为 Core `sender_endpoint_id`；Message 保存 Core ID，Endpoint 保留平台引用证据，Core 不保存 SDK 对象。

## 3. ContentPart

ContentPart 单独保存顺序和类型：

```text
part_id/message_id/revision_no/ordinal
kind
text_inline or payload_id
content_type/size/hash
filename/metadata
trust_label
```

文本超过内联阈值、二进制和附件写 Payload Store。文件名只作展示，不能作为磁盘路径。

## 4. 入站事务

同一事务：

1. 插入 `inbound_inbox(channel_instance,platform_event_id)`；
2. 解析/创建 Conversation 和 Session 引用；
3. 分配持久 `receive_sequence`；
4. 插入 Message 与 ContentPart；
5. 创建 Turn；
6. 写 Event Outbox。

重复入站返回已有 `message_id/turn_id`，不重复创建 Turn。

## 5. 出站事务

最终 Agent 内容先创建不可变 Assistant Message，再创建 Delivery Outbox 并绑定 `message_id`。流式 provisional Delivery 最终化时，在一个事务中创建 Message、绑定 Delivery 并写 finalization Outbox。

## 6. 平台编辑与撤回

平台编辑事件创建 `MessageRevision`：

```text
message_id/revision_no
platform_edit_id/platform_revision
edited_at/observed_at/editor_endpoint_id
content_parts
raw_payload_id
```

同一平台消息的编辑先按 `platform_revision` 比较；平台没有版本时按稳定 `platform_edit_id` 去重，并使用 `(edited_at, observed_at, event_id)` 决定确定性顺序。迟到的旧编辑可以保存为历史 Revision，但不得回退 `current_revision_no`。撤回设置可见状态并保存事件证据；是否删除内容由保留策略决定。已经用于 Context 的旧 Snapshot 仍指向当时 Revision。

## 7. 顺序

历史默认按 `receive_sequence, message_id` 排序，不依赖平台时间。平台时间作为展示字段。Thread/回复关系通过引用表达，不改变 Context Partition Lane 顺序。

## 8. 查询

查询使用基于 `(conversation_id,receive_sequence)` 的游标分页，支持 Session、时间、角色和内容类型过滤。普通 Query 默认不加载大型 Payload；需要时通过权限检查的 Payload API 获取。

## 9. Summary 与 Snapshot

Summary 记录覆盖的 Message 序号范围、源 Revision、模型/算法版本和 Payload。新 Message 或旧 Message 编辑不会原地修改旧 Summary；后台任务生成新版本。

## 10. 删除与保留

删除 Conversation 先从普通查询隐藏，再异步清理 Message Payload、附件、FTS、Embedding 和 Summary。MemoryItem 的来源链接转为 tombstone，不自动删除用户确认的长期事实，除非删除命令明确包含长期记忆。

## 11. 索引

唯一约束：平台入站键、`(conversation_id,receive_sequence)`、`(message_id,revision_no)`、`(message_id,revision_no,ordinal)`。FTS 仅索引允许的文本和当前 Revision，可从 Message 重建。

## 12. 测试

覆盖重复事件、相同文本重复发送、编辑乱序、撤回、附件失败、事务中断、分页稳定性、Summary 版本、删除级联和流式最终化。
