# 审批与命令

> **文档编号**：APPROVAL-COMMANDS  
> **适用范围**：Approval、Command、幂等、并发控制、冲突优先级与本地控制面  
> **权威边界**：本文是审批对象、Command Envelope、重复/竞态处理和冲突仲裁的权威来源。  
> **关联文档**：DOMAIN-CONTRACTS, RUNTIME-FLOWS, EXECUTION-LIFECYCLE, PROACTIVE-TASKS, SECURITY-OBS

## 1. Approval

Approval 是可被 Turn、Task 或独立管理命令引用的持久对象：

```text
approval_id
subject_type: turn | task | tool_call | command
subject_id
requester_attempt_id
action_summary
requested_permissions
risk_level
arguments_snapshot_ref
action_hash
policy_version
status: pending | approved | rejected | expired | cancelled
expires_at
allowed_responder_principal_ids
responder_principal_id
response_reason
version
created_at
responded_at
consumed_at
```

`requester_attempt_id` 只用于追踪发起来源，不是审批的生命周期所有者。等待期间原 RunAttempt 已结束，Approval 由其 `subject_type + subject_id` 恢复对应 Turn 或 Task。

### 1.1 审批展示内容

发送审批请求时，必须包含完整决策信息（非安全摘要）：

```text
将执行什么动作
作用目标（文件路径、API 端点、数据库表等具体标识）
关键参数（脱敏后的参数值或参数范围）
需要的权限（具体列表，非模糊的"高危权限"）
风险级别（low | medium | high | critical）
预计影响（可逆/不可逆，影响范围）
过期时间（审批在此时间后自动失效）
```

### 1.2 审批消费事务

同一事务内：

1. 验证 Approval 为 `pending` 且未过期；
2. 验证响应 Principal 在 `allowed_responder_principal_ids` 中；
3. 条件更新 Approval 版本和状态；
4. 验证 `action_hash`、参数快照与当前 Policy 一致；
5. 将目标 Turn/Task 改为 `queued`，或将管理 Command 标记可执行；
6. 写入 Audit 和 Event Outbox。

参数、目标或权限发生变化时，原 Approval 失效并创建新 Approval。已消费的 Approval 不得重复恢复同一 Turn 或 Task。

Tool Approval 还绑定 `capability_id/version`、完整规范参数 Payload 引用、64 位
SHA-256、Policy/Auto Mode 版本和 `constraints_json`。批准只能接受原约束，不能
扩大范围。消费以版本条件更新原子写入 `consumed_at`；Capability、Schema、参数
Hash、Policy 或约束变化时必须重新申请。拒绝、取消和过期结束原 Attempt，目标
保持等待态，直到显式 Command 取消或重新申请。

## 2. Command Envelope

所有改变状态的入口使用：

```text
command_id
command_type
idempotency_key
actor_principal_id
target_type
target_id
expected_version
payload_ref or inline_payload
created_at
expires_at
origin
trace_context
```

`idempotency_key` 由调用方在逻辑操作第一次发起时生成，网络重试必须复用。服务端生成新的 `command_id` 不能替代调用方幂等键。

唯一约束：

```text
(actor_principal_id, command_type, idempotency_key)
```

重复 Command 返回第一次处理结果，不再次执行副作用。相同幂等键携带不同 payload 时返回 `idempotency_conflict`，不得沿用旧结果或执行新请求。

Agent Tool 使用的 Schedule/Skill Command 固定 `origin=agent_tool`，携带 actor、
`expected_version`、payload ref 和 trace context；Handler 与 Audit、Outbox 在同一
事务提交。文件型 Skill 更新失败时同步回滚文件和数据库状态。

## 3. 并发控制与冲突仲裁

### 3.1 版本控制

涉及聚合状态的 Command 必须携带 `expected_version`。版本不匹配返回 `conflict` 和当前安全状态摘要，由调用方刷新后重新决定，不能静默覆盖。

例外：天然幂等的"确保状态"命令可以不要求版本，例如 `EnsureConnectorPaused`，但 Handler 仍需条件更新。

### 3.2 竞争操作优先级

审批、取消和重试可能同时发生（用户在审批页面点击"批准"的同一毫秒，Agent 侧 Tool 超时触发自动取消）。最终结果由数据库条件更新决定，不依赖请求到达应用进程的先后顺序。优先级：

```text
1. 已提交终态（completed / failed）
     → 任何后续操作返回 target_terminal，不做变更

2. 取消/过期
     → 审批响应（即使同时到达）无效
     → 取消优先于审批——用户已明确表示停止

3. 审批响应（approved / rejected）
     → 覆盖普通重试请求
     → 审批是用户显式决策，优先于系统自动行为

4. 普通重试
     → 仅在无取消、无审批、无终态时生效
```

### 3.3 条件更新的具体规则

```sql
-- 取消 Turn
UPDATE turns
SET status = 'cancelled', version = version + 1
WHERE turn_id = ? AND version = ? AND status NOT IN ('completed', 'failed', 'cancelled')

-- 审批消费
UPDATE approvals
SET status = ?, version = version + 1
WHERE approval_id = ? AND version = ? AND status = 'pending' AND expires_at > ?

-- 如果前置条件不满足（受影响行数 = 0），返回当前状态给调用方
```

## 4. 关键命令语义

```text
CancelTurn(turn_id, reason, expected_version)
RetryTurn(turn_id, expected_version)
Approve(approval_id, action_hash, expected_version)
Reject(approval_id, reason, expected_version)
RetryTask(task_id, expected_version)
ReplayDelivery(delivery_id, expected_version)
ConfirmMemory(memory_id, expected_version)
DeleteMemory(memory_id, expected_version)
```

`ReplayDelivery` 创建新的 Delivery 或新的显式重放代次，不把已发送 Delivery 直接改回 `sending`。

## 5. 本地控制面边界

当前部署目标是单机个人使用，不建设多租户认证系统。最低运行约束：

- Interaction API 和内部 Gateway API 默认只监听 loopback 或受权限控制的本地 Socket；
- 不允许通过配置无提示监听公网地址；
- 本地 Web 写请求校验 Origin，避免普通网页直接触发 loopback Command；
- Command 仍记录本地 Owner Principal、来源和 Audit；
- 一旦启用局域网或远程访问，必须先增加认证、TLS、会话失效和防重放设计。

这组轻量控制不替代 Tool Sandbox。Shell、文件、网络、MCP 和插件能力仍必须经过 Policy、权限 Scope、预算和隔离执行。
