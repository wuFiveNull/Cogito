# 流式投递

> **文档编号**：STREAMING-DELIVERY  
> **适用范围**：流式回复、Delivery、平台编辑、Receipt 与恢复  
> **权威边界**：本文是所有流式平台副作用及其 Delivery 生命周期的权威来源。  
> **关联文档**：DOMAIN-CONTRACTS, RUNTIME-FLOWS, ACCESS-DELIVERY, EXECUTION-LIFECYCLE

## 1. 核心原则

1. placeholder、append、replace 和 finish 都属于 Delivery Attempt，不允许 Stream Session 直接调用平台。
2. Delta 是临时展示数据，不是最终 Assistant Message。
3. 不要求逐 Token 写 SQLite，但每个已发生的平台副作用必须能关联到 Delivery、Attempt 和平台 Receipt。
4. 最终 Assistant Message 是内容事实源；Delivery 是发送状态事实源。

## 2. 对象模型

```text
Delivery
├── content_mode: provisional | final
├── final_message_id
├── stream_status
└── DeliveryAttempt
    ├── lease
    ├── operation_seq
    ├── last_confirmed_revision
    ├── platform_message_id
    └── receipts
```

Delivery 在第一次平台发送前持久化。流式过程中可以引用临时 Buffer；最终完成时绑定不可变 Assistant Message。

## 3. 状态机

以下状态并入 Delivery 通用状态机，并仅在选择流式策略时使用：

```text
pending → scheduled → sending → streaming → finalizing → sent
                         │          │           │
                         ├──────────┴───────────├→ interrupted
                         ├──────────────────────├→ retry_scheduled
                         └──────────────────────└→ failed/unknown
```

`unknown` 表示平台调用可能成功但本地未获得确认。必须先查询平台 Message ID、幂等键或执行 reconcile，不能直接发送新消息。

## 4. 流程

### 4.1 建立流

```text
创建 provisional Delivery
→ Delivery Worker 获取 Lease
→ Policy 检查目标与内容级别
→ start_placeholder Attempt operation
→ 保存 platform_message_id/Receipt
→ Delivery = streaming
```

### 4.2 更新

Delta 在内存中按时间或大小合并，形成受限频率的更新：

```text
append_delta / replace_content
→ operation_seq + 1
→ 平台调用
→ 保存确认的 revision、request_hash 和 Receipt
```

每个更新都复用稳定的 `delivery_id + operation_seq` 幂等键。平台不支持幂等编辑时，超时结果进入 `unknown`。

### 4.3 最终化

同一短事务内：

1. 写入不可变 Assistant Message；
2. Delivery 绑定 `final_message_id`；
3. 写入 finalization Outbox；
4. Turn 可以标记完成。

Worker 随后执行 `finish` 或最终 `replace_content`。平台不支持编辑时，根据已有 placeholder Receipt 选择补发最终消息或停止，不能无条件再发一条。

## 5. 崩溃恢复

Recovery Scan 对 `streaming`、`finalizing` 和 `unknown` Delivery：

1. 检查 Lease 是否失效；
2. 加载最后确认的 revision 和平台 Message ID；
3. 查询平台当前状态；
4. 如果最终 Message 已存在，尝试完成最终编辑；
5. 如果只有部分内容，按策略编辑为失败提示、保留部分结果或撤回；
6. 无法判断平台状态时进入人工检查或等待，不重复创建平台消息。

## 6. 能力降级

```text
原生流式       按平台流协议更新
支持编辑       placeholder + 批量 replace
不支持编辑     可选“正在处理”提示 + 最终独立 Delivery
仅最终发送     不创建 placeholder
```

降级方案在首次副作用前确定并写入 Delivery，运行中不静默切换为可能重复发送的模式。

## 7. 审计与指标

记录：

- 首 Token 时间和最终完成时间；
- 平台更新次数、限流和失败；
- placeholder 是否残留；
- 最终 Message 与平台 Message ID 的映射；
- 重复发送、未知状态和 reconcile 结果。
