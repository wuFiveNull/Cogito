# 流式投递

> **文档编号**：STREAMING-DELIVERY  
> **适用范围**：流式回复、Delivery、平台编辑、Receipt 与恢复  
> **权威边界**：本文是所有流式平台副作用及其 Delivery 生命周期的权威来源。  
> **关联文档**：DOMAIN-CONTRACTS, RUNTIME-FLOWS, ACCESS-DELIVERY, EXECUTION-LIFECYCLE

## 1. 核心原则

1. placeholder、append、replace 和 finish 都属于 Delivery Attempt，不允许 Stream Session 直接调用平台。
2. Delta 是临时展示数据，不是最终 Assistant Message。
3. 不要求逐 Token 写 SQLite，但每个已发生的平台副作用必须能关联到 Delivery、Attempt 和平台 Receipt。
4. 最终 Assistant Message 是内容事实源；Delivery 是发送状态事实源。

## 2. 对象模型

```text
Delivery
├── content_mode: provisional | final
├── final_message_id
├── stream_status: placeholder_created | streaming | done
├── turn_id                      # Plan 05 M5：关联拥有本 Delivery 的 Turn
└── DeliveryAttempt
    ├── operation_seq
    ├── last_confirmed_revision
    ├── platform_message_id
    └── receipts
```

Delivery 在第一次平台发送前持久化（status=`streaming`，content_mode=`provisional`）。
流式过程中可以引用临时 Buffer；最终完成时绑定不可变 Assistant Message。

> **所有权**：流式 Delivery 由 `AgentRunner.run_once` 在持有 Turn 的 RunAttempt Lease 期间创建并驱动，
> **不经过 DeliveryWorker**（Worker 只选择 `pending`/`sending`/`retry_scheduled`，永不选择 `streaming`）。
> 因此流式 Delivery 的生命周期与「拥有它的 Turn」严格绑定；崩溃恢复也以 Turn 状态为判定依据（见 §5）。

## 3. 状态机

流式 Delivery 实际使用以下子状态（并入 Delivery 通用状态机）：

```text
创建(provisional, status='streaming')
  → placeholder_created   # 首 token 已发送 "…" 占位
  → streaming             # 增量 edit 进行中
  → done (status='sent', content_mode='final')   # 定稿：绑定 final_message_id，Turn 完成
  └→ interrupted          # 取消 / 错误 / 崩溃撤回（status='interrupted', stream_status=NULL）
```

通用状态机中的 `sending`/`finalizing`/`unknown`/`retry_scheduled` 属于非流式 Worker 路径；
`finalizing` 与 `unknown` 在本模块预留但当前流式控制器只产出 `streaming → sent` 与 `streaming → interrupted`。
`unknown` 表示非流式平台调用可能成功但本地未获确认，需先 reconcile，不直接重发。

## 4. 流程

### 4.1 建立流

```text
run_once 判定走流式（channel 支持 edit+streaming 且 streaming_enabled）
→ StreamingDeliveryController.run_streaming_turn
  → 创建 provisional Delivery（status='streaming', content_mode='provisional', turn_id=当前 Turn）
  → 由当前 Turn 的 RunAttempt Lease 拥有（非 DeliveryWorker）
```

### 4.2 更新

Delta 在内存中按 `StreamPolicy` 节流合并（throttle_ms / max_operations），形成受限频率的更新：

```text
首 token  → gateway.send_text("…") 占位
          → mark_placeholder: stream_status='placeholder_created', 绑定 platform_message_id
后续 delta → gateway.edit(全量文本, operation_seq+1)
          → record_edit: stream_status='streaming', last_confirmed_revision=seq, 写 Receipt
```

每个 update 复用稳定的 `delivery_id + operation_seq` 幂等键。最终完成前会再推一次 `is_final=True` 的 edit，
保证前端即便在节流间隔内也收到完整全文。

### 4.3 最终化

同一短事务内（`StreamingDeliveryController._finalize`）：

1. 写入不可变 Assistant Message（content 事实源）；
2. Delivery 绑定 `final_message_id`，`content_mode='final'`，`status='sent'`，`stream_status='done'`；
3. 推送最终 `is_final` edit 到平台；
4. 写入 `TurnCompleted` Outbox，`dispatcher.complete` 标记 Turn 完成（发送状态事实源）。

取消或异常时调用 `withdraw`：`status='interrupted'`，不再发送，前端收到 `assistant.delete` 移除占位气泡。

## 5. 崩溃恢复（Plan 05 M5）

流式 Delivery 由 Turn 的 RunAttempt Lease 拥有。进程在流式过程中崩溃时，该 Delivery 会永久卡在
`status='streaming'`（平台已创建占位气泡），而其 Turn/RunAttempt 由 `recover_stale_turns` 标记为
`abandoned` / `queued`。`RecoveryService.recover_streaming_deliveries` 负责收拾这类孤儿：

1. 选出所有 `status='streaming'` 的 Delivery，经 `turn_id` 关联 Turn；
2. 孤儿判定（与 `recover_all` 中 `recover_stale_turns` 的先后配合）：
   - Turn 不存在，或 Turn 已不在 `running`（queued/completed/abandoned 等）→ 孤儿；
   - Turn 名义 `running` 但其 active RunAttempt 已不 `running`（崩溃但 Lease 尚未过期的边界态）→ 孤儿，并把 Turn 复位为 `queued` 以便重放；
   - Turn `running` 且 active RunAttempt 仍 `running`（存活的流式回合）→ 跳过；
3. 条件 UPDATE 撤回孤儿 Delivery：`status='interrupted'`、`stream_status=NULL`、`lease_version+1`
   （条件 `WHERE delivery_id=? AND status='streaming'`，与运行中的 `run_once` 不冲突）；
4. 若 Turn 因边界态被复位，则 `UPDATE turns SET status='queued'` 使其可重新被 Worker 领取并重放；
5. 重放时旧 Delivery 已 `interrupted`，新 `run_once` 会创建全新的 `streaming` Delivery 并定稿，
   不会出现「Turn queued 但存在合法 streaming Delivery」的误杀。

**前端占位清理（best-effort）**：`WebChannelAdapter.subscribe` 在订阅某会话时，查询本会话
`status='interrupted'` 的流式 Delivery，回灌 `assistant.delete` 事件，浏览器重连后删除遗留的 "…" 气泡。
该操作幂等（重复 delete 无害），无需标记已清理。

> 非流式 `unknown` Delivery（来自 DeliveryWorker）保持待对账，不在此处自动转回重试，需人工/平台 reconcile。


## 6. 能力降级

```text
原生流式       按平台流协议更新
支持编辑       placeholder + 批量 replace
不支持编辑     可选“正在处理”提示 + 最终独立 Delivery
仅最终发送     不创建 placeholder
```

降级方案在首次副作用前确定并写入 Delivery，运行中不静默切换为可能重复发送的模式。

## 7. 审计与指标

记录：

- 首 Token 时间和最终完成时间；
- 平台更新次数、限流和失败；
- placeholder 是否残留；
- 最终 Message 与平台 Message ID 的映射；
- 重复发送、未知状态和 reconcile 结果。
