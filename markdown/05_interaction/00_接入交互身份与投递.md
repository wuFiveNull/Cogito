# 接入、交互、身份与投递

> **文档编号**：ACCESS-DELIVERY  
> **适用范围**：Channel Gateway、Interaction、Identity、多用户隔离、PII redaction、Session 恢复、Delivery  
> **权威边界**：本文是其范围内的规范性来源；总体架构文档只负责概念与边界。  
> **细化规范**：LangBot 协议、Message 历史、Command 和流式发送分别以 `LANGBOT-BRIDGE`、`MESSAGE-PERSISTENCE`、`APPROVAL-COMMANDS`、`STREAMING-DELIVERY` 为准。  
> **关联文档**：AGENT-COGNITION, PROACTIVE-TASKS, SECURITY-OBS

## 阅读说明

**目的**：定义从消息平台进入系统到最终投递的完整接入边界。

**边界**：不包含 Agent 推理和主动策略算法。

**建议读取方式**：开发 Channel、Web UI、身份映射、Session 策略或发送逻辑时读取。

## 1. Channel Gateway

### 1.1 职责

Channel Gateway 负责：

- 启动和维护平台连接；
- 接收消息、按钮、引用、文件、图片、语音和平台事件；
- 将平台对象转换为 `ChannelEnvelope`；
- 生成稳定入站幂等键；
- 保留完整 Reply Route；
- 调用 Agent Core；
- 接收由 Agent Core 持久化后的 Delivery 操作；
- 转换为平台消息格式；
- 发送、编辑、撤回或更新平台消息；
- 上报发送结果和平台消息 ID；
- 处理平台限流、Token 过期和连接重建。

不负责：

- Prompt 构建；
- Memory；
- Agent 推理；
- 主动相关性判断；
- 业务任务编排；
- 直接操作核心数据库。

### 1.2 LangBot 集成边界

推荐把 LangBot 作为独立 Channel Gateway，而不是把其内部 Adapter 深度复制到 Agent Core。

桥接流程：

```text
LangBot Platform Event
→ LangBot Converter
→ Bridge Adapter
→ ChannelEnvelope HTTP/WebSocket
→ Agent Core
→ Delivery Stream
→ LangBot MessageChain
→ Platform
```

Bridge 只依赖版本化 Contracts。Agent Core 不接收 LangBot 内部 Message 对象。

平台账号、群成员和回复目标等平台身份细节由 LangBot Adapter 解析。Agent Core 不复制 LangBot 的平台对象模型，但必须在 `ChannelEnvelope` 中收到稳定 ID、发送者信息和完整 Reply Route；历史记录不得只保存显示名称或临时 SDK 对象。

### 1.3 ChannelDriver 接口

```python
class ChannelDriver(Protocol):
    async def start(self) -> None: ...
    async def stop(self) -> None: ...
    async def send(self, request: ChannelSendRequest) -> ChannelSendResult: ...
    async def edit(self, request: ChannelEditRequest) -> ChannelSendResult: ...
    async def delete(self, request: ChannelDeleteRequest) -> None: ...
    def capabilities(self) -> ChannelCapabilities: ...
```

`ChannelCapabilities` 包含：

```text
supports_streaming
supports_edit
supports_buttons
supports_threads
supports_files
supports_reactions
max_message_length
max_file_size
reply_token_ttl
rate_limit_profile
```

### 1.4 内容标准化

统一使用 `ContentPart`：

```text
TextPart
MarkdownPart
ImagePart
AudioPart
FilePart
LocationPart
QuotePart
ActionPart
StructuredPart
```

图片和文件的跨进程标准形式是 `payload_ref + sha256 + content_type + size`。
小型、受限的 base64 Data URI 只作为兼容入站形式，由 Core 立即物化到 Payload
Store 并从 Message 中移除原始二进制。Gateway 提供的普通 HTTP URL 不由 Core
自动抓取，避免 SSRF、授权泄漏和链接过期导致的不可重放输入。

平台不支持的 Part 由 Renderer 降级，例如：

- Markdown 转纯文本；
- 按钮转编号列表；
- 长消息分片；
- 文件转安全本地链接或附件；
- 流式输出退化为"正在处理"后编辑最终消息。

### 1.5 错误处理

错误分类：

```text
auth_error
rate_limited
route_expired
message_too_large
unsupported_content
temporary_network_error
permanent_platform_error
```

Gateway 返回结构化错误，不把 SDK 异常对象直接传播到 Agent Core。

## 2. Interaction

### 2.1 组成

Interaction 包含：

- Dashboard；
- Web Channel；
- Query API；
- Command API；
- 实时事件流；
- Approval UI；
- 配置和插件管理界面。

### 2.2 Query API

只读查询：

```text
GET /status
GET /channels
GET /conversations
GET /turns/{id}
GET /turns/{id}/attempts
GET /tasks
GET /tasks/{id}
GET /traces/{id}
GET /memory
GET /deliveries
GET /connectors
GET /plugins
GET /usage
```

Query API 只调用 Repository 的只读视图，不直接暴露数据库表结构。

### 2.3 Command API

可写命令：

```text
POST /commands/cancel-turn
POST /commands/retry-task
POST /commands/approve
POST /commands/reject
POST /commands/disable-plugin
POST /commands/pause-connector
POST /commands/replay-delivery
POST /commands/confirm-memory
POST /commands/delete-memory
```

所有命令：

- 接受调用方幂等键并生成 Command ID；
- 校验 Principal 和权限；
- 经过 Application Service；
- 写入 Audit；
- 不允许 Dashboard 直接更新业务表。

Command Envelope、`expected_version`、重复请求和冲突规则见 `APPROVAL-COMMANDS`。

### 2.4 实时流

WebSocket/SSE 事件：

```text
turn.updated
run.progress
span.completed
task.updated
delivery.updated
approval.created
channel.status_changed
connector.status_changed
```

实时流是观察通道，不是唯一事实源。断线后通过 Query API 重新同步。

### 2.5 Web Channel

Web Channel 在 UI 上属于 Interaction，在协议上仍实现 Channel 接口，与 Telegram、飞书使用相同 Turn Orchestrator。

## 3. Identity & Conversation

### 3.1 身份解析

输入：

```text
channel_type
channel_instance_id
platform_sender_id
platform_conversation_id
sender_endpoint_ref
conversation_endpoint_ref
```

输出：

```text
principal_id
sender_endpoint_id
conversation_endpoint_id
conversation_id
session_id
context_partition_key
identity_confidence
```

不允许仅凭显示名称合并身份。

### 3.2 PII Redaction

在构建 system prompt 时，部分身份标识需要脱敏后注入。脱敏仅在 prompt 文本层生效——路由、Session Key 和 Adapter 操作始终使用原始值：

```text
脱敏规则:
  user_id    → user_<sha256[:12]>
  chat_id    → <platform>:<sha256[:12]>  （保留 platform 前缀）
  显示名称   → 不脱敏（脱敏后的名称无意义且破坏 multi-user 可读性）

例外:
  - 只有本地 Renderer/Delivery Adapter 可以使用 raw ID 完成 @mention
  - 模型 Prompt 始终使用稳定别名或脱敏 ID，不因平台格式要求暴露 raw ID
```

脱敏在 Context Builder 装配 system prompt 时执行，不在数据库中修改存储值。

### 3.3 Endpoint 绑定

Endpoint 绑定到 Principal 的方式：

- 本地配置预绑定；
- 一次性验证码；
- Web Dashboard 确认；
- 平台账号白名单。

未绑定的外部用户默认使用受限 Principal，不共享 Owner 记忆和工具权限。

### 3.4 Session 策略

SessionPolicy 接口：

```python
class SessionPolicy(Protocol):
    async def resolve(self, conversation: Conversation, message: Message) -> SessionRef: ...
```

策略先按 Channel Instance、平台 Conversation、可选 Thread 和多用户隔离配置计算 `context_partition_key`，再按 `reset_generation` 创建或复用 Session。不同 Partition 不共享短期 Message、Summary 或 Context Snapshot，只共享经过 Principal/Scope 校验的长期 Memory、Preference 和 Goal。完整公式见 `SESSION-CONTEXT / 2. Session 解析`。

### 3.5 多用户隔离

群聊中多个用户发送消息时，隔离策略决定他们是共享同一个 Session 还是各自拥有独立 Session：

```yaml
# config.yaml
session:
  group_sessions_per_user: true     # 群聊中每个用户独立 Session（默认）
  thread_sessions_per_user: false   # Thread 中所有人共享 Session（默认）
```

| 场景 | 默认行为 | 配置控制 |
|------|---------|---------|
| DM | 始终私有（每人独立 Session） | 不可改变 |
| Group/Channel | 每用户独立 Session | `group_sessions_per_user` |
| Thread（论坛/子话题） | 共享 Session | `thread_sessions_per_user` |

共享 Session 时，system prompt 声明多用户会话；每条 Message 通过结构化 `sender_principal_id/sender_endpoint_id` 标识发送者。用于模型展示的发送者名称由 Context Builder 渲染，Gateway 不修改消息正文。

### 3.6 Context Partition Lane 实现

可使用数据库队列 + 进程内锁：

- 入站 Message 先持久化；
- 为 Conversation 写入待执行 Turn；
- Dispatcher 按 `context_partition_key` 只分配一个活动 Turn；
- Turn 完成或进入等待态后释放 Lane；
- Worker 崩溃后通过 Lease 超时恢复。

## 4. Delivery

### 4.1 职责

- 接收被动回复和主动通知；
- 选择或验证目标 Endpoint；
- 渲染平台内容；
- 分片；
- 处理限流；
- 发送和重试；
- 记录平台 Message ID；
- 支持编辑、撤回和状态更新；
- 反馈投递结果。

### 4.2 目标选择

被动回复：

```text
必须优先使用输入消息的 reply_route 快照
```

主动通知：

```text
DeliveryPolicy
→ preferred endpoint
→ channel availability
→ quiet hours
→ content capability
→ fallback endpoints
```

主动系统先产生 `DeliveryRequest` 或 `ScheduledDeliveryRequest`，其中只保存建议目标。真正进入 ready-to-send 时执行上述 Policy 并创建 Delivery，此时固定 `TargetSnapshot`。Delivery 创建后目标不再静默变化；Endpoint 失效时按快照中的 fallback policy 失败或创建新的 Delivery。保留原始建议目标和最终选择供审计。

### 4.3 Delivery 状态

```text
pending
scheduled
sending
streaming
finalizing
sent
partially_sent
interrupted
unknown
retry_scheduled
failed
cancelled
expired
```

### 4.4 Delivery Attempt

每次尝试记录：

```text
attempt_no
started_at
completed_at
channel_instance_id
request_hash
platform_message_id
error_code
retry_after
```

### 4.5 流式输出

所有流式平台操作都通过持久化 Delivery Attempt 执行。统一操作：

```text
start_placeholder
append_delta
replace_content
finish
```

具体平台根据能力选择：

- 原生流；
- 编辑同一消息；
- 周期性批量更新；
- 最终一次性发送。

Delta 可以批量合并且不逐 Token 入库，但 placeholder、编辑和最终化都必须记录 Delivery、Attempt、operation sequence 和平台 Receipt。崩溃恢复、未知状态对账和降级规则见 `STREAMING-DELIVERY`。

### 4.6 长消息和附件

Renderer 根据平台上限进行：

- 语义分片；
- 代码块完整性保护；
- 附件大小检测；
- 超大内容转文件；
- 敏感附件访问限制。

### 4.7 Gateway 重启与 Core 恢复

Gateway 只恢复平台连接和未确认 Delivery Attempt，不扫描或修改 Session、Turn、RunAttempt。Session/Turn 恢复完全由 Agent Core 根据 SQLite 中的状态、Checkpoint、Lease 和 Receipt 决定。

#### 4.7.1 Gateway 启动

```text
Gateway 启动
→ 恢复 Channel 连接
→ 向 Agent Core 上报 GatewayStarted(instance_id, started_at, previous_stop_kind)
→ 拉取/接收待处理 Delivery
→ 对 unknown Delivery Attempt 执行平台级 reconcile
```

`.clean_shutdown` 可以作为 Gateway 的诊断信号，用于生成 `previous_stop_kind`，但不是业务事实源，不能据此重置 Session 或改变 Turn 状态。

#### 4.7.2 Agent Core Recovery Scan

Core 启动或收到 GatewayStarted 后检查：

```text
running Turn/RunAttempt with invalid execution right
→ 按 EXECUTION-LIFECYCLE 标记旧 Attempt abandoned
→ 检查 Checkpoint 和 Tool Receipt
→ queued / waiting_external / manual_review

sending/streaming/finalizing/unknown Delivery
→ 按 STREAMING-DELIVERY 检查平台 Receipt
→ reconcile / retry_scheduled / manual_review
```

Gateway 不能直接写 Core SQLite；所有结果通过版本化 Delivery Result Contract 上报。

#### 4.7.3 Session 重置

Session 只因以下业务事件创建新 generation：

- 用户显式 `/new`、`/reset`；
- 已配置的 idle/daily reset policy；
- Core 判定当前 Context Partition 连续恢复失败并通过 `ResetSession` Command 执行。

进程或 Gateway 崩溃本身不会创建新 Session。连续恢复失败的计数属于 Turn/RunAttempt 恢复状态并持久化到 SQLite，不使用 Gateway 本地文件计数。

#### 4.7.4 Drain Timeout

优雅关闭超时时，Core 将仍在运行的 Attempt 写入取消/中断状态或保留可恢复 Checkpoint；Gateway 仅停止接受新平台操作，并把无法确认的发送标记为 unknown。下次启动分别由 Core 和 Delivery Worker 恢复，不依赖“下一条用户消息”触发。
