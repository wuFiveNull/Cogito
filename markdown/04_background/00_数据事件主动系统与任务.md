# 数据、事件、主动系统与任务

> **文档编号**：PROACTIVE-TASKS  
> **适用范围**：Connector、Event、三路数据源、Proactive Context、Candidate、Proactive Decision、能量模型、Task、Scheduler、Digest  
> **权威边界**：本文是其范围内的规范性来源；总体架构文档只负责概念与边界。  
> **细化规范**：Task、Event、Connector 和主动/Drift 分别以 `TASK-SCHEDULER`、`EVENT-OUTBOX`、`CONNECTOR-INGESTION`、`PROACTIVE-IDLE` 为准。  
> **关联文档**：ACCESS-DELIVERY, AGENT-COGNITION, STORAGE-DATA, SECURITY-OBS

## 阅读说明

**目的**：定义外部数据摄取、事件生成、主动决策和持久任务调度。

**边界**：不定义 Tool/MCP 的执行实现，也不定义 Channel 的发送协议。

**建议读取方式**：开发 Connector、Scheduler、Digest、主动推送或长期任务时读取。

## 1. Data & Event

### 1.1 Source Connector

Connector 只负责外部数据进入系统，与用于执行动作的 Tool 分开。

Connector 类型：

```text
polling
webhook
stream
file_watch
manual_import
```

接口：

```python
class SourceConnector(Protocol):
    async def poll(self, cursor: ConnectorCursor) -> ConnectorBatch: ...
    async def acknowledge(self, batch_id: str) -> None: ...
    def capabilities(self) -> ConnectorCapabilities: ...
```

### 1.2 三路数据源分类

Connector 摄入的数据进入 Proactive Decision 前，按性质分为三路。不同路由的处理策略和延迟要求不同：

```text
alert（高优先级告警）
  来源：安全告警、服务中断、异常事件、用户设置的阈值触发
  处理：跳过逐条 LLM 评分，hard safety gate 通过后直接决策
  延迟目标：< 30s

content（内容流）
  来源：RSS 新文章、邮件、GitHub 通知、日历事件、新闻
  处理：逐条 novelty + relevance 评估，LLM 可选评分
  延迟目标：1-5min

context（背景上下文）
  来源：天气变化、股价波动、系统健康指标、长期趋势
  处理：不逐条推送。按概率注入 Proactive Loop 作为决策背景
        或作为 fallback——当 alert 和 content 均无可推送时，
        从中选取最显著的信号供 LLM 判断
  延迟目标：与 Proactive tick 同频
```

每路数据在 Connector 注册时声明默认分类，用户可在配置中覆盖特定来源的分类。

三路分类影响的是处理路径——alert 不排队、不冷却、不受 Quiet Hours 限制（但仍受 hard safety gate 约束）。

### 1.3 Connector Cursor

保存：

```text
connector_id
cursor_value
last_success_at
last_attempt_at
consecutive_failures
next_poll_at
lease_owner
```

Cursor 只在原始数据成功归档和事件写入同一事务后推进。

### 1.4 原始归档

外部数据首先写 Payload Store，再写 Event 记录：

```text
raw_payload_ref
source_item_id
source_updated_at
content_hash
```

这样 Normalize 逻辑升级后可以重放，而无需重新请求外部系统。

### 1.5 标准化与去重

去重键优先级：

1. 来源提供的稳定 ID；
2. 来源 ID + 更新时间；
3. 规范化内容哈希；
4. 模糊相似度，仅用于提示，不作为唯一硬去重。

### 1.6 Event Store

Event 是不可变事实。纠正错误通过新事件表达：

```text
ConnectorItemDiscovered
ConnectorItemUpdated
ConnectorItemRetracted
```

不直接修改旧事件含义。

### 1.7 Event Consumer

每个 Consumer 保存：

```text
consumer_name
event_id
status
attempts
processed_at
error
```

Handler 必须幂等。失败不会阻塞其他 Consumer。

## 2. ProactivePolicy 与 Context 视图

### 2.1 职责

`ProactivePolicy` 是 SQLite 中的权威聚合；`PROACTIVE_CONTEXT.md` 是其人工可读派生视图。主 Agent 通过 Command 更新聚合，Proactive Agent 每轮读取版本化 Policy。

Proactive Agent 将聚合中的规则视为必须遵守的 Owner Policy；Markdown 文件不能绕过导入校验直接改变运行规则：

```markdown
# Proactive Context

在这里写用户当前对主动推送的明确要求和规则。

- 白名单/黑名单：哪些主题可以推、哪些不可以
- 过滤条件：只在特定条件下推送某类内容
- 优先级规则：什么情况下提升/降低优先级
- 验证步骤：推送前必须确认的事项

只写结论，不写冗长过程。这份文件是你（主 Agent）和 Proactive Agent
之间的协议——写得模糊，Proactive Agent 就会做错。
```

### 2.2 维护规则

- 主 Agent 在用户表达推送偏好时提交 `UpdateProactivePolicy` Command；
- 投影任务从 SQLite 聚合渲染文件，Proactive Agent 不修改文件；
- Proactive Agent 将版本化聚合作为硬约束，文件仅用于人工查看；
- 人工编辑文件必须通过 Import Command 校验、版本检查和 Audit；
- Preference 与 ProactivePolicy 冲突时，以更具体、更新的显式 Policy 为准并保留解释。

## 3. Proactive Decision

### 3.1 职责

主动决策输入：

```text
三路数据（alert / content / context）
Memory Summary
Preferences
Goals
Proactive Context 文件
Current Context
Delivery History
Push Budget
Quiet Hours
能量模型输出
```

输出：

```text
send_now
send_later
add_to_digest
silent_process
create_task
ask_permission
discard
```

### 3.2 决策顺序

```text
1. alert 快速通道（跳过 2-5，直接进入 7）
2. hard safety gate
3. source allow/deny policy
4. duplicate and novelty gate
5. user relevance rule
6. 能量模型调整 urgency
7. timing and quiet hours
8. daily/topic/channel budget
9. optional model scoring
10. decision aggregation
11. render and delivery
```

LLM 不应绕过硬规则。

### 3.3 能量模型

能量模型为主动决策提供一个自适应的"互动饥渴度"信号，替代单一的固定轮询间隔配置。

```python
def compute_energy(last_user_at: datetime, now: datetime) -> float:
    """三时间尺度指数衰减。
    τ₁=30min  短时：对话余温
    τ₂=240min 中时：同一天语境
    τ₃=2880min 长时：关系连续性（48h）
    """
    t = max(0.0, (now - last_user_at).total_seconds() / 60.0)
    return (
        0.50 * math.exp(-t / 30.0)
        + 0.35 * math.exp(-t / 240.0)
        + 0.15 * math.exp(-t / 2880.0)
    )
```

能量在 `[0, 1]` 之间：

- **高电量**（≥ 0.7，刚聊完 ≤ 15min）：降低 Proactive tick 频率（8min 间隔），减少推送，提高 novelty/relevance 阈值；
- **中电量**（0.3-0.7，聊完 15min-4h）：标准频率（2-4min 间隔），正常阈值；
- **低电量**（< 0.3，超过 4h 未互动）：提高频率（1min 间隔），降低阈值——用户久未互动时更主动。

能量不直接决定"推不推送"，而是调整 **urgency 权重** 和 **Proactive tick 间隔**。高电量时不禁止推送，但要求更高的信号质量。

### 3.6 摘要

`add_to_digest` 写入 Digest Bucket：

```text
digest_date
topic
candidate_ids
priority
```

摘要任务负责合并、排序和去重，不能简单拼接所有消息。

### 3.7 Dry Run

系统支持全局或按策略 `dry_run`：

- 执行完整判断；
- 不创建真实 Delivery；
- 在 Dashboard 展示"本应发送"的内容和原因；
- 记录用户人工评价用于调节策略。

### 3.8 反馈

主动通知反馈：

```text
opened
ignored
dismissed
useful
not_useful
muted_topic
requested_more
```

反馈只产生 Preference Candidate，不直接无限增强或永久禁用主题。

## 4. Task & Scheduler

### 4.1 Task Handler

```python
class TaskHandler(Protocol):
    async def execute(self, context: TaskContext, payload: object) -> TaskOutcome: ...
```

TaskOutcome：

```text
Complete(result)
Retry(error, next_at)
WaitUser(approval_request)
WaitExternal(condition, next_check_at)
Spawn(children)
Cancel(reason)
Fail(error)
```

### 4.2 Lease

Worker 获取任务时原子更新：

```text
status = running
lease_owner = worker_id
lease_expires_at = now + lease_duration
heartbeat_at = now
attempt = attempt + 1
```

Lease 到期且无 Heartbeat 时，任务进入可恢复队列。

### 4.3 Retry Policy

```text
max_attempts
initial_delay
backoff_multiplier
max_delay
jitter
retryable_error_codes
```

永久错误不重试。副作用未知结果先 Reconcile，再决定重试。

### 4.4 Scheduler

支持：

```text
one_time
interval
cron
calendar_aware
condition_watch
```

Schedule 与 Task 分离。每次触发生成新的 Task，使用：

```text
schedule_id + scheduled_fire_time
```

作为幂等键，防止重启后重复触发。

### 4.5 调度表达式

除标准 5-field cron 外，调度器还支持更易读的自然语言格式：

```text
Duration:     "30s" / "5m" / "2h" / "1d" / "1h30m"
Every 短语:   "every 2h" / "every monday 9am" / "every day 08:00"
ISO 时间戳:   "2026-07-15T09:00:00Z"  （one-shot）
```

解析顺序：ISO 时间戳 → "every" 短语 → Duration → cron 表达式。Duration 在内部转换为 interval 类型的 Schedule。

### 4.6 Waiting User

等待用户的 Task 保存：

```text
approval_id
resume_token
expires_at
allowed_responses
resume_checkpoint
```

Approval 使用 `subject_type=task`、`subject_id=task_id` 关联 Task；发起它的 Attempt 只作为 Trace 来源。响应消费、参数快照、幂等和版本检查见 `APPROVAL-COMMANDS`。响应过期后按 Policy 取消、失败或重新询问。

### 4.7 子任务

父任务可以创建子任务，但必须声明：

```text
join_policy: all | any | none
failure_policy: fail_parent | continue | compensate
```

防止隐式递归创建无限任务树。
