# 主动推送与后台空闲处理

## 1. 三路数据源

Connector 摄入的外部数据进入 Proactive Decision 前，按性质分为三路。不同路由处理策略不同：

### 1.1 alert（高优先级告警）

```text
来源: 安全告警、服务中断、异常事件、用户设置的阈值触发、紧急日历提醒
处理:
  - 跳过逐条 LLM 评分
  - hard safety gate 通过后直接进入 decision aggregation
  - 不受 quiet_hours、cooldown、daily budget 限制
  - 不受能量模型降频影响（alert 在任何电量下均可推送）
延迟目标: < 30s
数量限制: 默认每小时最多 5 条（可配置），超出降级为 content
```

### 1.2 content（内容流）

```text
来源: RSS 新文章、邮件、GitHub 通知、日历事件、新闻、社交媒体更新
处理:
  - 逐条经过完整 decision pipeline
  - novelty + relevance 评估
  - LLM 可选评分
  - 受 quiet_hours、cooldown、budget 完整约束
  - 能量模型调整 urgency 权重
延迟目标: 1-5min
```

### 1.3 context（背景上下文）

```text
来源: 天气变化、股价波动、系统健康指标、长期趋势、统计摘要
处理:
  - 不逐条推送
  - 按概率加入 `External Untrusted Context` 数据段作为决策背景，不进入 System Policy
  - 当 alert 和 content 均无可推送时（idle fallback）：
    从中选取最显著的 1-2 条信号供 LLM 判断"是否有值得一提的变化"
  - 本身不直接产生推送——LLM 判定值得推送时才创建 Candidate
延迟目标: 与 Proactive tick 同频
```

### 1.4 分类配置

每路数据在 Connector 注册时声明默认分类。用户可在配置中覆盖：

```yaml
proactive:
  streams:
    alert_sources: ["security-monitor", "uptime-checker"]
    content_sources: ["rss-feeds", "github-notifications", "email"]
    context_sources: ["weather", "stock-portfolio", "system-health"]
  alert:
    max_per_hour: 5
    require_acknowledgment: true
```

## 2. ProactivePolicy 与 Markdown 视图

### 2.1 文件位置与格式

主动规则的权威对象是 SQLite 中的 `ProactivePolicy`：

```text
policy_id
principal_id
allow_topics / deny_topics
filters / quiet_hours / cooldown / budgets
version
updated_by / updated_at
```

`~/.cogito/workspace/PROACTIVE_CONTEXT.md` 是该聚合的人工可读派生视图，格式示例：

```markdown
# Proactive Context

在这里写用户当前对主动推送的明确要求和规则。

## 白名单（可以推的主题）
- 用户明确关注的 GitHub 仓库的 release
- 与当前 Goal 相关的进展

## 黑名单（不要推的主题）
- 新闻类内容（用户说最近不想看新闻）
- 低优先级的 CI 通知

## 过滤条件
- 同一主题一天内最多推 1 次
- 周末不推工作相关的内容

## 验证步骤
- 推送前检查是否与上次推送重复
```

### 2.2 维护规则

- **主 Agent**在用户表达推送偏好时提交 `UpdateProactivePolicy` Command，不直接写文件；
- Command 更新 SQLite、写 Audit/Outbox，再由投影任务原子渲染 Markdown；
- **Proactive Agent**每轮读取版本化 ProactivePolicy，将其作为 Owner Policy 注入；
- Markdown 被人工编辑时必须通过显式 Import Command 校验、展示 Diff 并更新聚合，不能直接覆盖运行时规则；
- 文件缺失或损坏时从 SQLite 重建，不改变实际 Policy；
- ProactivePolicy 与普通 Preference 冲突时，以更具体且版本更新的显式 ProactivePolicy 为准，并记录冲突解释。

## 3. 能量模型

### 3.1 目的

能量模型为 Proactive Loop 提供一个自适应的"互动饥渴度"信号。它根据用户最近一次互动的时间，动态调整 Proactive tick 频率和推送阈值——替代固定的轮询间隔配置。

### 3.2 计算公式

```text
E(t) = α·exp(-t/τ₁) + β·exp(-t/τ₂) + γ·exp(-t/τ₃)

τ₁=30min   短时：对话余温
τ₂=240min  中时：同一天语境连续性
τ₃=2880min 长时：关系连续性（48h）

α=0.50, β=0.35, γ=0.15
```

`last_user_at` 取最后一次收到用户消息的时间；从未收到消息时 E=0.0。

### 3.3 行为调整

| 电量区间 | tick 间隔 | urgency 权重 | novelty 阈值 | 行为描述 |
|---------|----------|-------------|-------------|---------|
| 高（≥ 0.7） | 8min | ×0.5 | 高 | 用户刚聊完，保持安静 |
| 中（0.3-0.7） | 2-4min | ×1.0 | 标准 | 正常主动频率 |
| 低（< 0.3） | 1min | ×1.5 | 低 | 用户久未互动，更主动 |

能量不直接决定"推不推送"——它只调整参数。高电量时仍可以推送 alert 或经用户明确要求的内容；低电量时仍需要满足最低 novelty 要求。

### 3.4 与 Quiet Hours 的关系

能量模型和 Quiet Hours 是正交的两个维度：

- **能量**：控制"用户互动热度"→ 调整推送频率和阈值；
- **Quiet Hours**：控制"时间段"→ 在指定时间内将非 alert 内容转为 `send_later` 或 Digest。

两者同时生效：低电量 + Quiet Hours = 不推送（Quiet Hours 优先级更高）；低电量 + 非 Quiet Hours = 降低阈值，更积极推送。

## 4. 主动候选

Candidate 来源于 SourceEvent、Task 结果、Goal 截止期和系统健康告警：

```text
candidate_id
source_ids
principal_id
stream_type               — alert | content | context
topic
summary
novelty/relevance/urgency/confidence
recommended_action
expires_at
policy_version
```

候选创建幂等键由来源事实和候选类型组成。

---

## 5. 决策顺序

```text
1. alert 快速通道（alert 类跳过 2-5）
2. hard safety/source policy
3. duplicate/novelty
4. principal relevance
5. 能量模型调整 urgency
6. urgency/expiry
7. quiet hours/cooldown
8. daily/topic/channel budget
9. optional model score
10. deterministic aggregation
```

输出 `send_now | send_later | digest | silent | create_task | ask_permission | discard`。模型评分不能覆盖硬规则。

---

## 6. 发送与 Digest

`send_now` 立即提交 DeliveryRequest；`send_later` 创建持久 ScheduledDeliveryRequest/Task，只保存建议目标和原因。到达可发送时间时重新执行 DeliveryPolicy，随后创建 Delivery 并固定 TargetSnapshot。Digest Bucket 以 Principal、日期和主题分组，同一 Candidate 被消费后不得再次即时发送，除非新 Event 明确升级紧急程度。

content 类候选使用 digest 默认最大延迟 6h（可配置）。alert 类不进入 digest——立即发送或丢弃。

---

## 7. Quiet Hours 与冷却

紧急级别必须由确定性规则或用户配置产生。alert 不受 Quiet Hours 限制。普通通知在 Quiet Hours 转为 `send_later` 或 Digest；同主题冷却使用最后实际发送时间，不使用候选创建时间。

---

## 8. 反馈

打开、忽略、关闭、有用、无用、静音主题等反馈写 FeedbackEvent。反馈生成 Preference Candidate，不直接修改不可解释的永久权重。策略调整保留版本和回滚点。

---

## 9. 空闲判定

系统只有同时满足以下条件才进入 Drift：

- 无高优先级 Turn/Approval/Delivery backlog；
- 即时模型和 Tool 并发低于保留阈值；
- SQLite/Payload 健康；
- 不在关闭或恢复阶段；
- Drift 日预算和单次时间仍可用。

空闲不是"CPU 当前为零"，而是资源管理器给出的可撤销执行许可。

---

## 10. Drift 任务目录

允许：Memory 去重建议、Embedding 补全、索引重建、旧摘要压缩、Payload GC 扫描、候选质量统计、ProactivePolicy 视图一致性检查。默认禁止：发送消息、修改外部系统、确认 Memory、删除用户数据和安装插件。

每类 Drift 使用正常 Task/Checkpoint，不创建隐藏线程或无限循环。

---

## 11. 抢占

新用户 Turn 到达时，Drift 停止领取新步骤；当前步骤在安全点写 Checkpoint 并释放资源。不可中断本地原子操作可以完成，但不得开始外部副作用。

---

## 12. Dry Run

主动策略支持全局、Principal 和规则级 dry-run。保存本应采取的动作、内容预览、规则结果和预算估算，但不创建真实 Delivery 或外部副作用。

---

## 13. 指标与测试

指标：三路候选比例、候选→发送转化、alert 延迟、能量模型分布、重复抑制、Digest 命中、反馈、Quiet Hours 延迟、Drift 时长/抢占和资源影响。

测试覆盖：alert 快速通道不受 Quiet Hours 阻断、能量模型数值边界（刚发完/1h/8h/48h/无消息）、context fallback 触发、预算边界、跨午夜、冷却、紧急升级、重复 Event、用户到达抢占、dry-run 零副作用、Markdown 缺失重建和人工编辑导入冲突。
