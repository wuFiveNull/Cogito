# 长期记忆生命周期

## 1. 双层存储架构

系统记忆分为两层，数据库是唯一权威事实源：

```text
数据库层（SQLite，权威事实源）
  ├── memory_items         — 完整字段、状态机、事务保证
  ├── memory_relations     — 冲突/替代/细化关系链
  └── memory_embeddings    — 派生向量索引

Markdown 层（人工可读、可重建的派生视图）
  ├── MEMORY.md            — 已确认 fact + preference 的稳定摘要
  ├── SELF.md              — 版本化 AgentProfile/Policy 的派生说明
  ├── PENDING.md           — 待确认候选清单
  ├── HISTORY.md           — Episode 时间线（只追加）
  └── RECENT_CONTEXT.md    — 近期对话压缩（不属于长期记忆，共享目录）
```

关键原则：

- 数据库写入先行，Markdown 滞后派生，不一致时以数据库为准；
- Context Builder 的动态检索结果是记忆注入的唯一选择路径；
- MEMORY.md 只作为人工查看和可选稳定摘要缓存，不能覆盖或替代实时 MemoryResult；
- PENDING.md 是 candidate 的派生清单，不是数据库 Commit 的缓冲层。

## 2. 记忆构成

```text
fact        可验证事实
preference  用户偏好，通常具有时间性
episode     对未来可能有用的经历摘要
goal        有状态、优先级和期限的目标
constraint  用户明确要求长期遵守的限制
```

Conversation Summary 属于短期压缩，不是长期 MemoryItem。

## 3. MemoryItem

在 `DOMAIN-CONTRACTS` 基础上增加：

### 3.1 核心标识

```text
canonical_key         — principal + scope + kind + normalized subject + predicate
                        用于候选查重，不作为绝对事实唯一键
version               — 乐观锁版本号
```

### 3.2 来源与置信

```text
importance            — 提取时或 Consolidation 时评估（0.0-1.0）
source_trust          — 来源可信度（0.0-1.0），由 explicitness 映射，不单独存储
confirmation_method   — auto | user_affirmed | user_explicit
confirmed_by          — 确认者 principal_id
confirmed_at          — 确认时间
```

**来源集合**：一个 MemoryItem 可关联多条来源（`memory_sources` 一对多）。每条来源携带 `source_type`（message/event/task/connector_item/knowledge_resource/manual）、`source_id`（指向稳定 ID，不写显示名或路径）、`source_revision`、`evidence_ref`（指向最小证据，不复制 Secret）、`evidence_hash`、`trust_label`。`memory_items.source_type/source_id` 在过渡期保留，降级为 primary source cache。一个 consolidation 产物可关联多个来源（参见 DOMAIN-CONTRACTS §1.13）。

### 3.3 检索强化

```text
reinforcement         INTEGER DEFAULT 0      — 仅由明确正向信号增加
exposure_count        INTEGER DEFAULT 0      — 被召回展示次数，只用于评估，不提升权重
emotional_weight      INTEGER DEFAULT 0      — 0-10，允许因纠正、负反馈或重评降低
last_retrieved_at     TEXT                   — 最近一次被检索命中的时间
retrieval_count       INTEGER DEFAULT 0      — 累计被检索命中次数
```

### 3.4 衰减与权重

```text
decay_rate            REAL                   — 按 kind 取默认值，单位 /天
retrieval_weight      REAL DEFAULT 1.0       — 动态计算的综合检索权重
last_weight_update    TEXT                   — 上次权重计算时间
```

### 3.5 Embedding

```text
embedding_model       TEXT
embedding_version     TEXT
```

### 3.6 Goal 结构化字段

`kind=goal` 使用 `goal_status`、`goal_priority`、`goal_deadline`、`goal_progress`；MemoryItem 自身的 `status` 仍表示候选/确认生命周期。两组状态不能混用：只有 `status=confirmed AND goal_status=active` 的 Goal 进入活动视图。

## 4. 检索权重算法

`retrieval_weight` 由后台维护任务定期重算，不是查询时实时计算。以下公式是**唯一权威版本**，代码 `compute_retrieval_weight()` 纯函数实现必须与此一致（含参数名与系数）。

### 4.1 公式

```text
base_score = importance * 0.6 + source_trust * 0.2 + confirmation_score * 0.2
decay_factor = exp(-kind_decay_rate * days_since_last_active)
reinforcement_bonus = min(0.4, log(1 + reinforcement) * 0.1)
emotional_bonus = min(0.2, emotional_weight * 0.02)

retrieval_weight = clamp(
    base_score * decay_factor
    + reinforcement_bonus
    + emotional_bonus,
    0.0, 2.0
)
```

### 4.2 参数定义

| 参数 | 来源 | 默认值/映射 |
|------|------|------------|
| `importance` | MemoryItem.importance | 0.0-1.0，提取时评估 |
| `source_trust` | 由 explicitness 映射，不单独存储 | explicit_user_statement=1.0, confirmed_inference=0.9, external_source=0.7, system_generated=0.6, model_inference=0.4 |
| `confirmation_score` 由 status/confirmation_method 推导 | confirmed=1.0, candidate=0.5, rejected/expired=0.0 |
| `kind_decay_rate` | 按 kind 取 §5.1 默认值，支持 per-item decay_rate 覆盖 | 见 §5.1 表 |
| `days_since_last_active` | 最近一次 user_affirmed / task_succeeded / 事实更新距今 | 普通召回**不**刷新 |
| `reinforcement` | 由 memory_signals 累加聚合（仅 user_affirmed/task_succeeded/user_corrected 贡献） | 初始 0 |
| `emotional_weight` | MemoryItem.emotional_weight | 0.0-1.0，可因纠正/负反馈降低 |

### 4.3 语义约束

- `days_since_last_active` 取最近一次用户明确引用、成功 Task 依赖或事实更新时间；普通召回（含 recall_memory）不刷新 active 时间。
- 表中的”半衰期”对应指数衰减 `ln(2)/kind_decay_rate`。
- `memory_items.retrieval_weight` 是缓存写回值，记录 `last_weight_update` 与 `algorithm_version`，唯一事实是 memory_signals 追加表。

## 5. 衰减速率与遗忘策略

### 5.1 分 kind 衰减

| kind | decay_rate/天 | 无访问半衰期 | 说明 |
|------|-------------|------------|------|
| fact | 0.001 | ~693 天 | 事实不因时间变假，仅降检索优先级 |
| preference | 0.005 | ~139 天 | 偏好随时间漂移，最近明确表达优先 |
| episode | 0.02 | ~35 天 | 经历细节快速失去检索价值 |
| goal | 0（活跃） / 0.1（完成后） | — | 活跃不衰减，完成后快速退出检索 |
| constraint | 0 | 无限 | 只有用户或明确 Policy 能取消 |

### 5.2 遗忘分档

```text
retrieval_weight < 0.1  → archived（不进入默认检索，保留完整数据+来源+关系链）
retrieval_weight < 0.05 → 进入遗忘候选队列，等待用户确认后过期
```

遗忘**不删除数据**。仅在用户明确删除时（§12）才清理 Embedding 和派生 View，保留最小 audit tombstone。

### 5.3 Reinforcement 规则

强化信号使用追加式 `memory_signals` 表，`reinforcement` 为聚合投影（非唯一事实）。统一分类如下：

```text
exposed（进入 Context 或工具结果）:
  exposure_count += 1, retrieval_count += 1
  reinforcement 不变
  写入 signal_type="exposed"

referenced（模型在最终结果中引用但用户未确认）:
  reinforcement 不变，记录 usage
  写入 signal_type="referenced"

user_affirmed（用户明确确认有用）:
  reinforcement += 2
  last_retrieved_at = now
  写入 signal_type="user_affirmed"

task_succeeded（成功 Task 有可验证依赖）:
  reinforcement += 1
  写入 signal_type="task_succeeded"

user_corrected（用户纠正并确认新事实）:
  新事实 reinforcement += 2，旧事实不强化
  写入 signal_type="user_corrected"

negative_feedback（用户负面反馈）:
  可降低 emotional_weight/quality 信号
  不产生负 reinforcement 溢出
  写入 signal_type="negative_feedback"
```

约束：

- 普通召回（含 `recall_memory`）本身是 `exposed`，**不增加 reinforcement**——这与 RETRIEVAL-CONTEXT 一致。
- 主动调用 `recall_memory` 本身不是正向信号。
- 周期性衰减（每 30 天）仅对 last_retrieved_at 超过 30 天的条目：`reinforcement = max(0, reinforcement - 1)`。
- emotional_weight 可根据纠正、负反馈、重评降低。
- reinforcement 有上限，用户确认过的记忆不低于 1。

## 6. 候选产生

候选来源：用户明确陈述、用户纠正、完成的 Task、受信 Connector 事实和重复观察。Tool 输出或外部网页默认不能自动成为 Owner 事实。

仅当内容满足至少一项时提取：

- 对未来行为有持续影响；
- 用户明确要求记住；
- 是稳定偏好、约束或目标；
- 后续任务恢复必须依赖；
- 多次出现且具有明确主体。

寒暄、一次性请求、模型推测、Secret 和无来源总结默认不提取。

## 7. 写入流程

```text
source committed
→ extract candidate
→ schema/PII policy
→ normalize canonical key
→ exact and semantic duplicate check
→ relation classification
→ auto-confirm | pending user | reject
→ write MemoryItem + relation + Outbox
→ async embedding
→ 更新 PENDING.md（候选可见）
```

Candidate 与源 Message/Event 在同一事务或通过幂等 Event 派生，重复消费不得生成重复候选。

## 8. 确认策略

自动确认仅适用于：

- Owner 明确说"记住……"；
- 系统确定性产生的任务状态；
- 已验证 Connector 的结构化事实且策略允许。

模型推断、敏感事实、身份合并、可能影响高风险行为的偏好需要用户确认。确认时记录方法、Principal、时间和参数快照。

## 9. 冲突与关系

```text
supports
contradicts
supersedes
refines
derived_from
```

新事实不原地覆盖旧事实。明确更新使用 `supersedes` 并结束旧项 `valid_to`；无法判断时两者并存并降低检索优先级，必要时询问用户。

**与 reinforcement 联动**：被 supersede 的旧条目退出默认检索；冲突解决本身不自动增加 reinforcement，只有用户确认或成功业务使用才强化。

## 10. Consolidation

Consolidation 是定期后台维护，不是每条消息之后。任务包括：

```text
Consolidation 任务
├── 合并重复 Episode（保留所有 source_id，写新 MemoryItem）
├── 重算 retrieval_weight
├── 周期性 reinforcement 衰减（每 30 天）
├── 归档 retrieval_weight < 0.1 的条目
├── 生成遗忘候选（retrieval_weight < 0.05）
├── 刷新 Markdown 视图（§11）
└── 记录算法版本和变更 audit
```

必须遵守：

- 保留所有 source_id；
- 写入新 MemoryItem，不原地修改；
- 记录算法/模型版本；
- 不把未确认候选升级为 confirmed；
- 支持从原事实重建。

## 11. Markdown 视图与 Prompt Cache 保护

### 11.1 文件职责

| 文件 | 来源 | 触发条件 | 约束 |
|------|------|---------|------|
| MEMORY.md | status=confirmed, retrieval_weight ≥ 0.3 的 fact+preference | Consolidation 周期检查 | ≤ 8000 字符，按 memory_id 去重 |
| SELF.md | AgentProfile/Policy 投影 | 配置或版本变化 | 不接受 LLM 直接写入 |
| PENDING.md | status=candidate | Candidate 变化后重建 | 最多展示最近 200 条，完整列表在数据库 |
| HISTORY.md | kind=episode，按时间排序 | 每 50 条新 episode 追加一批 | source_ref 幂等去重 |
| RECENT_CONTEXT.md | 近期对话压缩 | 每次 Turn 后轻量刷新 Recent Turns | 不注入长期事实 |

### 11.2 Prompt Cache 策略

MEMORY.md 不再作为独立的全量记忆直接注入 system prompt。Context Builder 可以读取其稳定摘要作为候选输入，但最终必须通过 `RETRIEVAL-CONTEXT` 生成 Context Snapshot，并按 `memory_id` 与动态召回结果去重。策略：

- Consolidation 产生的增量事实写入 PENDING.md（不修改 MEMORY.md）；
- MEMORY.md 仅在 Consolidation 周期（默认 12 小时）后半段、且满足以下条件时刷新：(a) PENDING 已积累 ≥ 3 条新候选，(b) 上次刷新距今 ≥ 6 小时；
- 会话中途可以不刷新 MEMORY.md，但新的 confirmed MemoryItem 立即参与动态检索；
- 用户明确要求“立即生效”只要求数据库确认和新 Context Snapshot，不强制改写 system prompt。

与 akashic 的关键差异：cogito 的 PENDING 到 MEMORY 的数据流以数据库为中介——LLM 不直接写 Markdown，而是 Consolidation 从数据库渲染 Markdown 视图。Markdown 出岔子不会污染数据库事实。

### 11.3 MEMORY.md 渲染格式

```markdown
# 长期记忆

## 事实
- [2026-01-15] 用户是互联网公司产品经理，负责增长方向
- [2026-03-02] 用户的 GitHub 用户名是 example

## 偏好
- 偏好低压力、能长期坚持的创作方式
- 不喜欢在周末讨论工作
```

权重越高的条目越靠前。每条末尾附 `[id]`（对应数据库 `memory_id`），供引用和纠正。

## 12. 检索（与 Markdown 视图互补）

Markdown 视图提供"全景"，向量检索补充"细节"。规则见 `RETRIEVAL-CONTEXT`。要点：

- 默认检索排除 status ≠ confirmed 或 retrieval_weight < 0.1 的条目；
- MemoryResult 必须包含 `memory_id`、来源摘要、置信度和 retrieval_weight；
- 用户可通过 recall_memory 工具按 `memory_id` 精确引用或纠正；
- 检索命中后异步写入 `exposed` signal（更新 exposure_count/retrieval_count），不增加 reinforcement，不阻塞检索链路（参见 §5.3）。

## 13. Embedding

Embedding 是派生索引。输入包含规范化 subject、predicate、value 和必要时间信息，不包含无关 Secret。模型切换后新建版本，检索期间不得混合不可比较的向量空间。

Embedding 变更时同步记录 `embedding_model` 和 `embedding_version`。

## 14. 纠正与删除

用户纠正创建新事实并关联旧事实（`supersedes` 或 `corrects`）；用户删除先撤销正常检索，再由删除 Task 清理 Payload、Embedding 和派生 View，保留最小 Audit tombstone。被删除内容不得因索引重建重新出现。

## 15. 可观察性与测试

指标包括：候选率、确认率、拒绝率、冲突率、检索命中率、用户纠正数、过期数、retrieval_weight 分布、PENDING 积压量、MEMORY.md 刷新间隔。

测试覆盖：重复输入、矛盾偏好、跨 Session 共享、群聊隔离、Embedding 重建、删除后不可召回、衰减算法数值边界、prompt cache 破窗检测。
