# 执行生命周期与恢复

> **文档编号**：EXECUTION-LIFECYCLE  
> **适用范围**：Turn、RunAttempt、Checkpoint、重试、取消与恢复  
> **权威边界**：本文是即时交互执行对象、状态机和恢复语义的权威来源。  
> **关联文档**：DOMAIN-CONTRACTS, RUNTIME-FLOWS, SESSION-CONTEXT, APPROVAL-COMMANDS, STORAGE-DATA

## 1. 对象模型

系统只保留两层执行对象：

```text
Turn                         一次用户意图的逻辑生命周期
└── RunAttempt               Turn 的一次实际执行尝试
    ├── Checkpoint
    ├── ModelCall
    ├── ToolCall
    └── SideEffectReceipt
```

不设置位于 Turn 与 RunAttempt 之间的 `Run`。文档、接口和表结构中的 `run_id` 应迁移为 `attempt_id`；如果外部兼容协议暂时保留 `run_id`，它只能作为 `attempt_id` 的废弃别名，不能形成第三层生命周期。

### 1.1 Turn

```text
turn_id
input_message_id
conversation_id
session_id
status
priority
active_attempt_id
final_message_id
cancel_requested_at
next_attempt_at
version
created_at
completed_at
```

Turn 负责回答：用户的这次意图最终是否完成、正在等待什么、最终回复是什么。

### 1.2 RunAttempt

```text
attempt_id
turn_id
attempt_no
status
worker_id
started_at
finished_at
checkpoint_id
error_ref
resource_usage
trace_id
```

唯一约束：

```text
(turn_id, attempt_no)
```

RunAttempt 只描述一次执行占用。失败、进程丢失、等待后恢复或显式重试都创建新的 RunAttempt，不复活旧 Attempt。

## 2. 状态机

### 2.1 Turn 状态

```text
accepted → queued → running → completed
                      │
                      ├→ waiting_user → queued
                      ├→ waiting_external → queued
                      ├→ queued（计划重试，设置 next_attempt_at）
                      ├→ failed
                      ├→ cancelled
                      └→ expired
```

规则：

- `completed`、`cancelled`、`expired` 是终态；
- `failed` 只有通过显式 `RetryTurn` Command 才能回到 `queued`；
- `waiting_user` 和 `waiting_external` 不持有 Context Partition Lane 或 Worker Lease；
- 任一时刻最多只有一个非终态 RunAttempt；
- Turn 状态变化使用 `version` 条件更新。

### 2.2 RunAttempt 状态

```text
created → running → succeeded
                  ├→ failed
                  ├→ cancelled
                  └→ abandoned
```

`abandoned` 表示 Worker 丢失、Lease 失效或恢复扫描判定旧执行权无效。副作用结果不明确时，Attempt 可以结束为 `abandoned`，但关联 ToolCall 必须保持 `unknown` 并先对账。

## 3. 创建与执行事务

### 3.1 接收入站

同一短事务内：

1. 写入入站幂等记录；
2. 创建 Message；
3. 创建 Turn，状态为 `accepted` 或 `queued`；
4. 写入排序序号；
5. 写入领域 Event Outbox。

不在接收入站时提前创建 RunAttempt。Dispatcher 真正获得 Lane 后才创建 Attempt，避免队列中的 Turn 产生伪执行记录。

### 3.2 开始 Attempt

同一事务内完成：

```text
验证 Turn = queued 且已到 next_attempt_at
→ 获取 Context Partition Lane 执行权
→ 创建 RunAttempt(created)
→ Turn.active_attempt_id = attempt_id
→ Turn = running
→ RunAttempt = running
```

### 3.3 完成 Attempt

模型和 Tool 调用在事务外执行。提交结果时使用短事务并同时验证：

- Turn `version` 未被取消命令推进；
- `active_attempt_id` 仍指向当前 Attempt；
- Attempt 仍拥有有效执行权；
- Tool Receipt 与 Delivery 创建状态可对账。

成功提交应原子写入最终 Message、Memory Candidate、Delivery Outbox、Attempt 成功状态和 Turn 完成状态。

## 4. Checkpoint

Checkpoint 属于 Turn，由产生它的 Attempt 标记来源：

```text
checkpoint_id
turn_id
produced_by_attempt_id
step
completed_step_ids
prompt_inputs_ref
tool_call_states
partial_result_ref
pending_approval_id
context_snapshot_id
budget_consumed
created_at
```

Checkpoint 只保存恢复所需的最小确定状态，不序列化调用栈。每次恢复必须重新加载 Policy、配置和 Capability，并验证版本兼容性。

## 5. 重试、等待与恢复

### 5.1 自动重试

可重试错误发生后：

1. 当前 Attempt 标记 `failed`；
2. Turn 回到 `queued`；
3. 设置 `next_attempt_at` 和错误摘要；
4. 到期后创建新的 Attempt；
5. 累计预算不得重置。

### 5.2 等待用户或外部条件

当前 Attempt 正常结束，不跨等待期存活：

```text
Attempt → succeeded
Turn → waiting_user / waiting_external
```

等待条件满足后，Command 将 Turn 改为 `queued`，随后创建新的 Attempt 从 Checkpoint 恢复。

### 5.3 进程重启

Recovery Scan 查找：

- `running` Turn；
- 非终态 Attempt 但无有效执行权；
- `unknown` ToolCall；
- 已生成 Message 但未创建 Delivery 的不完整提交。

旧 Attempt 标记 `abandoned`。确认无未对账副作用后，Turn 才能重新进入 `queued`；否则进入 `waiting_external` 或人工检查。

## 6. 取消

`CancelTurn` 先持久化 `cancel_requested_at` 并推进 Turn 版本，再通知进程内取消令牌。旧 Attempt 后续提交因版本不匹配而被拒绝。

取消不能撤销已发生的副作用；补偿动作必须创建显式 Task。最终是否发送取消状态消息由 Delivery Policy 决定。

## 7. 数据库映射

核心表：

```text
turns
run_attempts
turn_checkpoints
model_calls
tool_calls
```

禁止重新引入 `runs` 表。历史兼容字段应通过 Migration 转换为 `attempt_id`。
