# 检索与上下文装配策略

## 1. 两条检索路径

系统提供两种检索方式，互补而非替代：

### 1.1 隐式检索（Context Builder 自动召回）

每次 Turn 开始时，Context Builder 自动执行多源召回并装配 ContextSnapshot，注入 Agent prompt。Agent 无需显式调用——它看到的上下文已经包含检索结果。

### 1.2 显式检索（recall_memory 工具）

Agent 在推理过程中可以主动调用 `recall_memory` 工具，发起额外的、更精确的检索请求。适用场景：

- 上下文中已有线索但缺少关键细节；
- 需要按特定时间范围、种类或主题精确查找；
- 当前隐式召回结果不足以完成任务时补充检索。

`recall_memory` 与隐式检索共用同一个检索管道，差异仅在于触发方（系统自动 vs Agent 主动）和参数来源（Query Plan vs Agent 提供的 query text）。

## 2. 双路召回

每条检索同时支持关键词检索和向量检索，两路并行执行后合并结果。

### 2.1 关键词检索（FTS + BM25）

基于 SQLite FTS5 全文索引，辅以 BM25 相关性评分：

- 适合精确匹配：术语、命名实体、ID、日期、数字；
- 适合低频词：对罕见关键词召回精确；
- 不受 Embedding 模型质量影响；
- 输入：规范化后的 query keywords（去停用词、词干提取）。

### 2.2 向量检索（Embedding + 余弦相似度）

基于 Embedding 向量的语义检索：

- 适合语义相近但词汇不重叠的查询（"喜欢简洁" ← "偏好极简主义"）；
- 适合模糊意图和主题探索；
- 依赖 Embedding 模型质量，换模型后版本隔离；
- 输入：完整 query text 的 embedding 向量。

### 2.3 合并策略

两路结果合并后按 §6 综合分数统一排序。任一链路不可用时降级走另一链路：

- Embedding 不可用 → 纯 FTS + recency；
- FTS 索引损坏 → 纯向量检索；
- 两路均不可用 → 仅靠近期消息和摘要。

## 3. 输入与输出

输入 `ContextQuery`：

```text
turn_id
session_id
principal_id
query_text
task_state
required_modalities
model_context_limit
reserved_output_tokens
policy_version
```

输出不可变 `ContextSnapshot`，记录所有选中条目、来源、分数、Token 和排除原因摘要。

## 4. 检索源

```text
current user message          必选 —— 当前输入全文
recent session messages       短期 —— 当前 Session 近期消息
conversation summary          短期压缩 —— 当前 Session 摘要
confirmed memory              长期 —— fact / preference / episode
preference/goal views         长期结构化 —— 活动偏好和活跃目标
task/checkpoint state         执行状态 —— 当前 Task 进度与恢复点
historical sessions           长期 —— 跨 Session 的历史对话摘要
authorized external context   外部数据 —— 当前请求附带的文档/链接
```

不同 Session 的近期消息不得召回。长期 Memory 先按 Principal、Scope、Trust 和 status 过滤。

### 4.1 目标/计划检索

Goal（目标）和 plan（计划）作为独立的检索源进入召回：

- 活跃 Goal（`kind=goal, status=confirmed, goal_status=active`）按 `goal_priority` 和 `goal_deadline` 排序注入，过期 Goal 不召回；
- 搜索 long-term plan 时使用 `kind=goal + retrieval_weight ≥ 0.3` 过滤；
- Goal 的语义检索权重设为基础 Memory 的 1.5 倍——当前目标相关的记忆优先展示。

### 4.2 历史会话检索

跨 Session 的历史对话通过 `SessionSummary` 和 `Episode` 召回：

- SessionSummary：每个已结束 Session 的压缩摘要（含主题、关键结论、时间）；
- Episode：从历史会话中提取的可检索经历片段，存为 `kind=episode` 的 MemoryItem；
- 历史会话检索使用 Session 级别的 Scope（同一 Channel/Conversation 优先），不跨 Principal。

## 5. Query Plan

### 5.1 查询改写（LLM Query Rewriter）

在确定性提取之前，先用轻量 LLM 对原始 query 做一次改写：

```text
输入: current user message + 最近 6 条会话消息
输出:
  needs_episodic: bool       — 是否需要检索历史经历
  episodic_query: str        — 改写后的经历查询
  needs_procedure: bool      — 是否需要检索 procedure/preference
  procedure_query: str       — 改写后的偏好/流程查询
  time_hint: str | None      — 用户提及的时间线索（"上周""去年"等）

超时: 800ms
失败策略: fail-open，所有字段取原始 query，needs_episodic=true
```

改写解决三个问题：

- 指代消解："上次那个项目" → "用户提到的 React 重构项目"；
- 意图分离："我喜欢什么风格，之前用过哪些编辑器" → procedure=风格偏好 + episodic=编辑器使用经历；
- 时间标注：提取用户 query 中的隐含时间范围，传递给后续过滤。

### 5.2 Plan 生成

1. 以改写结果为基础，提取实体、主题、时间范围和意图；
2. 对每个检索源设 `top_k`、最小分数和检索路径（keyword + vector）；
3. 执行并行只读召回（所有检索源 + 每源两路并行）；
4. 合并、去重、重排；
5. 按 Token Budget 装配。

Query Plan 记录版本，允许回放"为什么召回这些内容"。

## 6. 召回与排序

### 6.1 综合分数

```text
score = keyword_score   × w_keyword        // FTS/BM25 归一化
      + semantic_score  × w_semantic       // 向量余弦相似度归一化
      + recency         × w_recency        // 时间衰减因子
      + importance      × w_importance     // MemoryItem.importance
      + source_quality  × w_source         // MemoryItem.source_trust
      + goal_relevance  × w_goal           // 与当前活跃 Goal 的相关度
```

所有分项归一化到 `[0,1]`。权重按 Memory kind 可不同，但必须配置化并记录版本。确认状态、有效期和权限是硬过滤，不参与软打分。

### 6.2 关键词分数

```text
raw_bm25 = FTS5.bm25(query_terms, document_fields)  // 数值越小越相关
keyword_score = normalize_inverse(raw_bm25, result_set)  // 转为 [0,1]，越大越相关
// fields: summary, value, subject, predicate
```

FTS5 内置 BM25，不依赖外部模型。每次写入 MemoryItem 时同步更新 FTS 索引。

### 6.3 向量分数

```text
semantic_score = cosine_similarity(query_embedding, memory_embedding)
```

同一条 MemoryItem 在换 Embedding 模型后新建版本，检索时只使用当前活跃版本向量。

## 7. 去重与多样性

- 相同 `memory_id/message_id/session_id` 直接去重；
- `supersedes` 关系只保留当前有效事实，必要时附冲突提示；
- 高相似条目按来源质量、时间和置信度保留；
- 同一主题设置最大条目数，避免单一主题占满 Context；
- 摘要和原消息重复时优先保留满足任务所需的最小证据集合。

## 8. Scope 降级链

检索按 Scope 逐级放宽：

```text
1. session: 当前 session_id 精确匹配
2. conversation: 当前 conversation_id / context_partition_key 匹配
3. channel: 当前 channel_instance_id 匹配
4. principal-global: scoped 结果 < min_results 时放宽到当前 principal_id
5. disabled: Policy 禁止放宽，尤其是共享群聊中的非 Owner 请求
```

每级降级始终先过滤当前 `principal_id`、Trust 和 Memory status；共享 Session 不意味着共享个人长期 Memory。降级记录在 Trace 中，便于调试和评估。

## 9. Token 分配

```text
system/runtime policy   固定保留
current input           必选
task/tool state         按需保留
recent messages         35%
memory/preferences      25%
goals/plans             10%  ← 从 memory 分出独立的 Goal 配额
external context        15%
summary/metadata        15%
```

百分比是默认上限，可将未用额度转给其他区。裁剪顺序为低分外部内容、低分 Memory、较旧消息、非活跃 Goal；System Policy 和当前输入不裁剪。

未使用的 Goal 配额自动并入 Memory 区，优先给 retrieval_weight ≥ 0.5 的条目。

## 10. Context Snapshot

```text
snapshot_id
turn_id
session_id
message_upper_bound
query_plan_version
selection_policy_version
items[{type,id,source,score,tokens,trust_label,retrieval_path}]  ← 增加 retrieval_path
excluded_summary
total_tokens
created_at
```

`retrieval_path` 标识来源链路：`keyword | vector | keyword+vector`。

Snapshot 创建后不可变。重试默认复用 Snapshot；审批等待过久或 Session 版本变化时必须重新构建并记录父 Snapshot。

## 11. recall_memory 工具

Agent 可主动调用的检索工具，复用 §2-§8 的检索管道。

### 11.1 工具定义

```text
tool_name: recall_memory
description: 搜索长期记忆、目标计划或历史会话中与查询相关的内容

parameters:
  query           str    — 检索查询文本
  memory_kinds    list   — 限定种类: fact/preference/episode/goal/constraint，空=全部
  time_range      str    — 时间过滤: "7d" / "2026-01" / "2026-01-15..2026-03-01"
  top_k           int    — 返回条数上限，默认 5
  include_sessions bool  — 是否包含历史会话摘要，默认 false

returns:
  items[{id, kind, summary, score, source, happened_at, retrieval_weight}]
  query_rewritten  str   — LLM 改写后的实际查询（透明化）
  total_hits        int  — 全量命中数
```

普通 `recall_memory` 命中只增加 `exposure_count/retrieval_count`，不增加 reinforcement；只有用户明确引用、确认有用或成功 Task 证明依赖时才强化。

### 11.2 Tool 与隐式检索的关系

- 隐式检索的结果已被注入当前 Context，Agent 应优先使用；
- `recall_memory` 用于**补充**隐式结果——当 Agent 发现缺少关键信息时主动深挖；
- `recall_memory` 的结果**不**自动写入 ContextSnapshot——Agent 在对话中自行引用；
- 每次 `recall_memory` 调用对命中条目的 `reinforcement` +1（参见 MEMORY-LIFECYCLE §5.3）。

### 11.3 安全性

- Agent 只能检索其 Principal Scope 内的记忆，不可跨 Principal；
- 工具结果中排除 `source_trust < configured_minimum` 的条目；
- 工具输出不包含原始 Message 内容——只有摘要和元数据。

## 12. 降级

- Embedding 不可用：使用 FTS/keyword + recency，标记降级路径；
- FTS 索引损坏：纯向量检索；
- Summary 不可用：读取受限近期消息；
- Memory 索引损坏：从 Canonical Memory 重建，不阻塞即时对话；
- Token 估算器失败：使用保守字符估算；
- Query Rewriter 超时：fail-open，原始 query 直传；
- 外部检索超时：继续执行并标记缺失来源。

所有降级事件记录在 ContextSnapshot 的 excluded_summary 中。

## 13. 缓存

Embedding 和纯查询召回可以缓存；最终 Context Snapshot 不跨 Turn 复用。缓存键包含模型/Embedding 版本、Scope、过滤条件和数据版本。

## 14. 指标与评估

记录：召回命中数、两路路径分布（keyword/vector）、检索路径级联、Query Rewriter 命中率、被模型引用数、用户纠正数、无效召回数、Token 占比和延迟。

离线评测至少包含：事实查找、时间冲突、跨 Session 隔离、偏好召回、Goal 关联检索、历史会话检索、同义词汇不重叠和外语查询八类样本。

## 15. 验收条件

- 不泄漏其他 Principal/Session 的短期内容；
- 过期和 rejected Memory 不进入 Context；
- 关键词检索和向量检索任一路不可用时另一方正常工作；
- `recall_memory` 工具仅返回 Scope 内记忆，不跨 Principal；
- 相同输入和数据版本产生稳定选择；
- 降级后仍满足 Context Window；
- 每个选中条目可解释来源、分数和检索路径（keyword/vector）。
