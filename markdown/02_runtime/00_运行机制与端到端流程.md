# 运行机制与端到端流程

> **文档编号**：RUNTIME-FLOWS  
> **适用范围**：Turn Orchestrator、Durable Job、Event Bus、幂等、Outbox、状态机和错误处理  
> **权威边界**：本文是其范围内的规范性来源；总体架构文档只负责概念与边界。  
> **细化规范**：Turn、Agent Loop、Task 和 Event 的字段与状态分别以 `EXECUTION-LIFECYCLE`、`AGENT-LOOP`、`TASK-SCHEDULER`、`EVENT-OUTBOX` 为准。  
> **关联文档**：ACCESS-DELIVERY, PROACTIVE-TASKS, STORAGE-DATA, SECURITY-OBS

## 阅读说明

**目的**：定义同步 Turn、持久 Task、Event 通知的执行语义，以及顺序、幂等、事务和失败恢复。

**边界**：不定义单个业务模块内部算法。

**建议读取方式**：实现 Worker、重试、取消、审批或 Tool 副作用前应完整读取本文。

## 1. 三类控制模型与执行边界

### 1.1 为什么不能使用万能 Pipeline

系统中的工作具有不同语义：

```text
用户对话        需要低延迟、流式、会话排序和取消
后台任务        需要 Lease、Checkpoint、重试和跨重启恢复
事件通知        需要一对多、异步消费和幂等 Handler
普通服务调用    只需要清晰接口和函数返回值
```

如果全部包装为通用 Stage/Hook：

- 事务边界会被拆散；
- 同步和异步语义混淆；
- Hook 修改顺序决定隐式行为；
- 重试单位不明确；
- 人工审批容易错误地阻塞调用栈；
- Debug Trace 很丰富，但系统行为难以推导。

因此，系统只允许在明确用例中使用显式步骤编排，不把 Pipeline Engine 作为所有模块的总线。

### 1.2 Turn Orchestrator

Turn Orchestrator 处理一次用户输入的交互生命周期。

典型步骤：

```text
1. 接收入站 Envelope
2. 入站幂等检查
3. 身份和 Conversation 解析
4. 创建 Message 与 Turn
5. 进入 Context Partition Lane
6. 创建 RunAttempt
7. 构建 Context
8. 执行 Agent Loop
9. 调用 Tool Policy 与 Tool Executor
10. 形成最终 AgentReply
11. 写入 Message、Memory Candidate 和 Delivery Outbox
12. 完成 RunAttempt 与 Turn
13. 发布 TurnCompleted
```

Turn Orchestrator 可以调用领域服务，但不能：

- 在一个数据库事务中执行模型和网络调用；
- 为等待用户长期保持锁；
- 直接调用平台发送；
- 将每个内部函数都转成 Event；
- 隐式创建外部副作用。

### 1.3 Durable Job Runner

任何满足以下条件之一的工作都应转换为 Task：

- 预计运行时间明显超过普通 Turn；
- 需要在指定时间或重复计划运行；
- 需要等待用户、外部系统或资源；
- 失败后必须自动重试；
- 可能跨进程重启；
- 需要 Checkpoint；
- 需要多个外部副作用步骤；
- 不应占用 Context Partition Lane。

Task Runner 的基本循环：

```text
find eligible task
→ acquire lease transactionally
→ load checkpoint and budget
→ execute one handler attempt
→ persist result/checkpoint
→ transition state
→ publish event
→ release or renew lease
```

### 1.4 Event Bus

Event 是不可变事实。Event Handler 适合：

- 更新 Projection；
- 生成指标；
- 创建后续 Task；
- 创建主动候选；
- 导出 Trace；
- 发送非阻塞通知；
- 执行最终一致的派生处理。

Event Handler 不适合：

- 返回 Agent 当前步骤需要的同步结果；
- 直接改变已完成事实；
- 绕过 Command Handler 修改其他聚合；
- 执行无幂等的外部副作用。

### 1.5 选择控制模型的判断规则

```text
是否需要立即返回同步结果？
├─ 是 → 普通领域调用或 Turn Orchestrator
└─ 否
   是否可能跨进程、等待、重试或延迟？
   ├─ 是 → Durable Task
   └─ 否
      是否表达“已经发生”的事实并允许多个消费者？
      ├─ 是 → Event
      └─ 否 → 普通应用服务调用
```

### 1.6 Checkpoint 与恢复边界

Checkpoint 应保存“恢复执行所需的最小确定状态”，而不是序列化整个 Python 调用栈。

Checkpoint 可以包含：

```text
current_step
completed_step_ids
tool_receipts
partial_result_ref
pending_approval_id
external_cursor
context_snapshot_id
budget_consumed
```

恢复时重新构建运行对象，并验证：

- Lease 是否有效；
- 配置和插件版本是否兼容；
- 已完成副作用是否有 Receipt；
- Context 是否需要重新加载；
- 用户是否已取消；
- 外部等待条件是否满足。

### 1.7 Command、Event 与 Task 的因果关系

所有异步工作应保留：

```text
correlation_id  同一业务链路
causation_id    直接导致当前对象的 Message/Command/Event
origin          user | scheduler | connector | recovery | plugin
```

示例：

```text
User Message
→ SubmitInboundMessage Command
→ TurnCreated Event
→ Agent RunAttempt
→ CreateTask Command
→ TaskCreated Event
→ TaskCompleted Event
→ DeliveryCreated Event
```

## 2. 端到端运行场景

### 2.1 被动文本对话

```text
User
  → Platform
  → Channel Adapter
  → ChannelEnvelope
  → Agent API
  → Inbound Inbox/Dedup
  → Identity Resolution
  → Context Partition Lane
  → Turn + RunAttempt
  → Context Builder
  → Agent Runtime
  → Reply Builder
  → Message + Delivery Outbox
  → Delivery Worker
  → Channel Gateway
  → Platform
  → User
```

关键事务：

#### 事务 A：接收入站

同一事务中：

- 写入 Inbound Inbox；
- 创建标准 Message；
- 创建或关联 Turn；
- 记录幂等键。

#### 非事务外部执行

- 构建 Context；
- 调用模型；
- 调用只读或受控 Tool。

#### 事务 B：完成 RunAttempt

同一事务中：

- 保存 Agent 输出 Message；
- 保存 Memory Candidate；
- 创建 Delivery Outbox；
- 更新 RunAttempt/Turn 状态；
- 创建 TurnCompleted Event Outbox。

### 2.2 流式回复

流式输出分为“交互展示”和“最终事实”两层：

```text
模型 delta
→ Stream Session
→ WebSocket / 平台消息编辑
→ 临时状态

最终内容
→ Assistant Message
→ Delivery final/update
→ RunAttempt succeeded，Turn completed
```

约束：

- Delta 不作为最终 Message 逐条写入；
- 平台编辑失败可以降级为最终单条发送；
- 流式中断时保存已确认的部分结果引用；
- 最终事实以数据库中的 Assistant Message 为准；
- 新用户输入可根据策略排队或取消当前流式 Turn。

placeholder、编辑和最终化都必须通过持久化 Delivery Attempt 执行，Stream Session 不直接调用平台。具体对象、Receipt 和恢复规则见 `STREAMING-DELIVERY`。

### 2.3 Tool 调用

```text
Agent proposes ToolCall
→ Tool Manifest lookup
→ argument schema validation
→ Policy Gate
→ allow | reject | require approval
→ acquire resource budget
→ execute Tool
→ capture Result and Receipt
→ return structured result to Agent
```

如果 Tool 有副作用：

- ToolCall 在执行前持久化；
- 使用稳定 Idempotency Key；
- 成功后保存 Receipt；
- 超时且结果未知时进入 `unknown`；
- 恢复时先 reconcile，不直接重试。

#### 2.3.1 多模态感知

```text
Inbound ContentPart
→ Asset/Payload 持久化 + Message Link
→ Vision Cache 查询
→ cache hit：Context 注入短描述
→ cache miss：有界 inline 分析或 vision.analyze Durable Task
→ VisionAnalysis 持久化
→ Main LLM 可按 asset_id 调用 analyze_multimodal_asset 获取详情
```

Vision 超时或失败不改变 Turn 的文本执行资格。首次分析的有界等待不持有数据库
事务；超过等待预算后由 Task 接管。Task、自动处理和 Tool 通过唯一 Cache Key 与
条件 Claim 防止重复模型调用。

### 2.4 人工审批

```text
Tool Policy returns RequireApproval
→ persist ApprovalRequest
→ 当前 RunAttempt 结束，Turn = waiting_user
→ create approval Delivery
→ user response enters as high-priority Command
→ validate responder Principal
→ Approval = approved/rejected/expired
→ Turn queued for resume，恢复时创建新 RunAttempt
→ reconstruct Context and continue
```

审批内容必须包括：

```text
将执行什么动作
作用目标
关键参数
需要的权限
风险级别
预计影响
是否可撤销
过期时间
```

### 2.5 长期任务

```text
User/Agent/Scheduler creates Task
→ queued
→ eligible at scheduled_at
→ Worker acquires Lease
→ Handler loads Checkpoint
→ executes bounded step
→ heartbeat/renew
→ completed or waiting/retry/failed
→ Event
→ optional Delivery
```

Task Handler 应设计为“可重复进入”，不能假设只调用一次。

### 2.6 Connector 数据摄取

```text
Scheduler
→ Connector Poll Task
→ load Cursor
→ fetch external items
→ archive Raw Payload
→ normalize items
→ deduplicate by source key/content hash
→ save Event
→ advance Cursor transactionally
→ Event Consumers
```

Cursor 只有在对应项安全持久化后才能推进。

### 2.7 主动决策

```text
Event
→ Candidate Builder
→ source trust gate
→ novelty/dedup gate
→ user relevance scoring
→ optional LLM judge
→ quiet hours/budget/cooldown
→ decision
```

决策结果：

```text
send_now
send_later
add_to_digest
silent_process
create_task
ask_permission
discard
```

每个决定保存输入摘要、规则结果、模型评分、最终原因和预算消耗。

### 2.8 摘要生成

```text
Candidates marked digest
→ group by topic/time/source
→ deterministic selection
→ summarize
→ render
→ one Delivery
→ link included candidates
```

同一 Candidate 被摘要消费后不能再次即时推送，除非出现重要更新。

### 2.9 取消当前 Turn

```text
User sends stop/cancel
→ high-priority control classification
→ CancelTurn Command
→ persist cancel_requested_at
→ signal in-memory token
→ Agent/Tool checks safe point
→ stop or reconcile
→ RunAttempt cancelled，Turn cancelled
→ optional status Delivery
```

取消不能撤销已完成的外部副作用，只能阻止后续步骤，并在必要时生成补偿任务。

### 2.10 系统重启恢复

启动 Recovery Scan：

```text
running Turn/Task with no valid owner or execution right
→ inspect last checkpoint
→ inspect Tool/Delivery receipts
→ classify resumable/reconcile/manual review
→ enqueue recovery attempt
```

可能结果：

```text
resume
retry
mark_failed
waiting_user
manual_review
```

### 2.11 Delivery 失败

```text
Delivery ready
→ send attempt
→ network timeout
→ status unknown or retryable
→ inspect platform receipt/idempotency support
→ retry（仅 retry_scheduled）或 reconcile（仅 unknown）
```

`unknown` 状态下禁止调用 `retry`，只能调用 `reconcile` 向 Gateway 查询
平台侧证据后再转为 `sent` 或 `failed`；`retry` 仅在状态为
`retry_scheduled` 时由 `SqliteDeliveryService.retry` 接受，并通过条件
`UPDATE ... WHERE status='retry_scheduled'` 守护。

Delivery 失败不重新执行 Agent。内容需要重新生成时必须显式创建新 Turn/Task。

### 2.12 模型 Provider 降级

```text
primary provider timeout
→ classify retryable
→ check request idempotency and budget
→ retry same provider or fallback
→ adapt capability differences
→ record selected provider and degradation
```

如果 fallback 不支持 Tool Calling 或视觉，必须重新构造 Request，不能假设接口完全等价。

## 3. 并发、顺序、幂等与一致性

### 3.1 Context Partition Lane

每个 `context_partition_key` 拥有逻辑 Lane。Session 由 Channel、平台 Conversation、Thread 和多用户隔离策略决定，不跨 Partition 共享短期上下文：

- 普通用户 Turn 在 Lane 内按优先级和接收序列排序；
- 不同 Conversation 可以并行；
- 长期 Task 不持有 Lane；
- 取消、审批和控制消息以高优先级进入；
- 读取 Message 历史必须基于 Turn 开始时的 Context Snapshot 或版本边界。

Lane 可以通过数据库队列、进程内 Actor 或二者结合实现，但排序事实必须持久化。

### 3.2 优先级

建议：

```text
100 取消、停止、审批、权限响应
80  用户即时消息
60  waiting_user / waiting_external 恢复
40  计划任务和 Connector
20  主动推送生成
10  Drift、整理和维护任务
```

优先级不能导致低优先级永久饥饿。Scheduler 应使用 aging 或最大等待时间。

### 3.3 并发控制

#### 3.3.1 进程内

- 每个 Context Partition 一个异步锁或 Actor；
- Provider 使用全局 Semaphore 控制并发；
- SQLite 写操作尽量集中且短；
- Payload 文件写入使用临时文件 + 原子 rename。

#### 3.3.2 跨进程

- Task 和 Delivery 使用数据库 Lease；
- Lease 获取使用条件更新；
- Worker 提交结果时验证 `lease_owner` 和 `lease_version`；
- 过期 Worker 的结果只能作为诊断数据，不直接提交业务状态。

### 3.4 入站幂等

首选：

```text
inbound_key = channel_instance_id + platform_message_id
```

备用：

```text
platform event id
```

弱去重：

```text
sender + conversation + content hash + time bucket
```

弱去重必须标记 `best_effort`，避免误删用户重复发送的真实消息。

### 3.5 Tool 幂等

副作用 Tool 必须提供：

```text
idempotency_key
side_effect_receipt
reconcile(request, previous_receipt_or_unknown)
```

幂等键的作用域应包含：

```text
tool_name + target + semantic operation + logical request id
```

不能只使用随机 UUID，因为重试需要复用同一逻辑键。

### 3.6 Scheduler 幂等

周期触发键：

```text
schedule_id + scheduled_fire_time
```

同一触发时间最多创建一个逻辑 Task。错过触发时由 Misfire Policy 决定补跑、跳过或合并。

### 3.7 Transactional Outbox

同一 SQLite 事务中：

```text
业务状态变更
+ Event Outbox
+ Delivery Outbox
+ Payload Export Outbox
```

Worker 在事务外调用外部系统。Outbox Item 状态：

```text
pending → leased → succeeded
                 → retry_scheduled
                 → failed
```

### 3.8 Inbox 与 Consumer 幂等

Event Consumer 保存：

```text
consumer_name
event_id
status
attempt
processed_at
```

Handler 在处理前检查消费记录，状态变更与消费成功尽可能在同一事务中完成。

### 3.9 一致性模型

#### 强一致范围

同一聚合内的状态、当前状态机转移、幂等记录和 Outbox 写入。

#### 最终一致范围

- Dashboard Projection；
- Trace Export；
- Memory Consolidation；
- 主动 Candidate 派生；
- 指标；
- 外部平台发送结果；
- Payload 压缩和归档。

### 3.10 Exactly Once 的限制

系统只能提供：

```text
本地逻辑处理幂等
+ 外部调用至少一次尝试
+ 平台能力范围内去重
+ Receipt/Reconcile
```

对于不支持幂等的外部系统，可能出现“外部已成功但本地未知”。此时状态必须是 `unknown`，不能简单自动重试。

### 3.11 事务边界

禁止在数据库事务中执行：

- 模型调用；
- 网络请求；
- Channel 发送；
- MCP 调用；
- 长时间文件处理；
- 等待用户；
- Shell 执行。

推荐模式：

```text
短事务读取并声明意图
→ 事务外执行
→ 短事务提交结果
```

### 3.12 竞态处理

必须显式处理：

- 用户在 Tool 执行中取消；
- Approval 过期和用户批准同时发生；
- Endpoint 被禁用与 Delivery 正在发送；
- Task Lease 过期与旧 Worker 返回结果；
- Memory 候选被用户拒绝与后台 Consolidation 同时运行；
- 同一 Connector Item 被多个 Poll 重复发现；
- 平台消息编辑失败后最终发送重复。

采用版本号、条件更新和明确状态机解决，不依赖“通常不会同时发生”。

## 4. 状态机规范

### 4.1 Turn 与 RunAttempt 状态机

Turn 持有逻辑交互状态，RunAttempt 只表示一次执行尝试。失败、等待后恢复或显式重试都创建新 RunAttempt，不设置额外 Run 层。完整状态、事务和恢复规则见 `EXECUTION-LIFECYCLE / 2. 状态机`。

### 4.2 Task 状态机

```text
created → scheduled/queued → running
running → completed
running → retry_scheduled
running → waiting_user
running → waiting_external
running → failed/cancelled/expired
```

### 4.3 Delivery 状态机

```text
pending → scheduled → sending → sent
                         │
                         ├→ streaming → finalizing → sent
                         ├→ partially_sent
                         ├→ interrupted/unknown
                         ├→ retry_scheduled
                         ├→ failed
                         └→ cancelled
```

流式子状态、平台 Receipt 和 `unknown` 对账见 `STREAMING-DELIVERY`。

### 4.4 Approval 状态机

```text
pending → approved
pending → rejected
pending → expired
pending → cancelled
```

Approval 决定一旦消费，不得重复恢复同一 Turn 或 Task。关联字段和消费事务见 `APPROVAL-COMMANDS`。

## 5. 错误模型

### 5.1 ErrorEnvelope

```text
error_code
category
message
retryable
retry_after
source_component
safe_details
internal_payload_ref
caused_by
```

用户可见内容使用 `safe_details`，内部堆栈只保存到受限 Trace。

### 5.2 分类

```text
validation
policy_denied
authentication
authorization
rate_limit
timeout
dependency_unavailable
conflict
not_found
resource_exhausted
side_effect_unknown
internal
```

### 5.3 重试原则

- Validation、Policy Denied 不重试；
- Rate Limit 遵循 Retry-After；
- 网络临时错误指数退避；
- Side Effect Unknown 先 Reconcile；
- 数据库冲突有限重试；
- 模型内容错误可在预算内修复一次，不无限自我纠正。
