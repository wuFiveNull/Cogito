# 架构决策记录指南

## 1. 何时创建 ADR

出现以下变化时必须创建 ADR：

- 改变模块所有权、依赖方向或进程边界；
- 引入新的持久状态机、数据库或外部协议；
- 改变 Turn、Task、Delivery、Memory 的语义；
- 接受无法通过普通重构消除的技术折中；
- 改变安全、数据保留、兼容或恢复保证。

字段或实现细节未改变架构约束时，不创建 ADR，直接修改功能规范并更新版本。

## 2. 状态

```text
proposed → accepted → superseded
                   └→ deprecated
proposed → rejected
```

已接受 ADR 不原地改写结论；新决策通过 `supersedes` 关联旧 ADR。

## 3. 模板

```markdown
# ADR-NNN：标题

状态：proposed | accepted | rejected | superseded
日期：YYYY-MM-DD
决策者：owner
影响文档：DOC-ID 列表

## 背景
## 约束
## 候选方案
## 决策
## 后果
## 迁移与回滚
## 验证方式
```

## 4. 验收要求

ADR 被接受前必须说明：

1. 当前问题是否可复现或明确观察；
2. 为什么不采用更简单方案；
3. 数据和协议如何迁移；
4. 失败时怎样回滚；
5. 哪些规范和测试需要同步修改。

## 5. 存放与命名

ADR 存放于 `markdown/00_guides/adr/`，文件名为 `ADR-NNN_简短标题.md`。`manifest.json` 只登记 `accepted` 且仍有效的 ADR；草案可以存在但不得成为实现权威。
