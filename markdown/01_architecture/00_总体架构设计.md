# 主动式个人 Agent 总体架构设计

> **文档编号**：ARCH-OVERVIEW  
> **适用范围**：系统大局观、边界、顶层组件、核心运行模式与全局不变量  
> **权威边界**：本文是系统边界、顶层职责、依赖方向和全局不变量的权威来源；不定义具体字段、接口和表结构。  
> **细化规范**：全局约束见 `GLOBAL-INVARIANTS`，模块依赖见 `SYSTEM-BOUNDARIES`，功能字段和状态以对应 functional-spec 为准。  
> **关联文档**：DOMAIN-CONTRACTS, RUNTIME-FLOWS, ACCESS-DELIVERY, AGENT-COGNITION, PROACTIVE-TASKS, CAPABILITY-PLUGINS, STORAGE-DATA, SECURITY-OBS, OPS-GOVERNANCE

## 1. 系统定位

本系统是供单个所有者长期使用的、本地优先的主动式个人 Agent。它既处理来自 Web、Telegram、飞书等 Channel 的即时消息，也能持续摄取邮件、日历、RSS、GitHub 等外部信息，在明确的权限、预算和通知策略下执行工具、长期任务和主动推送。

系统不是公共 SaaS、多租户平台或通用插件市场。设计重点是：长期运行可靠性、个人数据可控、外部副作用可审计、各模块可替换，以及出现失败后可恢复。

## 2. 系统边界

系统内部负责：

- Channel 消息的标准化和回复路由；
- 会话、Turn、RunAttempt 和长期任务的生命周期；
- Agent 推理、模型调用、Tool/Skill 调度；
- Memory、Preference、Goal 和 Context 组织；
- 外部数据摄取、主动决策、摘要和投递；
- 权限、审批、幂等、Trace、审计、预算和恢复；
- Dashboard、Web Channel 和运行控制。

系统外部包括：消息平台、模型服务、邮件/日历/GitHub 等来源系统、MCP Server 和能够产生外部副作用的第三方服务。外部内容一律按不可信数据处理；外部系统的可用性和最终状态不由本系统保证。

## 3. 顶层架构

```text
Interaction
  Dashboard / Web Channel / Query API / Command API
                         |
                         v
Agent Core
  Conversation & Identity
  Turn Orchestrator -> Agent Runtime -> Model / Tool / Skill
  Cognition / Data & Event / Proactive / Task / Delivery
  Storage / Security / Policy / Observability
             |                         |
             v                         v
Channel Gateway                 External Capabilities
LangBot-based adapters          Model / MCP / Mail / Git
```

### 3.1 Channel Gateway

负责平台连接、协议转换、附件处理、平台事件和发送能力。它不拥有 Agent 逻辑、Memory 或主动策略。推荐将 LangBot 作为独立 Gateway 使用，通过版本化内部协议与 Agent Core 通信，避免 Agent Core 依赖 LangBot 内部对象。

### 3.2 Agent Core

Agent Core 是模块化单体，拥有业务状态和执行语义。内部模块通过稳定接口协作，共享同一事务事实源，但禁止绕过模块公开面直接访问其他模块内部表和实现对象。

### 3.3 Interaction

Interaction 提供 Dashboard、Web Channel、只读 Query API 和显式 Command API。Dashboard 不直接修改数据库；取消任务、批准 Tool、重试投递等操作都必须转化为业务命令。

### 3.4 Worker 与 Scheduler

后台 Worker 负责 Connector、长期任务、Delivery、Outbox 和异步维护任务。Scheduler 只负责产生到期执行请求，不直接执行业务副作用。

## 4. 三类运行模型

系统不使用一个万能 Pipeline 覆盖所有工作，而是区分三种控制模型。

### 4.1 Turn Orchestrator

处理用户即时交互：接收消息、加载 Context、运行 Agent Loop、调用 Tool、产生回复。它强调低延迟、会话内顺序、取消和流式输出。

### 4.2 Durable Job Runner

处理可能跨越进程重启或等待外部条件的工作：Connector 拉取、摘要、长期任务、审批等待、周期任务和重试。任务状态、Checkpoint、Lease 和重试时间全部持久化。

### 4.3 Event Bus

传播已经发生的事实，例如 `TurnCompleted`、`MemoryCandidateCreated`、`DeliveryFailed`。Event Consumer 必须幂等；事件不能被用来伪装需要同步返回值的领域调用。

## 5. 核心数据流

### 5.1 被动对话

```text
Channel 消息
→ Gateway 标准化并生成 ChannelEnvelope
→ 入站幂等与身份解析
→ Context Partition Lane 排队
→ Turn Orchestrator
→ Context / Memory / Agent / Tool
→ AgentReply
→ Delivery Outbox
→ Gateway 发送到原入口
→ 投递结果、Trace 与反馈
```

### 5.2 主动信息处理

```text
Connector 拉取
→ 原始内容归档
→ 标准化与去重
→ Event/Candidate
→ 硬规则、偏好、预算与相关性判断
→ silent / digest / task / send
→ Delivery Outbox
→ 用户反馈更新策略
```

### 5.3 长期任务

```text
TaskRequest
→ 持久化任务
→ Worker 获取 Lease
→ 分步执行并写 Checkpoint
→ waiting_user / waiting_external / retry / completed
→ 结果进入 Delivery 或后续 Event
```

## 6. 核心领域概念

系统通过以下概念避免把“消息、执行和投递”混为一体：

- **Principal**：系统识别的主体；单用户系统也必须显式建模。
- **Endpoint**：某个平台上的用户、Bot 或发送端点。
- **Conversation**：平台会话或线程。
- **Session**：Agent 上下文边界，可与 Conversation 分离。
- **Message**：不可变的入站或出站内容记录。
- **Turn**：一次用户意图处理的逻辑单元。
- **RunAttempt**：Turn 的一次实际执行尝试；每次重试或恢复创建新 Attempt。
- **Task**：可持久化、可等待、可恢复的长期工作。
- **Event**：已经发生的事实。
- **Delivery**：向某个目标发送内容的独立生命周期。
- **MemoryItem**：带来源、置信度、有效期和状态的长期认知事实。

字段和聚合边界见 `DOMAIN-CONTRACTS`。

## 7. 存储与一致性

SQLite 是唯一事务事实源，保存当前状态、索引、任务、Outbox、投递和审计元数据。Prompt、模型原始响应、附件、Connector 原文和大型 Trace Payload 存放于 Payload Store，并由 SQLite 保存引用、哈希和敏感级别。

跨外部系统无法实现真正 Exactly Once。系统使用：

```text
本地事务 + Transactional Outbox
+ 消费者 Inbox/幂等键
+ Tool Side Effect Receipt
+ 平台去重策略
```

使用户可见结果尽量接近一次。

## 8. 并发与顺序

Session 根据 Channel、平台 Conversation、Thread 和多用户隔离策略决定，不跨 Context Partition 共享短期上下文。不同 Session 只共享经过 Principal/Scope 校验的长期 Memory、Preference 和 Goal。同一 Context Partition 的逻辑 Lane 串行处理会改变上下文的 Turn；后台任务不长期占用 Lane，需要恢复会话时通过显式命令重新进入。

用户取消和审批响应优先于普通消息，普通消息优先于主动推送和 Drift 工作。所有跨进程竞争使用数据库 Lease、版本号或唯一约束解决。

## 9. 扩展架构

扩展面收敛为：

- **Provider**：替换完整能力实现，例如 Model、Embedding、Memory Retriever、Channel Driver。
- **Middleware/Policy Gate**：围绕少数固定调用点执行校验、变换或拒绝。
- **Capability**：Tool、Skill、MCP、Connector、Job Handler。
- **Plugin Package**：能力的安装、版本、权限和生命周期载体，不是新的业务控制流。

Envelope 默认不可变。需要修改数据时返回受校验的 Patch；普通扩展不得修改 `trace_id`、`origin`、`principal_id`、`reply_route` 和幂等键等受保护字段。

## 10. 安全与信任

模型只提出动作，不能授予自己权限。Tool 执行必须经过确定性的 Policy Engine。外部网页、邮件、仓库内容和 Tool 输出都携带 Trust Label，并始终作为数据而不是系统指令。

高风险副作用通过持久化 ApprovalRequest 进入 `waiting_user`，不能在内存调用栈中等待。Secret 不进入普通 Prompt、日志或默认 Trace Payload。Shell、代码执行和第三方插件优先隔离到独立进程或 Sandbox。

## 11. 可观察性与资源治理

每个 Turn、Task、Tool、Model 和 Delivery 都具有 TraceContext。默认 Trace 保存状态、耗时、Token、工具名和脱敏摘要；完整 Payload 只在临时调试级别记录，并受大小、保留期和加密策略控制。

模型调用、主动推送、Drift、Tool 次数、后台执行时间和存储增长都受到 ResourceBudget 约束。系统允许 Model、Channel、Plugin 或 Payload Store 局部降级，避免单点故障导致整个 Agent 停止。

## 12. 推荐部署形态

```text
channel-gateway     平台连接和消息转换
agent-api           Interaction API、即时 Turn 和控制命令
agent-worker        Durable Job、Connector、Delivery、Outbox
interaction-web     Dashboard 和 Web Channel 前端
sqlite + payloads   唯一事实源和大型内容存储
```

个人部署时可以将 `agent-api` 和 `agent-worker` 合并为一个进程，但必须保留逻辑任务边界、幂等和恢复语义。Channel Gateway 建议保持独立，以降低上游框架耦合和平台连接故障的影响。

## 13. 全局不变量

1. SQLite 是唯一事务事实源。
2. 每条入站消息具有稳定幂等键。
3. 同一会话内改变上下文的 Turn 不并发提交。
4. 所有外部副作用都经过 Policy，并产生 Side Effect Receipt。
5. 需要跨重启等待的工作必须持久化为 Task 或 Approval。
6. Delivery 独立于 Turn/RunAttempt；包括流式编辑在内的平台副作用都通过 Delivery Attempt 执行。
7. Event 表示事实；Command 表示请求；二者不可互换。
8. 模型输出不等于授权结果。
9. 外部内容不能提升权限或修改系统策略。
10. Trace 默认脱敏，完整 Payload 不是常态。
11. 总体架构只定义边界；具体字段和状态以专题文档为准。
