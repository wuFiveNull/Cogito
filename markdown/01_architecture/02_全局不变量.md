# 全局不变量

## 1. 数据事实

1. SQLite 是当前状态和事务意图的唯一事实源。
2. Payload 文件只有被 SQLite 引用后才属于业务事实。
3. Message、Event、Receipt 和 Audit 创建后不可原地改变含义。
4. Projection、Embedding、缓存和 JSONL 都可重建，不能成为唯一状态。

## 2. 执行

1. 一次用户意图对应一个 Turn，Turn 下可以有多个 RunAttempt，没有独立 Run 层。
2. 同一 Context Partition 同时最多一个改变短期上下文的 Turn 执行。
3. 等待用户或外部条件时不持有线程、事务、Lane 或 Lease。
4. 模型、网络、MCP、Channel 和 Shell 调用不得发生在数据库事务内。
5. 旧 Lease/旧版本 Attempt 的结果不得提交业务状态。

## 3. 幂等与副作用

1. 入站、Command、Task 触发、Event 消费和 Delivery 都有稳定幂等键。
2. 外部副作用执行前持久化意图，执行后保存 Receipt。
3. 结果未知时先 reconcile，不自动重复执行。
4. Event 只表示事实；需要改变其他聚合时提交 Command。

## 4. 上下文与认知

1. 不同 Channel/Conversation 不共享短期 Session。
2. 长期 Memory 必须保留来源、Scope、置信度和有效期。
3. 模型提取内容先成为 Candidate，不能直接成为不可纠正的永久事实。
4. Context Snapshot 必须可解释当时选择了哪些内容。

## 5. 安全

1. 模型只能提出动作，Policy Engine 决定权限。
2. 外部内容和 Tool 输出不能提升信任等级。
3. Shell、文件、网络和第三方插件使用受限 Runtime Context。
4. 当前控制面仅允许 loopback；开放远程访问必须先增加认证和 TLS。

## 6. 恢复

1. 重启后先恢复事实一致性，再恢复用户可见工作和后台任务。
2. 任何自动恢复不得增加重复副作用风险。
3. 预算、Attempt 次数和审批结果跨重启保留。
4. 数据备份必须能验证 SQLite、Payload manifest 和文件哈希。

## 7. 验证方式

每条不变量至少对应一种验证：数据库约束、Domain Service 条件更新、架构依赖测试、契约测试或恢复测试。仅写在文档中而无法自动或人工验证的不变量不算完成。
