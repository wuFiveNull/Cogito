# 系统边界与依赖规则

## 1. 进程边界

```text
interaction-web → agent-api ↔ agent-worker → sqlite/payload
                         ↕
                 channel-gateway(LangBot)
                         ↕
               model/MCP/external services
```

- `agent-api` 拥有即时 Turn 和 Command；
- `agent-worker` 通过 Lease 执行 Task、Delivery、Outbox 和维护任务；
- `channel-gateway` 拥有平台连接，不拥有 Agent 状态；
- `interaction-web` 只使用 Query/Command/Stream API。

个人部署可以合并进程，但逻辑所有权不改变。

## 2. 模块依赖方向

```text
domain ← application ← adapters/infrastructure
                   ↑
              plugin public API
```

禁止：

- Domain 导入数据库、HTTP、模型 SDK 或 Channel SDK；
- Agent Runtime 直接写 Repository；
- Event Handler 绕过 Command 修改其他聚合；
- Plugin 导入 Core 私有模块或核心 ORM；
- Dashboard 直接执行写 SQL。

## 3. 公开面

每个模块只公开：

```text
contracts
commands / queries
ports
service or facade
events
```

ORM、SDK 对象、内部缓存、锁和文件布局不属于跨模块契约。

## 4. 状态所有权

| 状态 | 唯一写入者 |
|---|---|
| Conversation/Session | Identity & Conversation Service |
| Turn/RunAttempt | Turn Service |
| Task/TaskAttempt | Task Service |
| MemoryItem | Memory Service |
| Delivery | Delivery Service |
| Approval | Approval Service |
| Plugin 状态 | Plugin Runtime |

其他模块通过 Command 或公开 Service 请求变更。

## 5. 外部边界

模型、LangBot、MCP、邮件、日历、GitHub 和文件系统都在系统边界之外。外部响应不能直接成为内部状态；必须经过契约转换、Schema 校验、Trust Label、Policy 和持久化提交。

## 6. 跨进程协议

所有协议必须包含 `schema_version`、ID、时间、TraceContext、origin 和安全错误。当前版本与前一兼容版本至少可并存；破坏性变化使用新版本和迁移窗口。

## 7. 架构测试

- 静态依赖测试检查禁止导入；
- Contract Test 检查协议兼容；
- Repository 测试检查只有所有者能更新聚合；
- Plugin 测试检查不能访问 Core 内部对象；
- 部署测试检查合并进程后仍使用相同端口和状态机。
