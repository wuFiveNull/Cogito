# PR 8.3 可靠性阶段收尾计划

## 1. 目标

本计划用于关闭 PR 8、PR 8.1、PR 8.2 遗留的可靠性与验收缺口，使当前 Stub 对话闭环具备一致、可恢复、可测试的执行语义。

完成后应满足：

```text
默认开发命令可运行
→ Lease 在外部调用前后都经过有效性校验
→ 不确定 Delivery 有持久 Receipt 可供 reconcile
→ Recovery 与 heartbeat 并发时保持聚合一致
→ 核心执行链数据库时间全部为 epoch milliseconds
→ Migration 可从 v5/v6 安全升级并可验证
→ 文档、代码和测试对实现状态描述一致
```

本阶段不新增模型、Agent Loop、Tool、Channel、Scheduler、Memory 检索或主动推送能力。

## 2. 权威依据

- `GLOBAL-INVARIANTS / 2. 执行`：旧 Lease 或旧 Attempt 的结果不得提交，外部调用不得发生在数据库事务内。
- `EXECUTION-LIFECYCLE / 3.3 完成 Attempt`：提交时验证 Turn 版本、active Attempt 和有效执行权。
- `EXECUTION-LIFECYCLE / 5.3 进程重启`：不明确副作用必须先对账，不能直接重试。
- `ACCESS-DELIVERY / 4.4 Delivery Attempt`：每次投递尝试保存请求 Hash、平台结果和错误信息。
- `ACCESS-DELIVERY / 4.7.2 Agent Core Recovery Scan`：unknown Delivery 通过 Receipt 执行 reconcile 或人工检查。
- `STREAMING-DELIVERY / 1. 核心原则`：每个已发生的平台副作用必须关联 Delivery、Attempt 和平台 Receipt。
- `STREAMING-DELIVERY / 3. 状态机`：unknown 表示平台可能成功但本地未确认，不得直接发送新消息。
- `DATABASE-SCHEMA / 1. SQLite 模式`：数据库时间统一使用 UTC epoch milliseconds INTEGER。
- `DATABASE-SCHEMA / 2. 表分组`：Delivery 聚合包含 `delivery_receipts`。
- `DATABASE-SCHEMA / 8. Migration`：Migration 必须包含兼容检查、数据校验和必要恢复说明。
- `TEST-EVALUATION / 3. 故障注入点`：覆盖外部成功与本地 Receipt 提交之间的崩溃窗口。

## 5. 工作包 A：可推进的测试时钟

### 5.1 问题

当前 Worker 接收单个 `datetime` 作为 clock。外部调用前后复用同一值，无法在确定性测试中表达“调用期间时间推进”。

### 5.2 实施

- 引入最小 `Clock` 接口或可调用时间源，至少提供 `now() -> datetime`；
- ProductionClock 返回当前 UTC 时间；
- FakeClock 支持显式 `advance()`，只用于测试；
- Dispatcher、OutboxWorker、DeliveryWorker、RecoveryService 使用同一时间来源；
- 保留现有显式 datetime 参数的兼容路径时，必须明确其作用域，避免两套时间语义长期并存；
- 禁止使用 `sleep` 测试 Lease。

### 5.3 验收

- Delivery 调用 Gateway 前读取一次时间；
- Gateway 返回后重新读取一次时间；
- 测试可以在 FakeGateway.send 内推进 FakeClock；
- 调用期间 Lease 到期时，不能提交 sent/failed/retry_scheduled。

## 6. 工作包 B：Delivery Receipt 与不确定结果

### 6.1 契约

在编码前先根据 `STREAMING-DELIVERY` 和 `DATABASE-SCHEMA` 固定最小 Receipt 契约。至少能够表达：

```text
receipt_id
delivery_id
delivery_attempt_id
operation_seq
request_hash
receipt_kind: confirmed | uncertain | reconciled
platform_message_id
safe_result
observed_at
lease_version
```

若字段超出现有权威文档，应先更新对应设计文档或记录 ADR，再创建物理 Schema。

### 6.2 Migration 0008

- 新建 `delivery_receipts`；
- 外键指向 Delivery 和 DeliveryAttempt；
- 为同一逻辑操作设置稳定唯一约束；
- 时间列使用 INTEGER epoch ms；
- 平台原始响应只保存受限 Payload 引用，不进入普通日志；
- 不修改 0001-0007。

### 6.3 外部发送流程

```text
短事务验证 sending + owner + version + expires_at
→ 事务外 Gateway.send
→ 重新读取当前时间
→ 短事务再次验证相同 Lease
   ├─ 仍有效：写 confirmed Receipt，提交 sent/failed/retry_scheduled
   └─ 已失权或过期：写 uncertain Receipt，Delivery = unknown
```

要求：

- 第二次校验必须使用 Gateway 返回后的新时间；
- uncertain 更新不能覆盖其他 Worker 的新 Lease；
- 如果当前 Delivery 已由 Recovery 进入 unknown，只补写幂等 Receipt；
- 如果发现 Delivery 已进入 sent/failed/cancelled，不回退状态，记录冲突供人工审查；
- reconcile 消费 uncertain Receipt，幂等写入 reconciled Receipt；
- unknown Delivery 不进入普通 lease_next。

## 7. 工作包 C：Recovery 聚合原子性

### 7.1 RunAttempt/Turn

- SELECT 保存 Turn version、active_attempt_id、Attempt lease_version 和 lease_expires_at；
- Attempt 条件更新必须验证读取到的 Lease 版本和过期时间；
- Attempt 更新成功后才能更新 Turn；
- Turn 更新必须验证 version 和 active_attempt_id；
- Turn 更新失败时整个聚合事务回滚，包括 Attempt 的 abandoned 更新；
- heartbeat 在 SELECT 与 UPDATE 之间续期时，Recovery 对该聚合零修改。

### 7.2 Outbox

- Recovery 条件更新验证 status、lease_version、lease_expires_at；
- 过期 Lease 推进 lease_version；
- 当前阶段没有真实 Broker 时保持至少一次发布语义；
- 后续 Event Consumer 必须依靠 Inbox 去重。

### 7.3 Delivery

- 过期 sending 进入 unknown；
- Recovery 写 uncertain Receipt 或确保后续发送 Worker可幂等补写；
- 不允许回 pending；
- 已有 Receipt 时按 Receipt 状态决定 reconcile/manual review，不猜测平台结果。

## 8. 工作包 D：时间与 Migration 收口

### 8.1 写路径审计

逐一检查以下模块，不允许向已迁移的 INTEGER 列写 ISO 字符串：

- `inbound_service.py`；
- `completion.py`；
- `dispatcher.py`；
- `outbox_worker.py`；
- `delivery_worker.py`；
- `recovery_service.py`；
- `repositories.py`；
- 测试 Fixture 和直接 SQL helper。

所有转换集中使用 `store/time_utils.py`。

### 8.2 Migration Runner

- 单个 Migration 和版本记录必须原子提交；
- 失败后不得出现“表已重建但版本未记录”；
- 执行后运行 `PRAGMA foreign_key_check`；
- 对无法解析的非空旧时间明确失败，不能静默转换为 0；
- 大型表重建若不属于 online-safe，应明确 maintenance 要求。

### 8.3 升级测试

- 构造真实 v5 数据库并升级到最新；
- 构造真实 v6 数据库并升级到最新；
- ISO、epoch ms 和 NULL 分别验证；
- 非法时间验证失败和回滚；
- Migration 中断后重新运行可恢复；
- 重复 migrate 幂等；
- foreign_key_check 返回空结果。

## 9. 工作包 E：测试矩阵

### 9.1 Lease

- NULL Lease 不能 complete/fail/heartbeat/publish/retry/deliver；
- 到期前可提交，到期时刻和到期后不可提交；
- heartbeat 只续期当前 owner/version；
- 旧 owner 和旧 version 不得提交；
- 两个独立 SQLite 连接最多领取一个逻辑对象。

### 9.2 Delivery 故障窗口

- Gateway 前 Lease 已过期：Gateway 调用次数为 0；
- Gateway 调用期间 Lease 自然过期：Delivery = unknown；
- Gateway 调用期间 Recovery 推进版本：Delivery = unknown；
- Gateway 成功、本地提交失败：存在 uncertain Receipt；
- 重复 reconcile 不产生第二个外部副作用；
- 新 Worker 不能领取 unknown Delivery；
- confirmed Receipt 存在时 Recovery 不回退 Delivery。

### 9.3 Recovery

- heartbeat 与 Recovery 竞态；
- Attempt 更新失败时 Turn 保持 running；
- Turn 更新失败时 Attempt 更新回滚；
- recover_all 重复执行幂等；
- 有效 Lease 不被回收。

### 9.4 工程命令

- 根目录默认 pytest 无需手工 PYTHONPATH；
- editable install 后模块 CLI 和 console script 均可运行；
- `init` 连续执行两次成功；
- `info` 不要求 API Key；
- config 兼容字段只产生明确弃用警告。
