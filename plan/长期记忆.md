# Agent 长期记忆与上下文管理系统设计文档

## 1. 文档信息

**文档状态：** Draft
**适用范围：** 单用户或多用户 Agent、聊天助手、任务型 Agent
**核心目标：** 在保证上下文连贯性的同时，为 Agent 提供可控、可追溯、可更新、可遗忘的长期记忆能力
**设计原则：**

1. 原始消息、会话摘要和长期记忆是三种不同的数据产物。
2. 长期记忆提取与上下文压缩独立运行。
3. 记忆必须可追溯到来源消息。
4. 用户明确表达的信息优先于模型推断。
5. 删除热记忆不等于删除历史数据。
6. 所有异步处理都必须绑定不可变消息范围。
7. 运行时召回应遵循最小必要原则，避免无关记忆污染上下文。
8. 不允许仅依赖 LLM 自主决定所有记忆生命周期。

---

# 2. 背景与问题定义

长时间运行的 Agent 通常会遇到以下问题：

* 模型上下文窗口有限，无法永久保留所有历史消息。
* 用户希望 Agent 记住偏好、身份信息、项目状态和过去决策。
* 部分信息只在当前任务中有效，不应成为全局长期记忆。
* 用户的偏好和事实可能发生变化。
* 错误记忆、过期记忆和冲突记忆会降低回答质量。
* 长期记忆全部注入提示词会导致 token 成本增加和注意力污染。
* 用户可能要求“忘记”某条信息或删除所有历史。
* 后台任务执行期间，用户可能继续发送新消息。
* 记忆提取和上下文压缩可能失败、重试或重复运行。

因此，本系统需要同时解决：

```text
历史保留
长期记忆
当前任务状态
上下文压缩
相关记忆召回
冲突处理
遗忘与删除
审计与隐私
```

---

# 3. 目标与非目标

## 3.1 目标

系统需要支持：

* 保存完整原始会话。
* 从对话中提取高价值长期记忆。
* 按用户、项目、会话和任务进行记忆隔离。
* 保留近期原始上下文。
* 对较早上下文生成滚动摘要。
* 按查询召回相关长期记忆。
* 更新、覆盖、合并、归档和删除记忆。
* 区分用户明确事实和模型推断。
* 支持用户查看、修改和删除记忆。
* 支持任务重试、幂等和并发处理。
* 支持记忆质量评估与线上监控。

## 3.2 非目标

第一阶段不追求：

* 完整模拟人类记忆。
* 自动构建通用知识图谱。
* 使用记忆替代业务数据库。
* 让模型直接修改所有底层数据。
* 永久保存所有低价值细节。
* 自动推断敏感个人属性。
* 仅依靠向量相似度决定所有召回结果。

# 4. 核心概念

## 4.1 原始消息

用户和 Agent 的完整对话记录，是系统的事实来源。

特点：

* 追加写入。
* 不因记忆提取完成而删除。
* 不因上下文压缩而删除。
* 可按数据保留策略归档或清理。
* 所有摘要和记忆都应能追溯到原始消息。

## 4.2 热记忆

小型、高密度、经常注入上下文的记忆集合。

典型内容：

* 用户首选语言。
* 稳定沟通风格。
* 当前活跃项目。
* 长期明确偏好。
* 极重要约束。

热记忆容量应严格限制，例如：

```text
用户热记忆：1,500～3,000 tokens
项目热记忆：1,000～2,000 tokens
```

## 4.3 语义长期记忆

结构化保存的事实、偏好、决策和经验，运行时按需检索。

例如：

```text
用户偏好 TypeScript。
项目 Alpha 使用 PostgreSQL。
用户不喜欢包含大量表情符号的回答。
2026 年 6 月项目已从 Python 3.10 升级到 3.12。
```

## 4.4 情景记忆

关于过去事件和经历的记录。

例如：

* 某次部署失败的原因。
* 用户曾经尝试过某个方案。
* 一次重要项目决策的过程。
* 某次任务的最终结果。

情景记忆通常带有时间、项目和会话来源。

## 4.5 会话摘要

对旧会话内容的结构化压缩，目的是维持当前会话连续性。

会话摘要不是长期记忆。

摘要可能包含临时任务状态，而长期记忆应只包含未来仍可能有价值的信息。

## 4.6 任务状态

当前尚未完成任务的结构化状态。

例如：

```json
{
  "goal": "完成数据库迁移",
  "completed_steps": ["建立新表", "迁移用户数据"],
  "pending_steps": ["迁移订单数据"],
  "blockers": ["等待生产环境凭据"]
}
```

任务状态可以存入当前会话，但不应默认成为用户全局记忆。

# 6. 记忆分层

建议采用四层结构。

| 层级         | 内容             |        保存周期 | 注入方式    |
| ---------- | -------------- | ----------: | ------- |
| L0 原始历史    | 完整消息和工具调用      |     按数据保留策略 | 不直接全量注入 |
| L1 会话状态    | 当前摘要、任务状态、近期消息 | 当前会话或任务生命周期 | 默认注入    |
| L2 热记忆     | 高价值用户偏好和当前项目约束 |    长期，但容量受限 | 经常注入    |
| L3 语义/情景记忆 | 事实、决策、经历、模式    |     按生命周期管理 | 检索后注入   |

推荐的运行时上下文结构：

```text
System Instructions

Stable User Profile Snapshot

Active Project Hot Memory

Retrieved Relevant Memories

Structured Conversation Summary

Recent Raw Messages

Current User Message
```

# 7. 记忆分类

## 7.1 用户事实

例如：

* 用户职业。
* 用户常用语言。
* 用户所在时区。
* 用户长期技术背景。

作用域：

```text
scope_type = user
```

## 7.2 用户偏好

例如：

* 回答偏好简洁。
* 代码默认使用 TypeScript。
* 不喜欢表格。
* 喜欢先给结论。

偏好必须记录：

* 是否由用户明确表达。
* 是否仅适用于特定任务。
* 最后确认时间。
* 是否可能被新偏好覆盖。

## 7.3 项目事实

例如：

* 项目使用 PostgreSQL。
* 仓库采用 monorepo。
* API 使用 REST。
* 当前生产版本为 v2。

作用域：

```text
scope_type = project
scope_id = project_alpha
```

## 7.4 决策记忆

例如：

* 团队决定不使用 Redis。
* 项目选择事件驱动架构。
* 数据保留周期定为 90 天。

决策应保留：

* 决策内容。
* 决策原因。
* 决策时间。
* 参与者。
* 是否已被后续决策替代。

## 7.5 任务记忆

例如：

* 当前正在完成支付模块。
* 已完成接口定义。
* 下一步是实现退款流程。

任务记忆通常带有过期条件：

```text
valid_until = task_closed
```

## 7.6 情景记忆

例如：

* 某次发布因数据库锁失败。
* 用户曾经尝试某个方案但效果不好。

情景记忆一般不放入热记忆，而通过检索召回。

## 7.7 临时上下文

例如：

* 本次回答不使用代码。
* 当前讨论只考虑移动端。
* 接下来三轮使用英文。

此类信息通常仅存于会话状态，不进入长期记忆。

# 8. 消息与会话存储设计

## 8.1 原始消息表

```sql
CREATE TABLE conversation_messages (
    id                  UUID PRIMARY KEY,
    tenant_id           UUID NOT NULL,
    user_id             UUID NOT NULL,
    conversation_id     UUID NOT NULL,
    session_id          UUID NOT NULL,
    sequence_number     BIGINT NOT NULL,
    role                VARCHAR(32) NOT NULL,
    content             JSONB NOT NULL,
    token_count         INTEGER,
    created_at          TIMESTAMP NOT NULL,
    deleted_at          TIMESTAMP,
    UNIQUE(session_id, sequence_number)
);
```

`content` 可以保存：

```json
{
  "text": "用户消息",
  "attachments": [],
  "tool_calls": [],
  "tool_results": []
}
```

## 8.2 会话状态表

```sql
CREATE TABLE conversation_state (
    conversation_id                 UUID PRIMARY KEY,
    latest_sequence                 BIGINT NOT NULL DEFAULT 0,
    memory_extracted_upto_sequence  BIGINT NOT NULL DEFAULT 0,
    provider_synced_upto_sequence   BIGINT NOT NULL DEFAULT 0,
    summarized_upto_sequence        BIGINT NOT NULL DEFAULT 0,
    active_summary_version          INTEGER,
    last_activity_at                TIMESTAMP NOT NULL,
    status                          VARCHAR(32) NOT NULL,
    updated_at                      TIMESTAMP NOT NULL
);
```

核心处理水位必须独立：

```text
memory_extracted_upto_sequence
provider_synced_upto_sequence
summarized_upto_sequence
```

不允许使用一个统一的 `processed=true` 代替。

## 8.3 处理任务表

```sql
CREATE TABLE processing_jobs (
    id                  UUID PRIMARY KEY,
    job_type            VARCHAR(64) NOT NULL,
    conversation_id     UUID NOT NULL,
    from_sequence       BIGINT NOT NULL,
    to_sequence         BIGINT NOT NULL,
    input_version       INTEGER NOT NULL,
    status              VARCHAR(32) NOT NULL,
    retry_count         INTEGER NOT NULL DEFAULT 0,
    idempotency_key     VARCHAR(256) NOT NULL UNIQUE,
    created_at          TIMESTAMP NOT NULL,
    started_at          TIMESTAMP,
    completed_at        TIMESTAMP,
    error_code          VARCHAR(64),
    error_message       TEXT
);
```

任务范围必须固定：

```text
from_sequence = 81
to_sequence = 120
```

执行过程中出现的新消息不进入当前任务。

# 9. 长期记忆数据模型

```sql
CREATE TABLE memories (
    id                      UUID PRIMARY KEY,
    tenant_id               UUID NOT NULL,
    user_id                 UUID NOT NULL,

    scope_type              VARCHAR(32) NOT NULL,
    scope_id                UUID,

    memory_type             VARCHAR(64) NOT NULL,
    canonical_key           VARCHAR(256),
    content                 TEXT NOT NULL,
    structured_value        JSONB,

    source_type             VARCHAR(32) NOT NULL,
    source_conversation_id  UUID,
    source_session_id       UUID,
    source_from_sequence    BIGINT,
    source_to_sequence      BIGINT,

    explicitness            VARCHAR(32) NOT NULL,
    confidence              DECIMAL(5,4) NOT NULL,
    importance              DECIMAL(5,4) NOT NULL,

    valid_from              TIMESTAMP,
    valid_until             TIMESTAMP,
    last_confirmed_at       TIMESTAMP,
    last_retrieved_at       TIMESTAMP,
    retrieval_count         INTEGER NOT NULL DEFAULT 0,

    status                  VARCHAR(32) NOT NULL,
    supersedes_memory_id    UUID,
    superseded_by_memory_id UUID,

    embedding               VECTOR,
    metadata                JSONB,

    version                 INTEGER NOT NULL DEFAULT 1,
    created_at              TIMESTAMP NOT NULL,
    updated_at              TIMESTAMP NOT NULL,
    archived_at             TIMESTAMP,
    deleted_at              TIMESTAMP
);
```

## 9.1 关键字段说明

### `scope_type`

支持：

```text
global
user
workspace
project
conversation
task
```

### `memory_type`

支持：

```text
user_fact
user_preference
project_fact
decision
task_state
episodic_event
skill_pattern
negative_preference
constraint
relationship
```

### `explicitness`

支持：

```text
explicit_user_statement
confirmed_inference
model_inference
external_source
system_generated
```

优先级：

```text
explicit_user_statement
>
confirmed_inference
>
external_source
>
model_inference
```

### `status`

支持：

```text
candidate
active
superseded
expired
archived
invalidated
deleted
```

# 10. 在线记忆写入流程

每轮用户消息写入后，系统可以运行低成本候选检测器。

## 10.1 候选检测

检测以下信号：

* “以后……”
* “请记住……”
* “我喜欢……”
* “我不喜欢……”
* “我们决定……”
* “项目使用……”
* “改成……”
* “之前说错了……”
* “不要再记得……”
* “这只适用于当前任务……”

候选检测输出：

```json
{
  "should_extract": true,
  "candidates": [
    {
      "type": "user_preference",
      "scope": "user",
      "content": "用户偏好使用 TypeScript",
      "explicitness": "explicit_user_statement",
      "confidence": 0.99,
      "importance": 0.82,
      "operation_hint": "upsert"
    }
  ]
}
```

## 10.2 在线路径适用范围

只立即处理以下内容：

* 用户明确要求记住。
* 用户明确修改既有记忆。
* 用户明确要求忘记。
* 高置信度、低歧义的稳定偏好。
* 当前项目的重要技术决策。
* 需要立即影响后续回答的约束。

不建议在线直接保存：

* 情绪推断。
* 偶然提到的信息。
* 未确认的个人属性。
* 对敏感信息的推断。
* 需要结合后续上下文判断的事实。

# 11. 批量记忆提取流程

## 11.1 触发条件

批量提取可以由以下事件触发：

```text
用户空闲超过 5 分钟
会话结束
上下文压缩前
达到消息数量阈值
达到 token 阈值
系统定期扫描未处理消息
```

推荐默认规则：

```text
idle_timeout = 5 分钟
minimum_messages = 4
minimum_tokens = 1,000
maximum_unprocessed_tokens = 12,000
```

## 11.2 处理流程

```text
1. 读取 memory_extracted_upto_sequence。
2. 固定当前 latest_sequence 为 cutoff。
3. 创建 [from_sequence, cutoff] 提取任务。
4. 运行显著性判断。
5. 提取候选记忆。
6. 查找相似或同 canonical_key 的现有记忆。
7. 执行新增、更新、覆盖、合并或忽略。
8. 写入来源范围。
9. 原子更新 memory_extracted_upto_sequence。
```

## 11.3 提取输出格式

```json
{
  "memories": [
    {
      "memory_type": "project_fact",
      "scope_type": "project",
      "scope_id": "project_alpha",
      "canonical_key": "project_alpha.runtime.python_version",
      "content": "Project Alpha 使用 Python 3.12",
      "structured_value": {
        "value": "3.12"
      },
      "explicitness": "explicit_user_statement",
      "confidence": 0.98,
      "importance": 0.75,
      "operation": "supersede",
      "supersedes": "memory-id"
    }
  ],
  "ignored_items": [
    {
      "reason": "temporary_instruction"
    }
  ]
}
```

# 12. 记忆合并与冲突处理

## 12.1 Canonical Key

对于可结构化的事实，使用稳定键：

```text
user.preferred_language
user.response_style
project_alpha.database
project_alpha.python_version
task_123.status
```

具有相同 canonical key 的记忆进入冲突判断。

## 12.2 冲突优先级

冲突时按以下顺序判断：

1. 用户最近的明确表达。
2. 用户明确确认过的信息。
3. 权威外部数据源。
4. 高置信度模型推断。
5. 较旧模型推断。

示例：

```text
旧记忆：用户偏好暗色主题
新消息：以后改用亮色主题
```

处理：

```text
旧记忆.status = superseded
旧记忆.superseded_by_memory_id = 新记忆.id
新记忆.supersedes_memory_id = 旧记忆.id
```

不建议直接物理覆盖旧记录，否则无法审计变化。

## 12.3 合并

以下记忆可以合并：

```text
用户喜欢简洁回答
用户不喜欢冗长解释
```

合并为：

```text
用户偏好简洁、直接的回答，除非明确要求详细解释。
```

但必须保留多个来源引用。

可以增加来源关联表：

```sql
CREATE TABLE memory_sources (
    memory_id          UUID NOT NULL,
    conversation_id    UUID NOT NULL,
    session_id         UUID NOT NULL,
    from_sequence      BIGINT NOT NULL,
    to_sequence        BIGINT NOT NULL,
    created_at         TIMESTAMP NOT NULL
);
```

# 13. 记忆重要性评分

建议将确定性规则与模型评分结合。

## 13.1 基础评分

```text
memory_value =
    0.25 × explicitness
  + 0.20 × future_utility
  + 0.15 × stability
  + 0.15 × scope_relevance
  + 0.10 × confidence
  + 0.10 × retrieval_frequency
  + 0.05 × recency
```

各项归一化为 0～1。

## 13.2 显式性评分

```text
用户明确要求记住          1.00
用户明确陈述              0.95
用户确认模型总结          0.90
权威外部来源              0.80
多次重复出现的模式        0.70
单次模型推断              0.35
```

## 13.3 未来价值

高价值内容：

* 长期偏好。
* 稳定身份信息。
* 项目架构决策。
* 用户长期目标。
* 重复使用的工作方式。
* 关键失败经验。

低价值内容：

* 一次性闲聊。
* 当前时间。
* 临时情绪。
* 已完成且不可复用的微小步骤。
* 无后续意义的工具输出。

# 14. 热记忆管理

热记忆不是简单复制高分长期记忆，而是经过压缩和策展的运行时快照。

## 14.1 热记忆容量

建议：

```text
用户热记忆最大 2,000 tokens
项目热记忆最大 1,500 tokens
总热记忆最大 3,500 tokens
```

## 14.2 热记忆刷新时机

```text
新 session 开始
用户明确更新关键偏好
项目切换
热记忆版本过期
重要记忆新增
定期整理任务
```

## 14.3 热记忆内容格式

```text
## User Preferences
- 默认使用中文。
- 偏好直接、结构化的回答。
- 技术示例优先使用 TypeScript。

## Active Project
- 项目：Alpha
- 数据库：PostgreSQL
- 运行时：Node.js 22
- 当前阶段：支付模块重构

## Stable Constraints
- 不要自动发送外部消息。
- 删除数据前必须明确确认。
```

## 14.4 热记忆溢出策略

当容量不足时，处理顺序应为：

```text
1. 删除已失效条目。
2. 删除已被覆盖条目。
3. 将低价值条目移出热记忆，但保留在长期记忆库。
4. 合并语义重复条目。
5. 压缩表达。
6. 最后才归档低价值长期记忆。
```

注意：

> 从热记忆移除不等于从长期记忆删除。

# 15. 记忆召回设计

## 15.1 召回输入

召回查询由以下内容组合：

```text
当前用户消息
当前任务目标
当前项目
当前会话摘要
最近若干轮消息
```

不建议仅使用当前一条用户消息进行向量搜索。

## 15.2 召回流程

```text
1. 根据 user_id 和 tenant_id 做强隔离。
2. 根据 scope 过滤。
3. 排除 deleted、expired、superseded 状态。
4. 执行关键词检索。
5. 执行向量召回。
6. 执行 canonical key 精确匹配。
7. 合并候选。
8. 使用规则和模型进行重排。
9. 执行 token 预算裁剪。
10. 注入提示词。
```

## 15.3 混合检索评分

```text
retrieval_score =
    0.35 × semantic_similarity
  + 0.20 × keyword_match
  + 0.15 × scope_match
  + 0.10 × importance
  + 0.10 × confidence
  + 0.05 × recency
  + 0.05 × retrieval_history
```

## 15.4 负向过滤

以下记忆不得注入：

* 与当前项目无关。
* 已被新记忆覆盖。
* 已过期。
* 用户已要求忘记。
* 敏感信息且当前任务无必要。
* 仅为低置信度推断。
* 与当前输入存在明显冲突。

## 15.5 召回数量

推荐：

```text
初始召回：20～50 条
重排后：5～12 条
最终注入：不超过 1,500～3,000 tokens
```

# 16. 上下文压缩设计

## 16.1 压缩触发

上下文压缩由 token 使用率触发，而不是由用户是否空闲触发。

建议阈值：

```text
soft_threshold = 65%
background_threshold = 75%
hard_threshold = 85%
emergency_threshold = 95%
```

处理逻辑：

```text
低于 65%：不处理
65%～75%：记录压力
75%～85%：后台预生成摘要
85%～95%：当前轮前完成压缩
超过 95%：使用快速摘要或强制裁剪兜底
```

## 16.2 压缩结构

```text
早期关键消息：保留
中间历史消息：压缩为摘要
最近消息：原样保留
```

推荐保留：

```text
最初 2～4 条关键消息
最近 10～30 条消息
至少最近 8,000～20,000 tokens
```

## 16.3 摘要格式

```json
{
  "conversation_goal": "",
  "user_intent": "",
  "confirmed_facts": [],
  "constraints": [],
  "decisions": [],
  "completed_work": [],
  "current_state": [],
  "open_questions": [],
  "pending_actions": [],
  "artifacts": [],
  "important_references": [],
  "errors_and_failed_attempts": [],
  "critical_quotes": [],
  "summary": ""
}
```

## 16.4 滚动摘要

第二次压缩时使用：

```text
旧摘要
+
尚未进入摘要的中间消息
→
新摘要
```

每个摘要必须记录：

```text
covers_from_sequence
covers_to_sequence
summary_version
model_version
prompt_version
```

## 16.5 压缩与长期记忆的关系

两者基于同一个消息快照，可以并行执行。

```text
消息 1～100
├── 长期记忆提取
└── 会话摘要生成
```

但只有摘要成功后，运行时上下文才能替换旧消息。

长期记忆提取失败不应阻塞当前会话，因为原始消息仍然存在，可以稍后重试。

# 17. 遗忘与删除设计

遗忘必须区分多种语义。

## 17.1 从热记忆移除

```text
remove_from_hot_memory
```

效果：

* 不再默认注入提示词。
* 长期记忆库仍保留。
* 后续查询仍可能召回。

适用于：

* 低频信息。
* 当前不活跃项目。
* 容量不足。
* 暂时不需要的偏好。

## 17.2 归档

```text
archive_memory
```

效果：

* 默认不召回。
* 数据仍可审计。
* 可以恢复。

适用于：

* 旧项目。
* 已完成任务。
* 很久未使用但仍可能有历史价值的信息。

## 17.3 失效

```text
invalidate_memory
```

效果：

* 记忆被认定为错误。
* 不再召回。
* 保留原因和审计记录。

适用于：

* 模型提取错误。
* 来源不可靠。
* 用户明确否认。

## 17.4 覆盖

```text
supersede_memory
```

效果：

* 新记忆取代旧记忆。
* 旧记忆保留，但状态为 superseded。

适用于：

* 用户偏好变化。
* 项目版本升级。
* 决策更新。

## 17.5 永久删除

```text
delete_permanently
```

效果：

* 删除记忆记录。
* 删除向量索引。
* 删除缓存。
* 根据用户请求决定是否删除来源消息。
* 写入合规审计日志，但审计日志不得保留被删除内容正文。

适用于：

* 用户明确要求删除。
* 合规请求。
* 敏感数据误存。
* 数据保留周期到期。

# 18. 自动遗忘策略

不建议仅通过提示词让模型自由删除。

采用三层机制。

## 18.1 确定性失效

满足以下条件时直接退出活跃集合：

```text
valid_until 已过期
任务已结束
项目已归档
被新记忆 supersede
用户明确删除
来源消息已被合规删除且记忆无法独立保留
```

## 18.2 自动降级

定义保留分数：

```text
retention_score =
    0.30 × importance
  + 0.20 × retrieval_frequency
  + 0.15 × explicitness
  + 0.15 × confidence
  + 0.10 × recency
  + 0.10 × scope_activity
```

建议规则：

```text
score ≥ 0.70：保留活跃
0.45 ≤ score < 0.70：保留长期记忆，但移出热记忆
0.25 ≤ score < 0.45：归档
score < 0.25：进入删除候选，不直接自动物理删除
```

## 18.3 LLM 语义整理

模型只负责：

* 合并重复记忆。
* 压缩表达。
* 识别可能过时的语义。
* 给出归档建议。
* 判断两条记忆是否等价。

模型不能无审计地直接物理删除高价值记忆。

# 19. 时间衰减

不是所有记忆都应该衰减。

## 19.1 不衰减或弱衰减

* 用户明确长期偏好。
* 身份信息。
* 安全约束。
* 稳定项目决策。
* 用户明确要求永久记住的信息。

## 19.2 中等衰减

* 工具偏好。
* 工作习惯。
* 常见任务模式。
* 项目环境信息。

## 19.3 强衰减

* 临时任务状态。
* 短期计划。
* 临时位置。
* 一次性意图。
* 当前情绪。

时间衰减示例：

```text
recency_score = exp(-age_days / half_life_days)
```

其中：

```text
用户长期偏好 half_life = 365 天
项目事实 half_life = 90 天
任务状态 half_life = 7 天
临时上下文 half_life = 1 天
```

时间衰减只影响排序和热记忆资格，不应自动证明事实已经错误。

# 22. 并发与一致性

## 22.1 快照范围

所有后台任务必须固定：

```text
conversation_id
from_sequence
to_sequence
input_version
```

用户继续发消息时，不影响当前任务范围。

## 22.2 水位更新

只有任务成功提交结果后，才能更新水位。

```sql
UPDATE conversation_state
SET memory_extracted_upto_sequence = 100
WHERE conversation_id = :id
  AND memory_extracted_upto_sequence = 80;
```

使用 Compare-And-Swap 防止并发覆盖。

## 22.3 幂等

任务唯一键：

```text
memory_extract:{conversation_id}:{from_seq}:{to_seq}:{prompt_version}
```

重复执行同一任务不得生成重复记忆。

## 22.4 乐观锁

记忆更新使用版本号：

```text
UPDATE memories
SET content = :new_content,
    version = version + 1
WHERE id = :id
  AND version = :expected_version;
```

冲突时重新读取并合并。

# 25. 安全与隐私

## 25.1 敏感数据分类

以下信息默认不得由模型自动推断并持久化：

* 种族和民族。
* 宗教。
* 性取向。
* 健康状况。
* 精确位置。
* 政治倾向。
* 财务账户。
* 身份证件。
* 密码和访问令牌。
* 私密关系信息。

仅在用户明确要求且业务允许时存储。

## 25.2 最小化原则

系统只保存未来交互真正需要的信息。

例如：

```text
不应保存：
用户某天晚上 11:32 感到疲惫。

可以保存：
用户明确表示偏好上午安排复杂任务。
```

## 25.3 租户隔离

所有查询必须强制带：

```text
tenant_id
user_id
```

不得依赖模型在提示词中遵守隔离。

## 25.4 加密

* 传输层使用 TLS。
* 数据库敏感字段加密。
* 向量索引与主库保持相同访问控制。
* 删除流程覆盖向量和缓存副本。
* 密钥采用独立 KMS 管理。

# 30. 推荐状态转换

```text
candidate
    │
    ├── 验证成功 ─────────→ active
    │
    ├── 低价值 ───────────→ archived
    │
    └── 不可信 ───────────→ invalidated

active
    │
    ├── 新事实覆盖 ───────→ superseded
    ├── 超过有效期 ───────→ expired
    ├── 长期不活跃 ───────→ archived
    ├── 用户否认 ─────────→ invalidated
    └── 用户永久删除 ─────→ deleted
```

物理删除可以在 `deleted` 状态后异步执行。

# 31. 核心伪代码

## 31.1 每轮消息处理

```python
def on_message(message):
    stored = append_message(message)
    update_conversation_activity(stored)

    candidate = detect_memory_candidate(stored)

    if candidate.is_explicit_forget_request:
        process_forget_request(candidate)

    elif candidate.should_write_immediately:
        upsert_memory_candidate(candidate)

    context = build_runtime_context(
        user_id=stored.user_id,
        conversation_id=stored.conversation_id,
        current_message=stored,
    )

    if context.usage_ratio >= 0.85:
        context = compress_context_sync(context)
    elif context.usage_ratio >= 0.75:
        enqueue_background_compression(context)

    return run_agent(context)
```

def on_conversation_idle(conversation_id):
    state = lock_conversation_state(conversation_id)

    from_seq = state.memory_extracted_upto_sequence + 1
    to_seq = state.latest_sequence

    if from_seq > to_seq:

```python
def apply_memory_candidate(candidate):
    existing = find_by_canonical_key(
        user_id=candidate.user_id,
        scope=candidate.scope,
        canonical_key=candidate.canonical_key,
    )

    if existing is None:
        return create_memory(candidate)

    if candidate.explicitness > existing.explicitness:
        return supersede(existing, candidate)

    if candidate.is_newer_explicit_statement:
        return supersede(existing, candidate)

    if semantically_equivalent(existing, candidate):
        return reinforce(existing, candidate)

    return create_conflict_review(existing, candidate)
```

# 32. 最终设计结论

推荐采用以下总体方案：

```text
完整原始消息作为事实来源
+
近期原始消息维持精确上下文
+
滚动摘要维持长会话连续性
+
热记忆保存最重要的稳定信息
+
语义长期记忆按需检索
+
结构化任务状态管理当前工作
```

长期记忆的生命周期应是：

```text
候选
→ 验证
→ 激活
→ 召回和强化
→ 更新或覆盖
→ 移出热记忆
→ 归档、失效或删除
```

最关键的工程约束是：

1. 不因记忆提取完成删除原始消息。
2. 不用单一 `processed` 状态代表所有后台处理。
3. 每个任务必须绑定固定消息范围。
4. 上下文压缩由 token 压力触发。
5. 记忆整理由显式事件、空闲和 session 生命周期触发。
6. 用户明确表达优先于模型推断。
7. 热记忆移除、归档、失效和永久删除必须是不同操作。
8. LLM 可以辅助语义判断，但确定性规则负责权限、作用域、过期和删除。
9. 所有记忆都应带来源、置信度、作用域和版本。
10. 用户必须能够查看和控制系统保存的长期记忆。
