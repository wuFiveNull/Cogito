# 02 运行时、认知、检索与恢复下一步开发计划

> 状态：draft  
> 对应目录：`markdown/02_runtime`  
> 主要设计依据：`RUNTIME-FLOWS`、`AGENT-COGNITION`、`EXECUTION-LIFECYCLE`、`SESSION-CONTEXT`、`AGENT-LOOP`、`RETRIEVAL-CONTEXT`、`MEMORY-LIFECYCLE`、`MODEL-ADAPTER`

## 1. 目标

在现有可运行 Turn、Agent Loop、Context、Memory 和 Model Router 基础上，把运行时从“主链路可执行”提升到“可解释、可取消、可恢复、可降级、可评测”。

本阶段核心结果：

1. Turn、RunAttempt、Checkpoint 和恢复语义完全对齐；
2. waiting_user/waiting_external 不占用事务、Lane 或 Lease；
3. Agent Loop 的 Tool、审批、取消和重复循环都有确定行为；
4. Context Snapshot 能解释每个条目的来源、分数、Token 和检索路径；
5. Memory 生命周期、Embedding 版本和 Markdown 派生视图可重建；
6. Model Adapter 具备能力协商、错误映射、流事件顺序和离线契约测试。

设计引用：

- `RUNTIME-FLOWS / 1. 三类控制模型与执行边界`
- `RUNTIME-FLOWS / 3. 并发、顺序、幂等与一致性`
- `EXECUTION-LIFECYCLE / 5. 重试、等待与恢复`
- `SESSION-CONTEXT / 5. Context Partition Lane`
- `AGENT-LOOP / 3. 单轮协议`
- `RETRIEVAL-CONTEXT / 10. Context Snapshot`
- `MEMORY-LIFECYCLE / 10. Consolidation`

## 2. 当前基线与缺口

### 2.1 已有

- Inbound 创建 Message/Turn/Outbox 的事务链；
- Dispatcher、Lane、RunAttempt、Completion 和 Recovery Service；
- Agent Loop 的最终回复、Tool 调用和终止守卫；
- Context Builder 和不可变 ContextSnapshot 基础；
- FTS5、Embedding、Query Plan、Memory 生命周期和记忆工具；
- Stub、Echo、OpenAI-compatible Provider 及 ModelRouter；
- 流式 Provider 调用和流式 Delivery 集成；
- Task/摘要/记忆提取后台处理。

### 2.2 缺口

- Checkpoint 还未覆盖 Tool Receipt、待审批、预算消耗和可恢复步骤；
- waiting_user 审批恢复到新 RunAttempt 的端到端路径不足；
- 部分 Loop 和 Runtime 逻辑直接依赖具体 Repository；
- Context Snapshot 的 selection policy/version、excluded_summary、retrieval_path 记录不完整；
- 检索评测、降级路径和稳定性测试不足；
- Memory Markdown 派生视图、删除 tombstone、Embedding 重建未完全闭环；
- Provider 契约测试尚未覆盖取消、限流、Context Overflow 和错误脱敏全矩阵；
- ResourceBudget 未成为 Turn/Attempt/Model/Tool 的统一硬约束。

## 4. 目标执行模型

```text
Inbound committed
→ Lane claim
→ create RunAttempt + immutable ContextSnapshot
→ bounded Agent Loop
→ Policy/Tool/Model outside DB transaction
→ checkpoint after recoverable boundary
→ commit final Message + Delivery/Event Outbox
→ complete Attempt/Turn
```

等待和恢复：

```text
require approval / external wait
→ persist checkpoint + waiting condition
→ end current Attempt and release Lane/Lease
→ Command/Event satisfies condition
→ create a new Attempt
→ validate old receipts/config/snapshot
→ resume from deterministic checkpoint
```

## 5. 里程碑

### M1：Turn/RunAttempt 生命周期收敛

#### 工作项

1. 对照 `EXECUTION-LIFECYCLE / 1～3` 核对字段、状态和事务；
2. 明确 Turn 逻辑状态与 Attempt 执行状态，不允许复活旧 Attempt；
3. 所有完成、失败、取消、等待操作使用条件更新：
   - expected version；
   - active attempt id；
   - lease owner/version；
4. 重试、恢复和审批续跑统一创建新 Attempt；
5. 旧 Worker 返回结果只能记录诊断，不能提交最终 Message/Delivery；
6. 统一 ErrorEnvelope 分类和 retryable 判断。

#### 测试

- 同一 Turn 双 Worker 竞争；
- Lease 过期后旧 Attempt 提交；
- 完成与取消同时发生；
- waiting_user 与 expired 同时发生；
- 重启后新 Attempt 编号单调；
- Delivery 失败不回滚 Turn 完成。

### M2：持久 Checkpoint 与恢复决策器

#### Checkpoint 最小结构

```text
checkpoint_id
turn_id / attempt_id
current_step
completed_step_ids
context_snapshot_id
tool_calls[{id,status,receipt_ref}]
pending_approval_id
partial_result_ref
budget_consumed
config_version
capability_snapshot_version
created_at
```

#### 工作项

1. 在以下边界写 Checkpoint：Context 完成、模型响应完成、Tool 意图持久化、Tool Receipt 完成、进入等待；
2. 不序列化 Provider SDK、数据库连接、Coroutine 或 Python 栈；
3. 实现 RecoveryDecision：resume、retry、reconcile、waiting_user、manual_review、fail；
4. 恢复前验证取消状态、Receipt、配置兼容、Provider/Tool 能力和预算；
5. side_effect_unknown 必须先进入 reconcile；
6. 每次恢复记录父 Attempt、Recovery 原因和决策证据。

#### 故障注入

- Tool 执行前；
- Tool 成功后 Receipt 写入前；
- Model 完成后最终事务前；
- Assistant Message 写入后 Delivery Outbox 前；
- waiting_user 持久化前后；
- 流式最终化前。

### M3：取消、等待和审批续跑

#### 工作项

1. CancelTurn Command 设置持久 cancel_requested_at，并触发进程内取消 token；
2. Agent Loop、Model 流、Tool Executor 在安全点检查取消；
3. 已发生的副作用不声称撤销，必要时创建补偿 Task；
4. Policy RequireApproval 时：
   - 持久化 ApprovalRequest；
   - 保存 ToolRequest 和参数快照；
   - 当前 Attempt 结束；
   - Turn 进入 waiting_user；
   - 释放 Lane；
5. 审批 Command 验证 Principal、状态、过期时间和 expected_version；
6. 成功消费审批后仅创建一个恢复 Attempt；
7. rejected/expired 形成可解释的 Agent 结果，不无限询问。

#### 验收

- 等待期间数据库无长事务、进程无被占 Lane；
- 重复 approve 不产生重复 Attempt；
- 非 Owner 不能批准高风险 Tool；
- 取消优先于普通消息并能终止流；
- 进程重启后审批仍可继续。

### M4：Agent Loop 协议与预算

#### 工作项

1. LoopState 明确记录轮次、模型调用数、Tool 调用数、重复签名和剩余预算；
2. Tool 参数在进入 Executor 前用原始 Schema 校验；
3. 并行 Tool 结果按 tool_call_id/声明顺序稳定合并；
4. 结构化输出失败只允许预算内修复一次；
5. 重复 Tool 调用签名达到阈值后停止并返回安全说明；
6. Provider 已交付不可回滚 Tool Call 后禁止 fallback；
7. ResourceBudget 至少限制：
   - max_loop_iterations；
   - max_model_calls；
   - max_tool_calls；
   - max_input/output_tokens；
   - max_wall_time；
   - max_cost；
8. Budget 跨 Attempt 恢复，不能通过重试归零。

#### 对应验收

完整覆盖 `AGENT-LOOP / 11. 验收测试` 的七类场景，并给每项稳定测试 ID。

### M5：Session、Lane 与 Context Snapshot

#### 工作项

1. Session Resolver 统一输入：Channel、Conversation、Thread、Principal 隔离策略、reset_generation；
2. 不同 Channel/Conversation 不共享近期 Message 或 Summary；
3. 群聊按配置选择 shared conversation 或 per-user partition，但发送者 Principal 永远独立；
4. Lane 排序使用持久 priority + receive_sequence；
5. 控制消息优先，普通 Turn 保序，后台任务不长期占 Lane；
6. 增加 aging，避免主动/后台工作永久饥饿；
7. ContextSnapshot 增加：
   - message_upper_bound；
   - query_plan_version；
   - selection_policy_version；
   - 每条 item 的 source/score/tokens/trust_label/retrieval_path；
   - excluded_summary；
8. 重试默认复用 Snapshot；等待时间过长或 Session 版本变化时创建带 parent_id 的新 Snapshot。

### M6：双路检索与可解释降级

#### 工作项

1. Query Plan 明确原始 query、改写 query、Scope、过滤、预算和版本；
2. 关键词 FTS/BM25 与向量召回独立执行；
3. 合并、去重、多样性和 Token 分配使用确定算法；
4. 相同数据版本和输入返回稳定选择；
5. 每条结果保留 keyword/vector/keyword+vector 路径和分数组成；
6. 降级链：
   - Embedding 不可用 → FTS + recency；
   - FTS 损坏 → vector；
   - Summary 不可用 → 受限近期消息；
   - Query Rewriter 超时 → 原 query；
7. recall_memory 与隐式检索复用同一 Scope/过滤管道；
8. 普通曝光只更新 exposure/retrieval_count，不自动 reinforcement。

#### 离线评测集

- 事实查找；
- 时间冲突；
- 跨 Session 隔离；
- 偏好；
- Goal；
- 历史摘要；
- 同义词无词面重叠；
- 跨语言查询。

### M7：Memory 生命周期闭环

#### 工作项

1. Candidate 来源、Scope、source_trust、canonical_key 和版本完整持久化；
2. auto-confirm 仅限 Owner 明确要求、确定性任务状态和受信结构化 Connector；
3. supports/contradicts/supersedes/refines/derived_from 关系可追溯；
4. 实现文档中的 retrieval_weight 公式和分 kind 衰减；
5. Consolidation 幂等执行：去重、重算、归档、遗忘候选、视图刷新；
6. Markdown 视图由数据库重建：MEMORY.md、PENDING.md、HISTORY.md、SELF.md；
7. 文件写入使用临时文件 + 原子替换；数据库始终权威；
8. 用户删除先退出检索，再异步清理 Embedding/Payload/视图，保留 tombstone；
9. Embedding 模型切换使用新版本空间，不混合不可比向量；
10. 索引可从 Canonical Memory 全量重建。

#### 数值测试

- fact/preference/episode/goal/constraint 衰减边界；
- reinforcement 上限和周期衰减；
- superseded 条目退出默认检索；
- retrieval_weight 0.1/0.05 阈值；
- 删除后重建索引不复活。

### M8：Model Adapter 契约和路由

#### 工作项

1. ModelRequest/Response/StreamEvent 对齐 `MODEL-ADAPTER / 2～4`；
2. StreamEvent 使用单调 sequence_no，Adapter 负责组装最终 Response；
3. Provider 能力快照包括 Context、输出限制、模态、Tool、并行 Tool、JSON Schema 和 Prompt Cache；
4. Tool 名称映射可逆，不支持的 JSON Schema 关键字调用前失败；
5. 统一错误映射和 retryable/retry_after；
6. Fallback 前重新检查上下文、模态、Tool、Schema 和预算；
7. 路由结果记录 provider/model/policy version 和降级原因；
8. 原始 Provider 响应和错误只进入受限 Payload，默认日志脱敏。

#### 离线契约 Fixture

- 文本；
- JSON Schema；
- 单/多 Tool；
- 流事件顺序；
- 取消；
- 超时；
- 限流；
- Context Overflow；
- Usage；
- 错误脱敏。

## 6. 关键 Schema/配置变更

可能新增或扩展：

- turn_checkpoints：step、snapshot、budget、approval、capability/config version；
- context_snapshots 及 items；
- memory relation/embedding/version/tombstone 字段；
- model_calls：router policy、degradation、usage 细分；
- runtime budget、retry、snapshot refresh 和检索权重配置。

所有变更执行 expand → backfill → switch → contract，并回查 `DATABASE-SCHEMA` 和 `CONFIG-PROFILES`。

## 7. 观测要求

每个 Turn 至少记录：

- trace/turn/attempt/session/partition；
- Lane 等待时长；
- Context 构建耗时和 Token；
- 检索路径与降级；
- Provider/Model/Usage/延迟；
- Tool/Approval/Receipt；
- Checkpoint 与恢复决策；
- Budget 消耗和终止原因。

不得在默认 Trace 中保存 Secret、完整外部原文或未脱敏 Tool 输出。

## 8. 完成定义

1. Turn/Attempt/等待/恢复路径有端到端测试；
2. 所有恢复点不会重复已确认副作用；
3. `AGENT-LOOP / 11` 全部通过；
4. `RETRIEVAL-CONTEXT / 15` 全部通过；
5. Memory 重建、衰减、删除和 Markdown 视图测试通过；
6. 每个 Provider 通过 `MODEL-ADAPTER / 11` 契约矩阵；
7. 预算跨重启有效；
8. 测试、Ruff、compileall 全绿，并有固定离线评测报告。
