# 05 接入、交互、身份、消息与投递下一步开发计划

> 状态：draft  
> 对应目录：`markdown/05_interaction`  
> 设计依据：`ACCESS-DELIVERY`、`APPROVAL-COMMANDS`、`STREAMING-DELIVERY`、`MESSAGE-PERSISTENCE`、`LANGBOT-BRIDGE`

## 1. 目标

在现有 Web Dashboard、Web Channel、QQ OneBot、Command/Query API 和流式回复基础上，完成 Gateway/Core 版本化边界、身份隔离、Command 幂等、Message Revision、Delivery Receipt/Reconcile 和断线恢复。

最终要求：

1. Agent Core 永远不接收平台 SDK 对象；
2. Endpoint/Principal/Conversation/Session 的映射稳定且不依赖显示名称；
3. 所有写操作通过 Command Envelope、expected_version、幂等和 Audit；
4. Message 是不可变内容事实，Delivery 是发送生命周期；
5. 流式 placeholder/edit/finalize 都可关联 Delivery/Attempt/Receipt；
6. Gateway 重启只恢复平台连接和投递，不擅自重置 Session/Turn；
7. Web、QQ 和 LangBot Bridge 使用相同 Core 契约与恢复语义。

## 2. 当前基线

### 已有

- ChannelAdapter Protocol、ChannelManager 和 Gateway；
- WebChannelAdapter、QQOneBotAdapter；
- 多个 legacy LangBot Adapter/Client；
- ChannelEnvelope 和 OneBot 转换；
- Web Query/Command API、WebSocket Chat；
- Message/ContentPart/Conversation/Session 持久化；
- Inbound 幂等、reply route 基础；
- Delivery Worker、Receipt、流式 Controller 和孤儿流恢复；
- Session 软删除、历史查询和 Dashboard 页面。

### 缺口

- README 仍称 LangBot Bridge 待实现，代码和文档状态不一致；
- Bridge 尚未形成独立版本化 HTTP/WebSocket DTO 完整闭环；
- 17 个 legacy Adapter 未逐个验证，且仍处于 lint 隔离；
- Command API 尚未统一要求调用方幂等键、expected_version 和 Principal 权限；
- Approval 展示/消费的完整字段和恢复事务不足；
- Message 编辑、撤回、附件 Payload 和 Summary 版本测试不完整；
- PII prompt redaction、Endpoint 绑定和受限外部 Principal 需要系统化；
- 实时事件流断线后的 cursor/re-sync 语义不足；
- 外部 Channel 真实环境验收尚未完成。

## 3. 目标边界

```text
Platform/LangBot SDK
  ↕  Gateway-owned objects
LangBot Bridge Adapter
  ↕  versioned Channel/Delivery DTO
Agent Core
  ├─ Identity/Conversation/Session
  ├─ Message/Turn
  ├─ Command/Approval
  └─ Delivery/Receipt
Interaction Web
  ↕  Query/Command/Realtime APIs only
```

禁止 Gateway 直接写 Core SQLite；禁止 Core 猜测平台身份结构和新 Reply Route。

## 4. 里程碑

### M1：Channel Contract 与能力快照

#### 工作项

1. 统一 ChannelDriver start/stop/send/edit/delete/capabilities；
2. 所有 Driver 返回结构化 ChannelSendResult/Error，不上抛 SDK 对象；
3. CapabilitySnapshot 包含 edit、stream、buttons、files、threads、delete、长度、文件大小、reply token TTL、rate limit；
4. Delivery 创建时固定 CapabilitySnapshot 和 TargetSnapshot；
5. 内容统一映射 ContentPart，平台不支持部分由 Renderer 降级；
6. Renderer 实现 Markdown→text、按钮→列表、语义分片、代码块保护、超大内容→文件；
7. Adapter 状态和最后错误通过 Query API 暴露安全摘要；
8. legacy Adapter 分批完成 import/start/stop/capability 合同验证，未通过者默认 disabled。

#### 验收

- Driver 不导入 Core Repository；
- Core 不导入平台 SDK；
- Capability 变化不静默改变已批准目标；
- unsupported/too_large/rate_limit/auth/temporary/permanent 映射一致。

### M2：LangBot Bridge 独立契约

#### 入站 DTO

对齐 `LANGBOT-BRIDGE / 2`：schema version、event/message ID、Channel/Instance、Conversation/Thread、Sender/Endpoint Ref、Content Parts、Reply Route、Trust、Capability、Raw Payload ref、Trace。

#### Delivery DTO

支持 send、start_placeholder、append_or_replace、finish、delete、reconcile；每次携带 delivery_id、attempt_id、operation_seq、idempotency key、target snapshot 和 content。

#### 工作项

1. Bridge 默认通过 loopback HTTP/WebSocket 或本地 Socket；
2. 当前和前一 Schema 版本并存；
3. Endpoint Ref 作为不透明稳定字符串，Core 不解析内部格式；
4. Reply Route 是快照，Token 过期返回 route_expired；
5. Bridge 重试复用 platform event ID；无稳定 ID 标 best_effort；
6. 入站附件只传受限引用或 Payload Manifest，不传永久 Secret URL；
7. 出站附件不接收宿主任意路径；
8. 健康接口报告每个 Instance 的连接、认证、限流和最后事件时间；
9. GatewayStarted 只触发恢复扫描，不修改 Session/Turn。

#### Fixture

- 私聊、群聊、多成员、Thread；
- 引用、编辑、撤回、附件；
- 重复 Event；
- Token 过期、限流、断连；
- 流式恢复；
- 当前/前一版本。

### M3：Identity、Endpoint 与 Session 隔离

#### 工作项

1. Identity Resolver 输入使用稳定 platform/ref 字段；
2. 禁止显示名称、昵称、头像作为身份键；
3. Endpoint 绑定支持配置预绑定、验证码、Dashboard 确认、平台白名单；
4. 未绑定用户创建受限 external Principal，不共享 Owner Memory/Tool 权限；
5. DM 始终 per-user；Group 默认 per-user；Thread 默认 shared，可配置；
6. 共享 Session 时 Message 仍保存结构化 sender Principal/Endpoint；
7. context_partition_key 包含 Channel Instance、Conversation、Thread、多用户策略和 reset generation；
8. 只有 `/new`、`/reset`、配置 reset policy 或显式 ResetSession Command 创建新 generation；
9. Gateway/进程崩溃本身不重置 Session；
10. Endpoint 解绑保留历史来源证据。

#### PII Redaction

1. 仅在 Context/Prompt 渲染层脱敏；
2. user_id → 稳定 hash 别名；chat_id 保留 platform 前缀；
3. 原始 ID 仅供路由、Session Key 和本地 Renderer mention；
4. 数据库存储不被 prompt 别名覆盖；
5. Trace/日志按安全策略脱敏。

#### 测试

- 同名用户不合并；
- 群聊 Owner/外部用户权限隔离；
- Group/Thread 策略切换只影响新 generation；
- 跨 Channel 不共享短期历史；
- PII 别名稳定且不可从默认 Trace 反查。

### M4：Command Envelope 和本地控制面

#### 工作项

1. 所有写 API 接受：command_id、command_type、actor、idempotency_key、target、expected_version、payload、expires、origin、trace；
2. 唯一约束 `(actor,command_type,idempotency_key)`；
3. 重复同 payload 返回首次结果；相同键不同 payload 返回 idempotency_conflict；
4. 聚合变更必须 expected_version；Ensure 类命令也用条件更新；
5. 统一 CancelTurn、RetryTurn、Approve/Reject、RetryTask、ReplayDelivery、Confirm/DeleteMemory；
6. ReplayDelivery 创建新 Delivery/重放代次，不把 sent 改回 sending；
7. 条件更新实现冲突优先级：终态 > 取消/过期 > 审批 > 重试；
8. 所有 Command 写 Audit 和 Event Outbox；
9. 错误统一 validation/permission/expired/idempotency/version/terminal/not_found/internal；
10. loopback 默认、远程 bind 显式拒绝；写请求校验 Origin；
11. 远程访问在认证/TLS/防重放落地前不可启用。

### M5：Approval 完整生命周期

#### 工作项

1. Approval 保存 subject、requester attempt、action summary、permissions、risk、arguments snapshot、action hash、policy version、allowed responders、expires/version；
2. UI 展示具体动作、目标、脱敏参数、权限、风险、影响、可逆性和过期时间；
3. 消费事务中验证 pending、未过期、Principal、action hash、Policy 和 version；
4. 同事务更新 Approval、目标 Turn/Task、Audit、Event Outbox；
5. 参数、目标或权限变化使旧 Approval 失效并创建新对象；
6. 已消费 Approval 不能重复恢复；
7. waiting Turn/Task 使用 subject 恢复，新建 Attempt；
8. rejected/expired/cancelled 有明确用户结果。

### M6：Message、Revision 与历史

#### 工作项

1. Message 创建后不可原地覆盖内容；
2. ContentPart 保存 revision_no、ordinal、类型、inline/payload、hash、metadata 和 trust；
3. 文本超过阈值、二进制和附件写 Payload Store；
4. 入站事务：Inbox → Identity/Session → receive_sequence → Message/Parts → Turn → Outbox；
5. 重复入站返回已有 message_id/turn_id；
6. 平台编辑创建 Revision；按 platform_revision 或稳定 fallback tuple 确定顺序；
7. 迟到旧编辑保留历史但不回退 current revision；
8. 撤回保留事件证据，可见性和内容清理由 Retention 决定；
9. 历史按 receive_sequence/message_id 游标分页；
10. Summary 保存覆盖范围、源 Revision 和算法/模型版本；
11. Conversation 删除先隐藏，再异步清理 Payload/FTS/Embedding/Summary；
12. Memory 来源转 tombstone，除非命令明确删除长期记忆。

### M7：Delivery 与非流式对账

#### 工作项

1. 被动回复优先使用原入站 reply_route 快照；
2. 主动发送 ready-to-send 时执行 DeliveryPolicy 并固定 TargetSnapshot；
3. Delivery 创建后不静默换目标；fallback 创建新 Delivery 或明确失败；
4. 每次 Attempt 保存 request hash、Channel Instance、platform ID、error、retry-after；
5. 平台调用前持久化发送意图；
6. timeout/unknown_result 进入 unknown；
7. Gateway 提供 reconcile，按 operation id/idempotency/platform query 对账；
8. 未确认前不自动重复发送；
9. 限流遵循 retry-after；永久错误不重试；
10. Delivery 失败不重新执行 Agent。

### M8：流式 Delivery 收敛

#### 工作项

1. 保持 Delta 为临时展示，不逐 Token 写 Message；
2. 首次平台副作用前持久化 provisional streaming Delivery；
3. operation_seq 和 idempotency key 单调稳定；
4. placeholder、edit、final edit 写 Receipt；
5. finalize 短事务创建不可变 Assistant Message、绑定 Delivery、写 TurnCompleted 并提交 Turn；
6. 取消/异常 withdraw 并标 interrupted；
7. 恢复扫描只撤回真正孤儿，不干扰活跃 Attempt；
8. Web 重连清理遗留占位符保持幂等；
9. 不支持 edit 的 Channel 在首次副作用前确定降级策略；
10. 当前阶段预留 finalizing/unknown，但不得伪装已实现。

#### 故障测试

- placeholder 后崩溃；
- edit 后崩溃；
- final platform edit 成功、本地事务失败；
- Turn Lease 丢失；
- WebSocket 断线重连；
- 降级 Channel 重复发送防护。

### M9：Query、Realtime 与 Dashboard

#### 工作项

1. Query API 只使用只读 Query Service，不暴露表结构；
2. Realtime 事件覆盖 turn/run/task/delivery/approval/channel/connector；
3. 每个事件有 sequence/cursor/schema/trace；
4. Realtime 是观察通道，断线后用 cursor + Query API 重同步；
5. Dashboard 写操作全部调用 Command API；
6. Approval UI 展示完整风险信息；
7. Delivery 页面提供 retry/reconcile/replay 区分；
8. Channel 页面展示 capability、health、auth/rate limit；
9. 前端安装依赖后执行 typecheck/build 和浏览器关键流程测试；
10. Mock 数据与真实 DTO 共用 Schema，防止漂移。

## 6. 上线策略

1. 先让 Web/QQ 走统一 DTO，但保留兼容 Adapter；
2. LangBot Bridge 先 shadow 记录，不发送；
3. 对录制 Fixture 回放并比较 Envelope/Delivery；
4. 单个 Channel Instance 灰度启用；
5. 观察重复入站、unknown Delivery、route expired 和占位残留；
6. 真实 QQ/LangBot 完成人工发送、编辑、撤回、重连验收后再标 stable。

## 7. 完成定义

1. Gateway/Core 仅交换版本化 DTO；
2. Command/Approval 的幂等、版本和 Audit 完整；
3. 身份和 Session 隔离通过群聊/Thread 测试；
4. Message Revision、删除和 Payload 生命周期闭环；
5. 非流式 unknown 可 reconcile；流式崩溃可恢复；
6. `MESSAGE-PERSISTENCE / 12` 和 `LANGBOT-BRIDGE / 11` 全覆盖；
7. Web typecheck/build、Python 测试、Ruff、compileall 全绿；
8. README 与实际 Bridge/Channel 状态一致。
