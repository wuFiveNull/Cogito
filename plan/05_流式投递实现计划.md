# Plan 05：流式投递（placeholder / edit / finish）实现计划

> 依据设计文档 `STREAMING-DELIVERY`(markdown/05_interaction/02_流式投递.md) 与 `ACCESS-DELIVERY`(00_接入交互身份与投递.md) 编写。
> 目标：让 Agent 回复以"先占位 → 增量编辑 → 最终定稿"的方式投递到支持编辑的渠道（Web 优先），失败或崩溃可恢复，且 Core 主链路不被污染。

## 0. 现状审计（代码事实，非文档描述）

通过通读代码确认，当前**只有流式脚手架，没有流式实现**：

| 维度 | 现状 | 缺口 |

| `channel/base.py` | `ChannelCapabilities` 已声明 `supports_streaming`/`supports_edit`/`supports_delete` | `ChannelAdapter` 协议**只有 `send`**，无 `edit`/`delete` 方法；无 `ChannelEditRequest`/`ChannelDeleteRequest` DTO |
| `service/channel_gateway.py` | 只 `send`/`send_request` | 无 `edit`/`delete` 路由 |
| `service/delivery_worker.py` | 非流式：`lease_next`/`deliver`/`reconcile` | 无 `start_placeholder`/`append_delta`/`replace_content`/`finish` 操作 |
| `model/openai_compat.py` | `stream()` 方法签名存在但 `raise NotImplementedError`；`capabilities().supports_streaming=False`；`generate()` 直接拒绝 `stream=True` | 需实现 SSE 解析；置 `supports_streaming=True` |
| `model/router.py` | 仅 `generate` | 无 `generate_stream` |
| `runtime/loop.py` | `run()` 一次性返回 `LoopResult`；`_build_request` 写死 `stream=False` | 无 `run_stream` 增量生成器 |
| `runtime/orchestrator.py` | `run_one()` 调 `loop.run` → `completion._complete`（一步落库） | 无流式分支 |
| `service/completion.py` | 一次性写 Message+Delivery(pending) | 无流式入口；但已读取 `capability_snapshot_json`（当前未填充） |
| `channel/drivers/web.py` | `send_request_sync` 已实现；`supports_streaming/ edit=False` | 无 `edit_request_sync` |
| `service/recovery_service.py` | `recover_delivery_leases` 仅处理 `sending`→`unknown` | 无 `streaming`/`finalizing` 恢复分支 |
| `interaction_web/chat.py` | WS 只推完整 `assistant` 消息 | 无 `assistant.delta` 增量事件 |

**结论**：流式投递是"从 0 补齐操作层 + 接入编排层"的工作。Core 主链路（Inbound→AgentLoop→Completion→Delivery→Gateway→Adapter）**不变**，只在 Delivery 与 Adapter 之间新增"流式操作层"。

## 1. 设计原则（引自 STREAMING-DELIVERY）

1. placeholder / append / replace / finish 都属于 **Delivery Attempt**，不允许 Stream Session 直接调平台。
2. Delta 是临时展示数据，**不是**最终 Assistant Message。
3. 不逐 Token 写 SQLite，但每个平台副作用必须关联 Delivery / Attempt / Receipt。
4. 最终 Assistant Message 是内容事实源；Delivery 是发送状态事实源。
5. 降级模式在**首次副作用前**确定并写入 Delivery，运行中不静默切换为可能重复发送的模式。

---

## 2. 目标架构与数据流

```
AgentLoop.run_stream()  ──delta──▶  StreamingDeliveryController
                                       │  1. 建 provisional Delivery (content_mode=provisional)
                                       │  2. 首 token → start_placeholder → adapter.send → platform_message_id
                                       │  3. 后续 delta（节流合并）→ append_delta/replace_content
                                       │       → ChannelGateway.edit → adapter.edit_request_sync
                                       │       →（Web）推入 per-conversation 队列
                                       │  4. 结束 → 写不可变 Assistant Message（bind final_message_id）
                                       │       → finish（最终 replace_content 或 final_only 新发）
                                       │       → content_mode=final, status=sent
                                       ▼
                          WebChannelAdapter 队列 ──WS──▶ 浏览器 (assistant.delta / assistant.final)
```

**关键不变量**：
- 流式操作全部经由 `ChannelGateway`（与现有 `send` 对称），不直接调 adapter。
- 每个操作 = DeliveryAttempt 内的一个 `operation_seq`，写 `delivery_receipts`（已支持）。
- 最终 Message 在 `finish` 短事务内写入并绑定 `final_message_id`；在此之前用户看到的都是可编辑的占位消息。
- Web 渠道无需真实平台，edit = 推送增量到内存队列，天然支持实时 diff，因此 **Web 作为第一个落地渠道**。

### M2 — Provider / Router / Loop 流式生成

**`src/cogito/model/openai_compat.py`**
- 实现 `stream(request)`：`payload["stream"]=True`，用 `httpx` 流式读 `response.aiter_lines()`，解析 SSE `data:` 帧（遇 `[DONE]` 结束），将 `choices[0].delta.content` 累积成 `ModelResponse`（每帧 `content_parts=(ContentPart(text=delta),)`，`finish_reason=stop` 仅末帧）。复用现有错误映射。
- `capabilities().supports_streaming = True`（或在 `Config` 增加 `provider.streaming` 开关；首版直接置 True，由 `stream=False` 旧路径仍可用）。
- `generate()` 保留对 `stream=True` 的拒绝（仅 `stream()` 支持流式）。

**`src/cogito/model/router.py`**
- 新增 `async def generate_stream(self, request, model_role="main") -> AsyncIterator[ModelResponse]`：解析 provider 链，对主 provider 调 `provider.stream(request)`；首帧前不 fallback（避免中途换模型），主 provider 抛可重试错误时整体失败（由上层 controller 降级为 final_only）。仍走 `on_call_completed` 记录（末次）。
- `ModelRequest.stream` 字段已存在（`loop._build_request` 当前写死 `False`）。

**`src/cogito/runtime/loop.py`**
- 新增 `async def run_stream(self, context, model_role="main", cancel_flag=None) -> AsyncIterator[tuple[str, bool]]`：
  - 复刻 `run()` 的迭代/终止/工具调用逻辑；
  - 当某次迭代 `_classify_response == "_final"` 时，改用 `_router.generate_stream` 并 **yield 每个 delta** `(delta_text, False)`，末帧后 yield `("", True)` 表示本段结束；
  - 工具调用等中间迭代不 yield 文本（可 yield `("", False)` 占位或 emitting 一个 `status` 事件，首版只 yield 最终段的文本）；
  - `cancel_flag()` 在每次 yield 前检查，触发则停止生成。
  - `_build_request` 增加 `stream=True` 参数（仅 final 段用流式；工具迭代仍 `stream=False`）。

### M3 — 流式投递核心（操作层）

**`src/cogito/store/repositories.py`**（新增 `DeliveryRepository`，供 Worker 与 Controller 复用）
- `create_streaming_delivery(target, content_ref, policy, degradation_mode, idempotency_key) -> delivery_id`（`status='pending'`, `content_mode='provisional'`）
- `mark_streaming(delivery_id, platform_message_id, attempt_id)`（`status='streaming'`, `stream_status='placeholder_created'`）
- `record_edit(delivery_id, attempt_id, operation_seq, text, platform_message_id, receipt_kind)`：写 `delivery_receipts`（`operation_seq` 递增）+ 更新 `last_confirmed_revision` + `stream_status='streaming'`
- `finish_streaming(delivery_id, final_message_id, platform_message_id, text)`：短事务写最终 Message 引用 + `content_mode='final'` + `status='sent'` + `stream_status='done'`
- `withdraw(delivery_id, reason)`：`status='interrupted'`（或 `cancelled`）

**`src/cogito/service/streaming_delivery.py`**（新增 `StreamingDeliveryController`）
```python
class StreamingDeliveryController:
    def __init__(self, conn, gateway: ChannelGateway, capabilities: ChannelCapabilities,
                 clock=None, policy: StreamPolicy | None = None): ...
    async def run_streaming_turn(self, *, turn, attempt, context, input_meta,
                                 on_delta, cancel_flag) -> str | None:
        # 1. 选降级模式：capabilities.supports_edit → edit_placeholder；否则 final_only
        # 2. create_streaming_delivery
        # 3. 迭代 loop.run_stream：
        #    - 首 token：start_placeholder（gateway.send 占位空/“正在思考…”，取 platform_message_id）
        #    - 累计文本，按 policy.throttle_ms 节流 → append_delta/replace_content
        #        （gateway.edit(ChannelEditRequest, 全量文本, operation_seq+=1)）
        #    - on_delta(conversation_id, text, op_seq) 推实时（Web 走 adapter 队列）
        # 4. 末帧：写不可变 Assistant Message；finish（edit 最终文本 或 final_only 新发）
        # 5. 异常/取消：withdraw 或 edit 为“已中断”
```
- `on_delta(conversation_id, text, op_seq, is_final)` 由 `RuntimeApplication` 注入（默认 no-op；Web 下指向 `web_channel_adapter.push_edit(...)`）。

**`src/cogito/service/channel_gateway.py`**
- 新增 `edit(self, target_snapshot, content_ref, platform_message_id, text, operation_seq) -> ChannelSendResult`：解析 `target_snapshot` 取 `adapter_id`/`conversation_id`，构造 `ChannelEditRequest`，调 `_call_adapter_sync(adapter, request)`（复用现有回主 loop 机制）。
- 新增 `delete(self, target_snapshot, platform_message_id, reason) -> None`：调 `adapter.delete_request_sync`。
- `send` 路径保持不变（非流式 Delivery 仍走 `DeliveryWorker`）。

**`src/cogito/channel/drivers/web.py`**
- 实现 `edit_request_sync(req: ChannelEditRequest) -> ChannelSendResult`：
  - 首次 edit（无已知 `platform_message_id`）先生成合成 id `web-msg-<delivery_id>`，推 `{kind:"placeholder", platform_message_id, text}` 入 per-conversation 队列；
  - 后续 edit 推 `{kind:"edit", platform_message_id, text, operation_seq}`；
  - 返回 `ChannelSendResult(status="sent", platform_message_id=id)`。
- `capabilities()` 置 `supports_streaming=True, supports_edit=True`（Web 实时可自由 diff）。

## 4. 降级策略（STREAMING-DELIVERY / 6）

| 渠道能力 | 模式 | 行为 |
|---|---|---|
| 原生流式 + 可编辑 | `native_stream` | 按平台流协议更新（首版 Web 归此类） |
| 可编辑不可流式 | `edit_placeholder` | 占位 + 批量 replace（Web 默认） |
| 仅最终发送 | `final_only` | 不建占位，走现有 `DeliveryWorker` 一次性发 |
| 不支持编辑 | `processing_then_final` | “正在处理”提示 + 最终独立 Delivery |

模式在 controller 首 token 前依据 `capability_snapshot` 写定，**运行中不静默切换**。首版只完整实现 `edit_placeholder`（Web）与 `final_only`（兜底）；`native_stream`/`processing_then_final` 留接口。

## 5. 风险与边界

- **并发**：同一 Delivery 的流式操作由持有 Lease 的 controller 串行执行；Web 的 `asyncio.Queue` 天然单消费者。
- **重复发送**：`operation_seq` + 幂等键保证 edit 可重放；平台不支持幂等编辑时超时进 `unknown`，由 recovery 对账，不盲目补发。
- **工具调用中断流**：含 tool_call 的 Turn，流式只覆盖"最终回复段"，中间工具步骤不流式（emit 状态事件可选，首版省略）。
- **跨线程**：controller 在 worker 协程内运行（与 Plan 04 单 loop 架构一致），`gateway.edit` 走 `send_request_sync` 同步路径，无需跨线程胶水。
- **向后兼容**：非流式渠道与现有 `DeliveryWorker` 完全不受影响；新字段均有默认值，迁移幂等。

## 6. 验收标准

1. `WebChannel` 对话时：浏览器先收到占位（`assistant.delta` 初始），随后增量更新，最终收到 `assistant.final`；DB `deliveries.content_mode='final'`、`final_message_id` 非空、`delivery_receipts` 有连续 `operation_seq`。
2. `cancel-turn` 命令在流式进行中触发 → 占位被 withdraw 或 edit 为"已中断"，不残留半成品。
3. Provider `stream()` 对真实 OpenAI-compatible 端点 yield 正确 delta（单测覆盖）。
4. `recovery_service.recover_streaming_deliveries` 对 Lease 失效的 `streaming` Delivery 不重复创建平台消息，按策略收尾。
5. `ruff check` + `pytest` 全绿；前端 `tsc -b` 零错误。
