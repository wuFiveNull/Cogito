# 长期记忆补全与验收计划

## 1. 文档目的

本文是基于 2026-07-07 当前代码审计结果制定的实施计划，用于把已有长期记忆骨架补全为真正可运行、可恢复、可验证的个人 Agent 长期记忆系统。

本文不是从零设计。当前仓库已经存在 Memory Domain、SQLite 表、Repository、MemoryService、记忆工具、Context 注入、FTS、Task/Watermark/Summary 表和生命周期骨架。本计划只处理以下问题：

```text
已有代码没有真实接线
已有代码仍是 stub/Noop
实现与权威设计不一致
测试只覆盖组件，没有覆盖真实运行闭环
运行配置导致功能默认失效
```

最终目标：

```text
用户自然语言表达稳定事实/偏好/约束/目标
→ Agent 正确写入长期记忆
→ SQLite 独立事务持久化
→ 进程重启后仍存在
→ 新 Session 自动召回
→ 用户更新时形成可追踪覆盖关系
→ 用户删除后所有派生索引同步失效
→ 后台提取、摘要和维护任务可跨重启恢复
```

## 2. 权威文档与约束

实施时按以下文档执行：

- `DOMAIN-CONTRACTS / 1.3.5 Memory Aggregate`
- `DOMAIN-CONTRACTS / 1.13 MemoryItem`
- `DOMAIN-CONTRACTS / 3.2 MemoryService`
- `RUNTIME-FLOWS / 即时 Turn 与后台 Task`
- `MEMORY-LIFECYCLE / 1. 双层存储架构`
- `MEMORY-LIFECYCLE / 6-10 候选、写入、确认、冲突、Consolidation`
- `MEMORY-LIFECYCLE / 12-14 检索、Embedding、纠正与删除`
- `RETRIEVAL-CONTEXT / 1-12 检索与 Context Snapshot`
- `SESSION-CONTEXT / 短期消息边界和跨 Session 共享规则`
- `DATABASE-SCHEMA / 1-10 SQLite、Schema、Migration`
- `TASK-SCHEDULER / Task、Attempt、Lease、恢复`
- `EVENT-OUTBOX / 幂等事件和消费`
- `TEST-EVALUATION / 测试、回放和质量评估`

个人 Agent 简化原则：

- 不实现多租户 SaaS。
- 暂不实现 KMS、字段级加密和复杂审批。
- 默认只有一个 Owner，但仍保留 `principal_id`，用于跨 Channel 身份归一和群聊隔离。
- 不使用正则或关键词表判断用户语义；确定性代码只负责字段校验、Scope、状态、幂等、时间和删除。
- 第一目标是可运行，其次才是向量检索和复杂生命周期。

## 3. 当前代码基线

### 3.1 已完成并可复用

- Context 已保留消息 role，并按历史正序、当前输入最后装配。
- `message_upper_bound` 已使用真实 `receive_sequence`。
- 多 ContentPart 消息可以合并读取。
- `memory_items` 已包含 Principal、Scope、canonical key、显式性、重要度、版本和软删除字段。
- `MemoryRepository` 已具备基础 CRUD、FTS 和简单评分。
- `SqliteMemoryService` 已具备 propose、remember、retrieve、confirm、reject、forget。
- `remember_memory`、`recall_memory`、`forget_memory` 已注册到 memory/core toolset。
- ContextBuilder 已能注入 confirmed Memory。
- 已存在 Task Repository、Dispatcher、Worker、Watermark 和 Summary 表骨架。
- 已存在 FTS5、Embedding Protocol、Markdown 视图和 Consolidation 骨架。
- 当前测试基线为 `557 passed`，源码可编译。

### 3.2 关键缺口

1. 默认 Router 只有 `main`，自动提取使用的 `memory_extractor` 角色无法路由。
2. 自动提取异常被静默吞掉，生产运行中看起来成功、实际没有写入。
3. 当前 `config.toml` 的自定义 System Prompt 覆盖 Memory Rules。
4. `AgentConfig._from_raw()` 未提供 System Prompt 时也会丢失 dataclass 中的 Memory Rules 默认值。
5. 记忆工具直接操作共享连接，没有独立、明确的事务提交边界。
6. `remember_memory` 的来源记录为 Turn，而权威设计要求保留源 Message/Event。
7. 按 `memory_id` 删除时没有验证当前 Principal 所有权。
8. 新记忆覆盖旧记忆时只设置旧项 `valid_to`，没有建立完整关系链。
9. `memory.extract` 和 `summary.generate` Task Handler 仍是 stub。
10. TaskWorker 没有接入 `cogito run/worker` 的生产循环。
11. Watermark 主键没有完整包含 Session，无法正确表示同一 Conversation 下多个 Session 的处理进度。
12. Context 压缩只能读取摘要，没有真实摘要生成流程。
13. Query Plan 只定义未调用，并使用关键词表和正则。
14. Embedding 写入是 `pass`，默认 Provider 是 Noop。
15. 被动召回会直接提高 `retrieval_weight`，不符合设计中的 reinforcement 规则。
16. Consolidation 会自动软删除低分记忆，与“低分只进入遗忘候选”冲突。
17. Markdown 视图使用进程 CWD 下的固定 `.workspace`，未使用配置的 workspace path。
18. Ruff 全仓检查失败，包含 `__main__.py` 未定义名称、重复导入等实际运行风险。

## 8. 里程碑 D：自动候选提取质量 ✅

优先级：P1。D1-D5 已完成（2026-07-07）。

### D1. 结构化输出

Memory Extractor 使用 JSON Schema 或 Provider 的结构化输出能力，不通过截取第一个 `{}` 修复任意文本。

输出：

```text
kind
subject
predicate
value
scope_type
scope_id
explicitness
confidence
importance
source_sequences
reason
```

### D2. 确认策略

自动 confirmed 仅限：

- Owner 明确要求记住；
- Owner 明确陈述稳定偏好/约束/目标；
- 系统确定性任务状态。

以下只能 candidate：

- 模型推断；
- 模糊偏好；
- 外部网页或 Tool 输出；
- 无法确定主体的陈述。

### D3. Canonical Key

Canonical Key：

```text
principal_id
+ scope_type/scope_id
+ kind
+ normalized subject
+ normalized predicate
```

value 不属于 canonical key；value 用于判断同值强化或新值覆盖。

确定性规范化只做：

- Unicode normalization；
- trim；
- 大小写归一；
- 连续空白折叠。

不使用业务关键词正则推断语义。

### D4. 冲突处理

```text
同 canonical key + 同值
→ 追加 source/supports 关系，不创建重复 Memory

同 canonical key + 新的明确值
→ 新建 Memory + supersedes

同 canonical key + 两个低置信推断冲突
→ 两者保留为 candidate + contradicts

用户明确纠正
→ 新 confirmed Memory + supersedes/corrects
```

### D5. 提取质量测试

- 引用别人的偏好不能写成 Owner 偏好。
- 用户一句话先表达后否定，不能保留被否定值。
- 一次性任务不进入长期记忆。
- Tool 输出不能自动变成 Owner 事实。
- 相同输入重复消费不产生重复项。

## 11. 里程碑 G：生命周期与派生视图对齐 ✅

优先级：P2。G1-G5 已完成（2026-07-07）。

### G1. 补齐生命周期字段

新增 Migration：

```text

```

增加：

```text

decay_rate
retrieval_weight
last_weight_update

```

### G2. Reinforcement 规则

被动召回：

```text
retrieval_count += 1
exposure_count += 1
last_retrieved_at = now
reinforcement 不变
```

明确正向信号：

```text
用户主动确认有用  → reinforcement += 2
成功 Task 明确依赖 → reinforcement += 1
用户纠正/否定      → 不增加，可降低 emotional_weight
```

禁止当前“每次被动召回直接增加 retrieval_weight”的实现。`retrieval_weight` 由维护任务按版本化公式重算。

### G3. 遗忘和归档

严格区分：

```text
移出热记忆
归档
失效
覆盖
用户永久删除
```

低分记忆：

- `< 0.1`：归档，不物理删除；
- `< 0.05`：进入遗忘候选；
- 只有用户明确删除或保留策略到期才清理内容。

Consolidation 不得自动软删除低分记忆。

### G4. Markdown 派生视图

统一使用 `config.workspace_path`：

```text
{workspace}/memory/MEMORY.md
{workspace}/memory/SELF.md
{workspace}/memory/PENDING.md
{workspace}/memory/HISTORY.md
{workspace}/memory/RECENT_CONTEXT.md
```

要求：

- 数据库是唯一事实源；
- Markdown 丢失可重建；
- MEMORY.md 有容量上限并按 memory_id 去重；
- HISTORY.md 追加时幂等；
- PENDING.md 最多展示最近 200 条；
- 不查询 Schema 不允许的 `superseded` status；通过 relation/valid_to 判断历史覆盖。

### G5. 删除传播

用户删除后处理：

```text
memory_items
memory_relations
memory_fts
memory_embeddings
Memory Context cache
Markdown views
当前 Session revoked override
```

删除后不得因 FTS/Embedding 重建重新出现。

## 15. 最终 Definition of Done

只有同时满足以下条件，才能宣布长期记忆设计完成：

### 核心闭环

- 自然语言记住、修改、忘记都可运行。

- 重启后数据和关系完整。
- 新 Session 自动召回正确 Scope 的记忆。
- 不同 Principal 的记忆不串用。

- Memory Extraction 使用 Durable Task，不依赖 fire-and-forget。
- Task 固定消息范围、幂等、可恢复。
- 水位只在成功提交后推进。
- Summary 真实生成并支持滚动更新。

### 检索

- FTS 和向量任一不可用时可以降级。
- Query Plan 已接入隐式和显式检索。
- 每条召回结果可解释来源、分数和路径。
- 过期、删除、被覆盖记忆不进入 Context。

### 生命周期

- 被动召回不错误强化记忆。
- 低分记忆不会未经确认自动删除。
- 覆盖、归档、失效和删除语义分离。
- Markdown 仅是可重建派生视图。

### 质量

- 全量测试通过。
- 本项目 Ruff 门禁通过。
- compileall 通过。
- 空库和升级 Migration 通过。
- 真实模型最小验收通过。
