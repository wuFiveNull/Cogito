# 长期记忆完整落地计划

## 1. 目标与实施原则

目标不是一次性实现全部高级能力，而是按可运行闭环递进：

```text
上下文正确
→ 可手动记忆
→ 自动注入
→ 自动提取
→ 摘要压缩
→ 智能检索
→ 生命周期治理
```

实施遵循以下权威设计：

- `DOMAIN-CONTRACTS / 1.13 MemoryItem`
- `MEMORY-LIFECYCLE / 1-14`
- `RETRIEVAL-CONTEXT / 1-15`
- `DATABASE-SCHEMA / 2-10`
- `RUNTIME-FLOWS / 即时 Turn 与后台 Task`

`plan/长期记忆.md` 作为目标设计参考，但不直接复制其中面向 PostgreSQL、多租户和 Vector 类型的示例结构。当前实现是本地优先、单所有者、SQLite 架构。

个人 Agent 第一版可以暂缓复杂安全、加密、多租户、审批和合规能力，但不能省略 `principal_id`、Scope、来源追踪、幂等、版本和覆盖关系，否则多 Channel 接入后会产生记忆串用和历史事实污染。

## 2. 阶段 0：修复 Context 基线

这是所有记忆工作的前置。

### 2.1 修改范围

- `src/cogito/runtime/context.py`
- `src/cogito/runtime/loop.py`
- `tests/runtime/test_context.py`
- `tests/runtime/test_loop.py`

### 2.2 实现内容

1. `ContextItem` 增加并保留原始 `role`。
2. 正确传递 `user/assistant/tool/system`，不能将所有非 system 内容转换为 user。
3. Context 固定按以下顺序装配：

```text
system policy
→ memory
→ summary
→ 历史消息（时间正序）
→ 当前用户输入
```

4. `message_upper_bound` 使用真实 `receive_sequence`，不能使用消息数量。
5. 多 ContentPart 消息必须完整读取，不能只取第一段。
6. Token 超限时从最旧的普通历史消息开始裁剪。
7. System Policy 和当前用户输入永远不能被裁剪。

### 2.3 验收条件

- 两轮及以上对话的消息角色正确。
- 当前输入永远位于消息序列最后。
- assistant/tool 历史不会变成 user。
- 多 ContentPart 内容顺序稳定。
- 当前输入和 System Policy 不被裁剪。
- 全部现有测试继续通过。

## 3. 阶段 1：长期记忆持久化与服务层

完成真正可用的 Memory 数据模型。

### 3.1 数据库 Migration

新增：

```text
src/cogito/store/migrations/0012_memory_mvp.sql
```

保留现有 `memory_items`，扩展至少以下字段：

```text
principal_id
scope_type
scope_id
canonical_key
explicitness
importance
confirmation_method
confirmed_by
confirmed_at
version
updated_at
deleted_at
```

继续保留：

```text
memory_id
kind
subject
predicate
value
source_type
source_id
confidence
status
valid_from
valid_to
supersedes_id
goal_*
created_at
```

第一版状态遵守 `DOMAIN-CONTRACTS`：

```text
candidate
confirmed
rejected
expired
```

第一版不增加 `active/superseded/archived` 状态：

- active 等价于当前有效的 confirmed；
- superseded 通过关系和 `valid_to` 表达；
- archived 在生命周期治理阶段实现；
- 删除通过 `deleted_at` 或物理删除表达。

增加索引：

```text
(principal_id, status, kind)
(principal_id, scope_type, scope_id, status)
(principal_id, canonical_key)
(source_type, source_id)
```

Migration 必须支持空库初始化、现有数据库升级、重复启动和外键检查。

### 3.2 Domain

修改：

- `src/cogito/domain/memory.py`
- `tests/domain/test_memory.py`

要求：

- 新字段进入序列化和反序列化。
- 校验 `confidence`、`importance` 范围。
- `version` 支持乐观锁。
- Goal 字段仍只对 `kind=goal` 生效。
- confirmed 记忆必须保留来源。

### 3.3 Repository

扩展 `src/cogito/store/memory_repo.py`：

```python
get(memory_id)
list_confirmed(principal_id, scope)
search(principal_id, query, scope, kinds, limit)
find_by_canonical_key(...)
insert(...)
update(...)
confirm(...)
reject(...)
expire(...)
supersede(...)
delete(...)
```

查询必须自动排除：

- 非 confirmed；
- 已过期；
- `deleted_at IS NOT NULL`；
- 已被有效新记忆覆盖的条目。

### 3.4 MemoryService

保留现有 Protocol，同时增加具体的 `SqliteMemoryService`。

职责：

- 唯一拥有 Memory 写入行为；
- 通过 UnitOfWork 管理事务；
- 从 Turn/Input Message 推导 Principal 和来源；
- 生成 canonical key 并去重；
- 同值重复时返回已有记忆；
- 新值覆盖旧值时建立 supersedes 关系；
- 使用 `version` 做条件更新。

将 `MemoryRepository` 接入 `UnitOfWork`，其他模块不得绕过服务直接修改记忆表。

### 3.5 验收条件

- 重启程序后记忆仍存在。
- 重复提交同一来源不会生成重复记录。
- 新偏好可以覆盖旧偏好。
- 旧偏好不再被召回。
- 不允许跨 Principal 查询。
- Migration 支持空库和已有数据库升级。

## 4. 阶段 2：用户驱动的记忆闭环

这是第一个真正可以交付使用的版本。

### 4.1 新增工具

新增：

```text
src/cogito/tools/remember_memory.py
src/cogito/tools/forget_memory.py
```

修复：

- `src/cogito/tools/recall_memory.py`
- `src/cogito/tools/registry.py`
- `src/cogito/service/agent_runner.py`

#### remember_memory

参数：

```text
kind
subject
predicate
value
scope_type
scope_id
confidence
```

规则：

- `principal_id` 不允许由模型传入；
- 根据当前 Turn 的 Input Message 推导 Principal；
- 用户明确要求记住时直接 confirmed；
- 保存 `source_type=message` 和 `source_id=message_id`；
- 重复调用依赖来源和 canonical key 幂等。

#### recall_memory

参数：

```text
query
kinds
scope_type
scope_id
limit
```

启动时必须注入真实 MemoryService/Repository，不能继续使用全局空 Repository 占位。

#### forget_memory

支持：

```text
按 memory_id 删除
按 canonical key 删除
按自然语言搜索后删除
```

存在歧义时返回候选，不直接批量删除。

### 4.2 模型提示词

System Prompt 增加简短规则：

```text
用户明确要求记住稳定事实、偏好、约束或目标时，调用 remember_memory。
用户修改已有事实时，写入新记忆并覆盖旧记忆。
用户要求忘记时，调用 forget_memory。
一次性请求、寒暄和模型推测不得写入长期记忆。
```

不使用正则检测“记住”“忘记”等表达，由模型进行语义判断。

### 4.3 验收场景

```text
用户：记住我默认用 Python。
Agent：调用 remember_memory。
重启 Cogito。
新 Session：
用户：我默认用什么语言？
Agent：回答 Python。
```

修改偏好：

```text
用户：以后默认改成 Rust。
```

预期：

- Rust 成为有效记忆；
- Python 记忆被 supersede；
- 默认召回不再返回 Python。

## 5. 阶段 3：隐式记忆注入

让用户不必每次要求 Agent 显式调用 `recall_memory`。

### 5.1 实现内容

在 `ContextBuilder` 接入 `MemoryService.retrieve()`。

第一版检索策略：

1. 根据当前输入消息定位 Principal。
2. 加载当前 Principal 的 confirmed 记忆。
3. Scope 顺序：

```text
session
→ conversation
→ principal-global
```

4. 记忆种类优先级：

```text
constraint
→ preference
→ active goal
→ fact
→ episode
```

5. 使用 `importance + confidence + recency` 做简单稳定排序。
6. 默认最多 20 条、约 2,000 tokens。
7. 注入格式明确标记来源和可靠程度：

```xml
<relevant_memories>
- [preference, explicit, confidence=1.0] 用户默认使用 Python。
- [constraint, explicit] 用户不希望自动推送夜间消息。
</relevant_memories>
```

8. Memory 以独立 `ContextItem(item_type="memory")` 注入。
9. 记录选中的 `memory_id`，为后续 Context Snapshot 持久化做准备。

个人 Agent 第一版记忆量较小时，可以加载受预算限制的 confirmed 集合，由模型理解语义，不需要立即引入向量检索。

### 5.2 验收条件

- 跨 Session 能召回个人偏好。
- 不共享不同 Session 的短期消息。
- 群聊不会读取其他 Principal 的个人记忆。
- 过期、拒绝、删除、被覆盖记忆不会注入。
- Memory 超出预算时可以稳定裁剪。

完成阶段 0-3 后形成 **Memory MVP v1**，已经适合日常运行。

## 6. 阶段 4：自动候选提取

MVP 稳定后，再让 Agent 自动学习。

### 6.1 在线提取

不增加语义关键词和正则规则。主模型回答后，可使用结构化输出提取候选：

```json
{
  "candidates": [
    {
      "kind": "preference",
      "subject": "owner",
      "predicate": "preferred_language",
      "value": "Python",
      "explicitness": "explicit_user_statement",
      "confidence": 0.98,
      "importance": 0.8,
      "reason": "用户明确说明默认使用 Python"
    }
  ]
}
```

仅自动写入：

- 明确稳定偏好；
- 明确长期约束；
- 明确目标；
- 用户纠正；
- 必须跨 Session 恢复的事实。

推断内容只能进入 candidate，不能自动 confirmed。

### 6.2 成本控制

- 主模型优先通过 `remember_memory` 工具主动写入；
- Session 结束或累计一定消息后再运行批量提取；
- 使用低成本模型角色 `memory_extractor`；
- 固定 JSON Schema；
- 提取失败不阻塞当前回复。

### 6.3 验收条件

- 普通一次性请求不入库。
- 用户明确偏好可以自动 confirmed。
- 模型推断只能成为 candidate。
- 候选带完整来源范围。
- 同一批次重复执行不会重复写入。

## 8. 阶段 6：上下文压缩

### 8.1 触发策略

按 token 压力触发：

```text
< 65%       不处理
65%-75%     记录压力
75%-85%     后台生成摘要
85%-95%     当前 Turn 前完成摘要
> 95%       快速摘要或裁剪兜底
```

Context 结构：

```text
system policy
memory
active goals
旧会话摘要
近期原始消息
当前输入
```

必须保留：

- 最初关键消息；
- 最近若干消息；
- 当前输入；
- 未完成任务状态。

Summary 激活后只替代 Prompt 中的旧消息，原始消息仍保存在数据库。

### 8.2 验收条件

- 超长会话不会超过模型 Context Window。
- 摘要覆盖范围无重叠和缺口。
- 最近消息不被摘要替换。
- 摘要失败时退回近期消息裁剪。
- Memory 提取失败不阻塞摘要激活。

## 9. 阶段 7：关键词检索与智能召回

先实现 FTS，再实现向量检索。

### 9.1 SQLite FTS5

新增 `memory_fts`，索引：

```text
subject
predicate
value
summary
```

实现要求：

- 写 Memory 时同步更新 FTS；
- 使用 BM25 排序；
- FTS 不可用时退化到 LIKE 查询；
- canonical key 精确命中单独执行。

### 9.2 Query Plan

输入：

```text
当前用户消息
最近 6 条消息
当前任务目标
当前 Scope
```

输出：

```text
query_text
kinds
time_range
scope
needs_episode
needs_goal
```

允许使用轻量模型进行语义改写，失败时直接使用原始输入。

### 9.3 排序

第一版评分：

```text
0.30 keyword
0.20 scope
0.15 importance
0.15 confidence
0.10 recency
0.10 explicitness
```

确定性过滤先于评分：

```text
Principal
Scope
status
valid_to
deleted_at
supersedes
```

### 9.4 验收条件

- 精确名称、日期和项目名可以召回。
- Scope 放宽过程可解释。
- 相同输入和数据库版本得到稳定排序。
- 每条结果包含来源和分数。

## 10. 阶段 8：Embedding 与混合检索

Embedding 不是 MVP 前置。

### 10.1 数据结构

新增 `memory_embeddings`：

```text
memory_id
embedding_model
embedding_version
vector

```

Embedding 是派生数据：

- MemoryItem 是权威事实源；
- Embedding 丢失后可以重建；
- 不同模型版本不能混合计算；
- 删除 Memory 时同步删除向量。

### 10.2 混合评分

```text
keyword
+ semantic similarity
+ scope
+ importance
+ confidence
+ recency
+ goal relevance
```

向量服务不可用时继续使用 FTS。

### 10.3 验收条件

- “喜欢简洁设计”可以召回“偏好极简界面”。
- 更换 Embedding 模型后可以重建。
- Embedding 故障不影响正常对话。
- 删除后不会因索引重建重新出现。

## 13. 第一里程碑完成标准

优先只完成 PR 1-4。

最终验收流程：

```text
1. 用户在 Channel A 说：记住我默认使用 Python。
2. Agent 调用 remember_memory。
3. SQLite 保存来源、Scope 和版本。
4. 重启 Cogito。
5. 用户从 Channel B 开启新 Session。
6. Context Builder 自动注入该偏好。
7. Agent 正确回答默认使用 Python。
8. 用户说：改成 Rust。
9. 新记忆覆盖旧记忆。
10. 用户说：忘掉我的默认语言。
11. 后续 Session 不再召回。
```

完成这个流程，就表示长期记忆已经真正落地并能运行。阶段 4-9 用于增强自动化、长会话能力、召回质量和生命周期治理，不是基本可用性的前置。
