# 00：AI 执行规则与证据格式

## 强制工作流

对每一项改动严格执行以下顺序：

1. 用 `rg`、目标源码和目标测试确认当前实现，不猜测文件或状态。
2. 列出将修改的生产模块、Event 类型、流类型和测试；若跨模块契约改变，检查 `DOMAIN-CONTRACTS`、`RUNTIME-FLOWS` 及对应专题文档。
3. 先实现 Event 写入，再实现 replay，再将命令/查询/恢复切到 replay，最后删除旧 SQL 分支。
4. 先运行聚焦单测，再运行阶段回归；失败时记录原因并修复，不把失败测试删除或降级。
5. 运行旧表依赖扫描；只有扫描允许项保留旧表名。

## 禁止事项

- 不新增任何持久化业务投影表、缓存表或“临时真相表”。
- 不在 `Event.attributes`、`summary` 或日志中写消息正文、prompt、完整模型响应、完整工具参数、访问令牌或密钥。
- 不以“Event 流为空”为由读取旧表。流为空表示实体不存在，除非仅在离线 legacy importer 中处理。
- 不伪造旧库中没有的中间生命周期；迁移只写最终导入快照 Event。
- 不在阶段 09 完成前执行删表、`0069` 或 `EventStoreCutover.apply=True`。
- 不使用可变表的最终状态来覆盖已经追加的 Event。

## Event 追加规则

- 所有 Event 都经过 Catalog 校验并指定 `producer`、`event_class`、`occurred_at` 和所属流。
- 一个命令先 replay 所需流，再用预期 `stream_version` 追加事实；跨流结果用 `append_many` 原子追加。
- 消费者的结果 Event 必须携带输入 Event 的 `causation_id`，并使用稳定的 `producer + idempotency_key`。
- 外部副作用只由 `*.requested` Event 发起；没有终结 Event 的请求才可被恢复扫描。

## 证据记录模板

在阶段文档或交接消息中记录：

```md
### 变更
- 模块：
- Event / 流：
- 删除的旧依赖：

### 验证
- 命令：
- 结果：
- 回放/幂等/恢复场景：

### 风险与后续
- 已知限制：
- 下一步：
```

任何未通过的测试、未清除的旧表读写或无法验证的副作用都必须列为阻塞项，不能以“计划完成”掩盖。
