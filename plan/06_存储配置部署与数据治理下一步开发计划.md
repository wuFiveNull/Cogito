# 06 存储、配置、部署、迁移与数据治理下一步开发计划

> 状态：draft  
> 对应目录：`markdown/06_infrastructure`  
> 设计依据：`STORAGE-DATA`、`OPS-GOVERNANCE`、`CONFIG-PROFILES`、`DATABASE-SCHEMA`、`LOCAL-OPERATIONS`

## 1. 目标

把当前 SQLite + TOML + 单进程/合并运行基线提升为可重复安装、Profile 隔离、配置可追溯、Payload 一致、Migration 可恢复、备份可验证、故障可演练的本地运行系统。

完成后应满足：

1. SQLite 是唯一事务事实源，Payload 通过受控协议与引用关联；
2. 每个 Profile 完全隔离配置、数据库、Payload、插件、Memory 和任务；
3. 配置有层级、严格 Schema、版本 Hash、Secret 引用、热更新和回滚；
4. Migration 分 online-safe 与 maintenance，升级前有一致备份；
5. 启动严格先恢复一致性，再开放 API/Gateway；
6. 备份包含 SQLite 快照、固定 Payload manifest、配置版本和插件版本；
7. 新安装、重启、强制终止、升级失败、磁盘不足和依赖故障可重复演练。

## 2. 当前基线

### 已有

- pyproject 和 CLI；
- 严格 TOML Config、config check/init/info/run/serve；
- SQLite Connection、WAL、foreign keys、busy timeout；
- Repository、UnitOfWork 和 28 个编号 Migration；
- UTC epoch milliseconds 时间工具；
- 大量唯一约束、状态表、FTS/Embedding 表；
- RuntimeApplication 装配和启动 Recovery；
- Web API/Worker 合并部署；
- config.example.toml 和少量 operations 文档；
- 空库、Schema、Query Plan、安装和启动测试基础。

### 缺口

- 项目环境不可重复：当前测试发现未安装包、缺 feedparser、MCP Python 路径错误；
- Web node_modules 未安装，本次无法验证前端构建；
- README 的测试数和 Migration 版本已过期；
- Profile 目前主要是配置字段，不是完整目录隔离和 CLI 生命周期；
- Payload Objects 有表，但缺完整内容寻址 Store、原子写和 GC；
- 配置版本、热更新和回滚未形成持久闭环；
- 备份/恢复 CLI、固定 manifest 和恢复演练缺失；
- Migration 缺 online-safe metadata、启动中断恢复和真实回填测试；
- 健康检查、优雅关闭、maintenance/recovery profile 和运维 Runbook 不完整；
- 缺正式 CI 发布门禁。

## 3. 关键决策

### 3.1 配置格式

文档中的 YAML 是示例，当前实现使用 TOML。除非 YAML 能解决明确问题，否则本阶段保留 TOML，避免无收益迁移。

需要创建 ADR，记录：

- TOML/YAML 候选；
- 注释、类型、生态和迁移成本；
- 是否支持旧 config.toml；
- 回滚和双格式窗口。

在 ADR 决定前，配置“语义 Schema”与文件格式解耦。

### 3.2 部署形态

默认保持：

```text
personal-agent = API + Worker + Scheduler + Interaction
channel-gateway = 独立进程（成熟后）
sqlite + payload = Profile 私有目录
```

开发模式可全部合并，但 Lease、Command、Outbox 和 Gateway 契约不省略。

## 4. 里程碑

### M0：可重复开发与质量基线

#### 工作项

1. 建立单一 Python 环境入口，要求 `python -m pip install -e ".[dev]"`；
2. 测试子进程统一使用 `sys.executable`；
3. CI 安装主依赖和 dev 依赖，RSS/feedparser 不依赖全局包；
4. Web 使用 `npm ci` 和 lockfile；
5. 固定命令：pytest、ruff、compileall、config check、npm build；
6. 分离环境失败和代码失败，禁止 README 手工宣称测试数字；
7. 测试报告自动生成收集数、通过数和耗时；
8. 清理当前 48 个 Ruff 问题和未定义符号；
9. README 通过脚本/发布流程更新版本和 Migration 数。

#### 验收

- 干净机器按 README 可安装；
- Python 和 Web 全部构建；
- 单一命令完成质量门禁；
- 测试不引用开发者绝对路径。

### M1：Profile Home 与目录隔离

#### 目标布局

```text
default home/
  config.toml
  .env or secret refs
  data/database/cogito.db
  data/payload
  data/cache
  plugins
  skills
  memory
  logs
  backups

profiles/<name>/
  同构且完全隔离
```

#### 工作项

1. 实现 `get_home(profile)` 和 `display_home()`；
2. 禁止业务代码硬编码 `.workspace`、`~/.cogito` 或当前 cwd；
3. CLI 支持 `-p/--profile` 和 list/create/delete；
4. create 从默认模板创建，不复制 Secret 明文；
5. delete 先展示路径、容量、运行状态并要求确认，再创建删除 Audit；
6. Profile 管理锚定全局 profiles root，不受活动 Profile 影响；
7. Profile 之间禁止共享写目录；共享 Skill 只能只读且明确声明；
8. test Profile 使用临时 HOME、固定时钟、Stub、禁止真实副作用；
9. recovery Profile 默认只读，真实副作用需要显式解锁；
10. 自动测试不得写真实用户 HOME。

### M2：配置 Schema、版本与 Secret

#### 工作项

1. 配置层级：defaults → profile → local override → environment → validated runtime override；
2. 后层只覆盖显式字段，未知字段报错；
3. 顶层覆盖 runtime/storage/interaction/channel/conversation/agent/model/memory/capability/sandbox/worker/scheduler/connector/proactive/security/observability/retention/backup；
4. 每项记录类型、默认值、范围、敏感性、可热更新和影响模块；
5. 跨字段校验：
   - remote bind 需要 auth/TLS；
   - Lease > Heartbeat；
   - output budget < context window；
   - Payload/DB 路径位于活动 Profile；
6. Secret 只保存 secret_ref，由环境/Keyring/Store 解析；
7. config dump、Dashboard、Trace 只显示引用名；
8. 规范化配置计算 config_version/schema_version/content_hash/source_layers；
9. Attempt、Task、Decision 保存使用的 config version；
10. 热更新先 parse/validate/dry-run，再原子激活；
11. 权限收紧可影响当前执行，放宽仅下一 Attempt；
12. 配置失败继续使用旧版本；支持回滚和 Audit。

#### 测试

- 层级覆盖；未知字段/错误类型；
- Secret 不出现在错误和 dump；
- 路径逃逸 Profile；
- 热更新竞态；
- 回滚；
- TOML 兼容迁移。

### M3：Payload Store

#### 对象

```text
payload_id
storage_uri
sha256
content_type
size_bytes
compression
encryption
redaction_level
retention_class
reference_count_hint
created_at
```

#### 写协议

1. 在目标文件系统的临时路径写入；
2. 流式计算 hash/size/type；
3. fsync + close；
4. 原子 rename 到内容寻址路径；
5. SQLite 短事务写 metadata 和业务引用；
6. 事务失败留下的文件由安全期后的 GC 清理。

#### 工作项

1. 实现 PayloadStore Port 和 LocalContentAddressedStore；
2. 禁止业务模块手工拼接 Payload 路径；
3. inline threshold 统一配置；
4. 下载/附件文件名仅展示，实际路径由 hash 决定；
5. 读取校验 size/hash，损坏时返回明确错误；
6. reference_count 只作提示，删除前扫描真实数据库引用；
7. GC 基于固定数据库快照、最小安全期和 dry-run；
8. 临时下载/Sandbox 文件使用 volatile retention；
9. 明文策略显式 `payload.encryption=none`，检查目录仅当前用户可读；
10. 远程/共享部署前阻止明文默认继续使用。

### M4：Repository、UoW 与 SQLite 写策略

#### 工作项

1. Repository 按 Aggregate，不提供业务层万能 execute SQL；
2. Application Service 使用 UoW 提交业务变更和 Outbox；
3. 每个进程独立连接，读写连接用途明确；
4. 写事务短小，外部模型/网络/MCP/Channel/Shell 不在事务中；
5. database busy 有有限次数、抖动和指标，不无限增 timeout；
6. 大查询使用只读连接和游标分页；
7. 队列查询添加部分索引并保留 EXPLAIN QUERY PLAN 测试；
8. 所有持久文件使用同目录临时文件 + fsync + 原子替换；
9. Scheduler、Backup、Config Migration 使用跨进程锁；
10. Windows/POSIX 文件锁分别测试，无法锁时明确降级限制。

### M5：Schema 清点与约束补齐

#### 工作项

1. 以 `DATABASE-SCHEMA / 2` 建立“规范表 → 实际表 → Migration → Repository → 测试”矩阵；
2. 核对缺失的 commands、side_effect_receipts、config_versions、traces/spans 等；
3. 所有可变聚合补 version；状态对象补时间；软删除补 deleted_at；
4. 状态、外键、幂等和查询字段不得藏在 JSON；
5. ToolCall 到 RunAttempt/TaskAttempt 使用二选一约束；
6. provisional Delivery 最终化后 message 绑定可验证；
7. Payload 引用统一；
8. 补齐文档列出的唯一约束；
9. 添加 queued/pending/ready/approval 等部分索引；
10. 时间列统一 INTEGER epoch ms，Contract 边界转 RFC3339。

### M7：备份

#### 备份内容

- SQLite online snapshot；
- 从该快照生成的固定 Payload manifest；
- manifest 引用的内容寻址文件；
- 非明文 Secret 的配置版本；
- Plugin manifest/version/state metadata；
- Migration/app version；
- 合法的 Plugin external backup paths。

#### 工作项

1. `cogito backup create/list/verify/prune`；
2. 备份获取进程锁，允许短时在线写但以 SQLite snapshot 为边界；
3. 从 snapshot 而非活动库读取 Payload 引用；
4. 每个文件记录 hash/size；
5. 插件外部路径仅允许用户 HOME 内，禁止网络和 initialize；
6. 归档不包含 Secret 明文；
7. 备份写入 staging，完整校验后原子发布；
8. prune 遵循 retention/hold，不删除最后一个已验证备份；
9. 定期自动 verify，并记录应用版本和耗时。

### M8：恢复与恢复演练

#### 恢复流程

```text
restore into isolated directory
→ verify archive/hash
→ SQLite integrity + foreign keys
→ verify Payload manifest
→ validate config/plugin compatibility
→ clear expired leases
→ scan unknown Tool/Delivery
→ rebuild FTS/Embedding/cache
→ start recovery profile
→ human confirmation
→ enable real side effects
```

#### 工作项

1. `cogito backup restore --target-profile ...` 默认不覆盖现有 Profile；
2. 原 Profile 覆盖恢复需要额外确认和自动 pre-restore backup；
3. 恢复前后生成差异报告；
4. 过期主动通知默认不发送；
5. unknown 副作用进入 reconcile/manual review；
6. 派生索引重建不修改 Canonical 数据；
7. 恢复演练使用真实备份，不只校验压缩包可打开；
8. 每个发布周期至少完成一次自动恢复 smoke；
9. 定期人工确认 Web/CLI/Message/Memory/Task 可读。

### M9：启动、健康和关闭

#### 启动顺序

严格执行：最小配置/Secret → SQLite/WAL/version → Migration → Payload → Policy/Provider → Plugin 验证 → Recovery → Worker/Scheduler/Event → API → Gateway → Web。

Recovery 完成前：

- readiness=false；
- 不发送积压主动通知；
- 只允许安全 Query/恢复 Command。

#### 健康

分别提供 liveness、readiness、dependency、backlog，并返回 degraded reason。单个 Connector/非关键 Plugin 故障不把只读系统判死。

#### 关闭

1. 停止新普通工作，保留短控制窗口；
2. 停 Scheduler 和新 Lease；
3. Attempt 到安全点写 Checkpoint；
4. 不可确认副作用标 unknown；
5. Flush Outbox/Trace，释放或缩短 Lease；
6. 超时强停但不伪造 completed。

## 5. 数据治理

### 保留类别

- hot：当前和近期；
- warm：压缩历史；
- archive：用户长期保存；
- volatile：临时/Sandbox；
- secret：受限访问和审计。

### 删除流程

```text
mark deleted
→ revoke/hide
→ deletion Task
→ clean/redact Payload and indexes
→ GC unreferenced objects
→ minimal audit tombstone
```

磁盘压力先清 cache/过期 Trace，再归档旧 Payload；不得直接删除 Message 和 confirmed Memory。
