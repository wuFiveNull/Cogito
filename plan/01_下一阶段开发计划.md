# Cogito 下一阶段开发计划

## 1. 结论

下一步不是直接接入真实模型、Telegram/LangBot 或主动推送，而是完成一个可验证的“入站消息到排队 Turn”最小纵向闭环。

在实现该闭环前，必须先收敛当前两套 Python 包和不一致的配置、状态机、Schema。否则新功能会继续建立在互相冲突的类型和导入路径上。

本阶段目标：

```text
可安装且唯一的 src/cogito 包
→ 可校验的配置和可重复 Migration
→ ChannelEnvelope 入站
→ Inbox 幂等
→ Conversation / Session / Message / Turn
→ Event Outbox
→ 重复请求返回原结果
```

该阶段不调用模型、不执行 Tool、不发送 Delivery，先建立后续所有运行时能力依赖的事务事实源。

## 2. 判断依据

### 2.1 已有基础

- `src/cogito/` 已包含领域对象、跨模块 Envelope、SQLite 初始 Schema、Migration 和 Service Protocol。
- `tests/` 已覆盖主要领域对象和初始数据库约束。
- 强制使用 `src/` 包并启用 pytest importlib 模式时，当前 61 个测试全部通过。
- `pyproject.toml` 已声明 `src` layout，因此 `src/cogito/` 应作为唯一正式实现。

### 2.2 当前阻塞

1. 仓库同时存在根目录 `cogito/` 和 `src/cogito/`。从仓库根目录执行时，旧包会遮蔽正式包，默认 `pytest` 报 `No module named 'cogito.store'`，`python -m cogito` 也无法启动。
2. `src/cogito/__main__.py` 的 `init` 仍是 TODO，尚未创建运行目录、加载配置或初始化数据库。
3. `config.toml` 使用 `llm/agent/channels/storage`，而 `src/cogito/config.py` 只读取 `database/workspace`，配置会被静默忽略。
4. 当前 Turn 状态只有 `created/running/...`，但权威规范要求 `accepted → queued → running`，并要求 `version`、`input_message_id`、`next_attempt_at` 等字段。
5. 当前 `TurnService.accept()` 的注释要求接收入站时创建 RunAttempt；权威规范明确要求 Dispatcher 获得 Lane 后才创建 RunAttempt。
6. 当前 Schema 使用 RFC 3339 文本时间且缺少聚合版本、Event/Outbox、MessageRevision 等关键结构；数据库规范要求库内时间统一为 UTC epoch milliseconds，并要求可变聚合使用 `version`。
7. Migration 仍是单个内嵌 `SCHEMA_SQL`，没有单调编号文件、兼容检查和中断/升级测试。

设计依据：

- `ARCH-OVERVIEW / 4. 三类运行模型`：Turn Orchestrator、Durable Job Runner、Event Bus 必须分离。
- `GLOBAL-INVARIANTS / 1. 数据事实`：SQLite 是当前状态和事务意图的唯一事实源。
- `EXECUTION-LIFECYCLE / 2. 状态机`：Turn 使用 `accepted → queued → running`，状态变更使用版本条件更新。
- `EXECUTION-LIFECYCLE / 3. 创建与执行事务`：入站事务不提前创建 RunAttempt。
- `MESSAGE-PERSISTENCE / 4. 入站事务`：Inbox、Message、ContentPart、Turn 和 Event Outbox 同事务写入。
- `EVENT-OUTBOX / 2. 发布事务`：业务状态与 Outbox 必须共同提交或共同回滚。
- `DATABASE-SCHEMA / 1. SQLite 模式`：库内时间使用 UTC epoch milliseconds。
- `CONFIG-PROFILES / 1. 配置层级` 与 `CONFIG-PROFILES / 6. 校验`：配置必须分层、拒绝未知字段并执行启动校验。

## 3. 实施顺序

### P0：恢复单一工程基线

目标：所有开发命令默认可用，仓库中只有一套正式 Python 包。

- 以 `src/cogito/` 为唯一实现目录。
- 对根目录 `cogito/` 做一次差异清单；只迁移符合现行文档且 `src` 尚缺失的状态机、端口或错误类型，然后删除旧包。
- 配置 pytest 的 import 模式或安装方式，确保无需手工设置 `PYTHONPATH`。
- 补齐 CLI entry point，使 `python -m cogito info` 和项目脚本从仓库根目录可执行。
- 更新 README，将仓库定位从“仅架构文档集”改为“架构知识库 + 早期实现”。
- 保留用户当前对 `CLAUDE.md` 的未提交修改，不在本阶段改写。

验收：

```powershell
python -m pytest -q
python -m cogito info
python -m compileall -q src
```

### P1：冻结配置、领域状态和数据库契约

目标：代码、配置和 Schema 对同一组字段与状态达成一致，再开始应用服务实现。

- 按 `CONFIG-PROFILES` 建立严格配置模型，至少支持 `runtime`、`storage` 和 `interaction`；未知字段报错，不再静默忽略。
- 将运行时数据库、Payload 和日志默认落到 `.workspace/`，并明确相对路径基准。
- 不在日志、`repr` 或错误中输出 API Key；当前明文 `config.toml` 只作为本机配置读取。
- 对齐 Turn/RunAttempt 字段、状态和状态转移，增加聚合 `version`。
- 对齐 Message、Inbox、EventEnvelope、Outbox 的幂等键、因果链和时间表示。
- 将 Schema 改为编号 Migration；先实现本阶段闭环需要的最小表集，不一次性实现全部 35 份文档描述的表。
- 增加空库、重复迁移、上一版本升级、外键、唯一约束和事务回滚测试。

验收：

- `cogito init` 可幂等创建 `.workspace/` 和数据库。
- 配置错误在启动前给出明确且脱敏的诊断。
- 重复运行 Migration 不改变业务数据。
- 领域状态、数据库 CHECK 和文档状态名称完全一致。

### P2：实现入站事务纵向闭环

目标：实现第一个真实应用服务 `accept_inbound`，但不启动 Agent Loop。

单个短事务完成：

1. 校验 `ChannelEnvelope` 和 Schema 版本；
2. 以 `(channel_instance_id, platform_event_id)` 写入或查询 Inbox；
3. 解析或创建 Principal、Endpoint、Conversation 和 Session；
4. 持久分配 `receive_sequence`；
5. 写入 Message 与 ContentPart；
6. 创建 `accepted` Turn 并推进到 `queued`；
7. 写入 `InboundMessageAccepted`/`TurnQueued` Event Outbox；
8. 提交后返回 `message_id`、`turn_id`，不返回 `attempt_id`。

实现约束：

- 通过应用服务和 Repository/Unit of Work 访问数据库，其他模块不直接操作表。
- 重复入站返回第一次创建的 ID，不创建第二个 Message、Turn 或 Outbox Event。
- 任意一步失败时，Inbox、业务对象和 Outbox 全部回滚。
- 接收入站期间不调用模型、网络、Channel、MCP 或 Tool。
- 本阶段不创建 RunAttempt；它属于后续 Dispatcher 获取 Lane 后的事务。

必需测试：

- 首次文本入站成功；
- 同一事件重复提交；
- 相同文本但不同平台事件不被误去重；
- Conversation/Session 复用与 `receive_sequence` 单调递增；
- ContentPart 多段内容；
- Outbox 与业务事务共同回滚；
- 两个并发重复请求最多创建一个逻辑 Turn；
- 不受信任内容只作为数据持久化，不改变策略字段。

### P3：Dispatcher、Lane 与 Stub Agent 闭环

P2 完成后再实施：

- Dispatcher 按 `context_partition_key` 和优先级领取 queued Turn。
- 领取 Lane 后原子创建 RunAttempt，并推进 Turn/Attempt 到 running。
- 使用固定 Stub Agent 生成 Assistant Message，不先接真实模型。
- 成功提交时原子写入最终 Message、Delivery Outbox、Event Outbox，并完成 Attempt/Turn。
- 覆盖取消竞态、旧版本提交、失败重试和进程恢复测试。

### P4：Outbox Worker 与 Fake Delivery

P3 完成后再实施：

- 实现 Outbox Lease、同聚合顺序、重试和 dead letter。
- 使用 Fake Gateway 验证 Delivery，不接真实 Channel。
- 覆盖“外部成功、本地结果未知”的 reconcile 路径。

完成 P4 后，才具备接入 Model Adapter、LangBot Bridge 和真实 Delivery 的可靠基础。

## 4. 建议拆分

| 变更集 | 内容 | 可独立验收 |
|---|---|---|
| PR 1 | 删除双包歧义、修复 CLI/pytest、更新 README | 默认开发命令全部通过 |
| PR 2 | 严格配置、`.workspace` 初始化、编号 Migration | `cogito init` 幂等且配置错误可诊断 |
| PR 3 | 对齐 Turn/Message/Event/Outbox 契约与 Schema | 领域、Schema、Migration 测试通过 |
| PR 4 | `accept_inbound` 应用服务和并发幂等测试 | 入站事务闭环通过 |
| PR 5 | Dispatcher + Lane + Stub Agent | 无外部依赖的 Turn 完成闭环 |
| PR 6 | Outbox Worker + Fake Delivery | 崩溃、重试、unknown 对账通过 |

每个 PR 都应同时更新测试和受影响的文档引用；不得通过一次大提交同时改写领域模型、Schema、运行时和外部集成。

## 5. 暂不实施

在 P0-P2 完成前暂缓：

- 真实 LLM/Embedding 调用；
- Telegram、QQ、Web UI 或 LangBot Gateway；
- Tool/MCP/Sandbox；
- 长期记忆检索和向量索引；
- Scheduler、Connector、主动推送和 Drift；
- 完整 Dashboard；
- 全量实现数据库规范中的所有表。

原因不是这些能力不重要，而是它们都依赖稳定的配置、事务、幂等、状态机和 Outbox。先建立这些基础可以避免后续外部副作用无法恢复或重复执行。

## 6. 阶段完成定义

本计划的“下一阶段完成”指 P0-P2 全部满足：

- 仓库只有一套正式 Python 包；
- 默认测试、CLI 和编译命令可运行；
- 配置字段不会被静默忽略；
- Migration 可重复、可升级、可回滚验证；
- 入站消息、Turn 和 Event Outbox 在同一事务中提交；
- 重复与并发入站保持逻辑幂等；
- 入站阶段不创建 RunAttempt、不调用任何外部系统；
- 所有新增保证都有确定性测试。

达到这一点后，开发下一步自然进入 P3，而不是直接跨越到真实模型或 Channel。
