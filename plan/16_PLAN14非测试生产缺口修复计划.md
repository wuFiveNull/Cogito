# PLAN-16：PLAN-14 非测试生产缺口修复计划

## 1. 计划结论

本计划只处理生产代码和运行闭环，不包含单元测试、集成测试、Ruff、类型检查、构建门禁等工作。

当前 PLAN-14 已具备 Memory/Knowledge 的主要 Schema、Service、Task Handler、部分 Outbox、Context 注入和控制面，但仍存在以下类型的未完成项：

1. Worker 启动和 Memory Extraction 存在可能中断运行或错误推进 watermark 的 P0 问题；
2. Memory/Signal/Knowledge 事实与 Outbox/Audit/Receipt 尚未形成可靠的同事务提交；
3. explicit remember、Session close、Task success、Connector ingestion 等生产入口尚未接线；
4. Knowledge 已能写入 Embedding，但真实 Turn/Tool 不会生成 query vector，语义检索没有进入生产路径；
5. ContextBuilder 只统一了 Memory 与 Knowledge，Message/Summary/TaskState 尚未统一为 RetrievalCandidate；
6. erase、expected_version、Receipt、Explain、指标与报告仍未收口；
7. PLAN-14/manifest 的 completed 状态早于真实生产闭环完成。

排除测试门禁后，当前生产实现完成度估算约为 **68%**。本计划完成后，PLAN-14 才能按原始 DoD 标记为 completed。

执行顺序：

```text
M0 状态纠偏与 Worker 启动
→ M1 无损 durable memory extraction
→ M2 Memory/Signal/Outbox 原子事务
→ M3 Signal、Correct、Erase 与维护策略闭环
→ M4 Connector → Knowledge durable ingestion
→ M5 Knowledge hybrid retrieval 与删除级联
→ M6 全来源 Unified Retrieval + Snapshot
→ M7 Query/Command/指标/报告与状态收口
```

## 5. M1：无损 durable memory extraction

### 5.1 统一请求服务

优先在现有 `memory_extractor.py` 或相邻 service 中增加薄的 `MemoryExtractionRequestService`，不新增无必要的框架层。职责：

- 读取 watermark；
- 计算 extraction window；
- 调用 `ExtractionTriggerPolicy`；
- 创建 durable `memory.extract` Task；
- 写 `MemoryExtractionRequested`；
- 使用确定性 idempotency key。

幂等键：

```text
memory.extract:{conversation_id}:{session_id}:{from}:{to}:{trigger}:{extractor_version}
```

### 5.2 三类生产触发

- `turn_completed`：现有 TurnCompleted Consumer 调用统一请求服务；
- `session_closed`：Session reset/close 的事实事件调用统一请求服务；
- `explicit_remember`：Tool 仍直接确认 Memory，同时可创建高优先级上下文提取 Task。

### 5.3 严格失败语义

- strict 模式下 JSON 解析失败必须抛出 `MemoryExtractionParseError`；
- response schema 不匹配必须抛错；
- 任意候选写入失败使整个 extraction 事务回滚；
- 有效模型响应且 candidates 为空，才是成功的零候选窗口；
- 模型、解析、授权、来源或 DB 错误都不得推进 watermark。

### 5.4 单一 watermark 所有者

- 从 `MemoryExtractor` 删除 watermark 写入职责；
- Task Handler 是唯一推进者；
- Handler 使用 payload 中的 conversation/session key；
- Memory、Source、Relation 全部成功后 CAS 推进；
- CAS 竞争已覆盖目标范围时幂等成功，否则重试。

### 5.5 完成条件

- 三类 trigger 都创建 durable Task；
- 不存在 fire-and-forget 或第二写路径；
- 失败窗口不会推进 watermark；
- 同一窗口重复请求只产生一个有效 Task。

## 6. M2：Memory、Signal 与 Outbox 原子事务

### 6.1 Memory 事务

- MemoryService 使用调用方 UnitOfWork；
- 删除 `_emit_memory_event()` 的 catch-all；
- MemoryItem/Source/Relation 与 Outbox 一次 commit；
- event aggregate version 使用 MemoryItem.version；
- event id/idempotency key 由业务事实确定。

### 6.2 Signal 事务

- SignalRepository insert 和 `MemorySignalRecorded` 同一事务；
- 事件失败必须让 signal 写入失败；
- 幂等冲突返回已处理，不能重复发事件。

### 6.3 Weight 事务

- recompute 完成后先写 `MemoryWeightRecomputed`；
- 权重缓存、algorithm version、last_weight_update、Outbox 一次提交；
- 事件 payload 保存 policy version、处理数量和 checkpoint。

### 6.4 Knowledge 事务

- 删除 KnowledgeService 聚合方法内部的 `commit()`；
- register/ingest/invalidate/erase 由 Task/Command 外层 UoW 提交；
- Audit、Receipt、Outbox 和事实修改共享事务。

### 6.5 完成条件

- 领域事件失败时对应事实不会提交；
- 同一 aggregate 的事件版本单调递增；
- Command 成功时 Audit 与事实同时存在。

## 7. M3：Memory Signal、Correct、Erase 与维护闭环

### 7.1 Signal 生产者

- Context 注入 Memory → `exposed`；
- recall_memory 命中 → `exposed`，但不增加 reinforcement；
- confirm-memory → `user_affirmed`；
- correct-memory 新 Memory → `user_corrected`；
- reject/负反馈 → `negative_feedback`；
- Task 明确声明并成功使用 Memory 依赖 → `task_succeeded`。

Task payload 或 TaskAttempt result 增加 `memory_dependencies`，禁止从任意文本猜测依赖。

### 7.2 Correct Service

新增/完善 `MemoryService.correct()`：

1. 校验 owner、status、expected_version；
2. 创建 corrected confirmed Memory；
3. 复制或重建可信 Source；
4. 旧 Memory 标记 superseded；
5. 建立 supersedes Relation；
6. 新 Memory 写 user_corrected；
7. 写 MemoryConfirmed/MemorySuperseded/MemorySignalRecorded；
8. 一次事务提交。

Command 不得访问 `svc._repo`。

### 7.3 Erase Service

新增 `erase-memory`：

- 要求 idempotency key 与 expected_version；
- 清除 value、敏感来源引用和 payload；
- 保留 memory_id、owner hash、时间、reason、receipt_id 等最小 tombstone；
- 写删除事件、Audit 和 Erasure Receipt；
- 重复 erase 返回原 Receipt。

### 7.4 配置化维护

在 `[memory.weight]` 增加：

```toml
recompute_interval_seconds = 600
consolidate_interval_seconds = 3600
batch_size = 500
algorithm_version = "v2"
```

Scheduler 不再硬编码时间窗口。

### 7.5 完成条件

- 计划定义的每类 Signal 都有真实生产者；
- correct/erase 只通过 MemoryService；
- 并发 Command 不会覆盖更新版本；
- erase 可通过 Receipt 对账。

## 8. M4：Connector 到 Knowledge durable ingestion

### 8.1 Connector 接线

普通 Connector 和 MCP Connector 在持久化新/修改内容后创建 `knowledge.sync_source` Task：

```json
{
  "stable_source_id": "connector:{connector_id}:{source_item_id}",
  "source_kind": "connector",
  "content_hash": "...",
  "payload_ref": "...",
  "principal_id": "...",
  "trust_label": "external"
}
```

来源删除创建 `knowledge.invalidate` 或 `knowledge.erase` Task。

### 8.2 Payload 边界

- 大正文写入现有 Payload/Object Store；
- Task 仅保存 payload_ref，不内联无界正文；
- Task payload 保存 config version、source version、parser policy version。

### 8.3 Durable 阶段

```text
knowledge.sync_source
→ register/change Resource
→ knowledge.ingest
→ parse + segment
→ knowledge.embed
→ refresh KNOWLEDGE.md
```

每一步有独立幂等键和 checkpoint；上一步完成后才创建下一步。

### 8.4 API 行为

- register/refresh Command 只登记意图和创建 Task；
- 不在 HTTP 请求中同步 parse/embed；
- 使用 Application 注入的 KnowledgeService 工厂；
- Command 返回 resource_id + task_id。

### 8.5 完成条件

- Connector/API 内容都经 durable Task 进入 Knowledge；
- 重复 content hash 不重复 parse/embed；
- 进程中断后可从未完成阶段继续。

## 9. M5：Knowledge hybrid retrieval 与删除级联

### 9.1 Query Embedding

KnowledgeReader 增加异步检索能力，或通过现有异步 Retrieval Orchestrator 调用 Embedding Provider。禁止在运行中的 event loop 内使用 `asyncio.run()`。

流程：

```text
query
→ query embedding
→ FTS/BM25 recall
→ cosine recall
→ principal/scope/status/trust hard filter
→ score normalization
→ merge + dedupe + rerank
→ RetrievalCandidate
```

Provider 不可用时显式降级为 FTS-only，并记录 retrieval path 和降级原因。

`search_knowledge` 与 ContextBuilder 必须调用同一个 KnowledgeRetriever。

### 9.2 删除传播

KnowledgeService 只处理 Knowledge 聚合：

1. Resource/Document/Segment/Embedding 失效或擦除；
2. 发布 `MemorySourceInvalidated`；
3. Consumer 调用 MemoryService；
4. MemoryService 根据剩余来源决定 keep/review/expire；
5. 不允许 KnowledgeService 直接 UPDATE memory 表。

### 9.3 Knowledge erase

- 清除 Segment 正文；
- 删除 Embedding vector；
- 清除 FTS 行；
- 删除外部 payload；
- 保留最小 Resource tombstone；
- 写 KnowledgeResourceDeleted、Audit 和 Erasure Receipt。

### 9.4 完成条件

- 配置 Embedding 后，真实 Turn 与 Tool 都执行 hybrid retrieval；
- 降级路径可解释；
- Knowledge 删除不跨边界直接修改 Memory；
- 被擦除正文和 vector 不再保留。

## 10. M6：全来源 Unified Retrieval 与 Snapshot

### 10.1 Source Adapter

在现有 ContextBuilder 内优先收敛，避免创建过度抽象：

- RecentMessageRetriever；
- SessionSummaryRetriever；
- MemoryRetriever；
- KnowledgeRetriever；
- TaskStateRetriever。

全部输出 `RetrievalCandidate`。

### 10.2 检索与预算顺序

1. 读取当前 query 与 Session 上界；
2. 各 source adapter 召回候选；
3. Principal/Scope/Trust/Status/version 硬过滤；
4. 各来源内部 score 标准化；
5. 标记 system policy、当前输入、active goal/constraint 为 protected；
6. 单次 `allocate_budget()` 分配所有来源；
7. 统一选择并生成 ContextItem；
8. 持久化 Snapshot。

MemoryRetriever 必须接收当前 query，不能只按 principal 列表召回。

### 10.3 Snapshot provenance

每条 selected/excluded candidate 保存：

- candidate_type/candidate_id；
- source refs 与 source version；
- keyword/semantic/recency/importance/trust/final score；
- retrieval path；
- token estimate/actual tokens；
- policy/config/query plan version；
- authorization filter reason；
- budget exclusion reason；
- selected position。

### 10.4 完成条件

- 所有来源共享一个候选和预算决策；
- active constraint/goal 不被 Knowledge 挤出；
- Snapshot 能还原某次 Turn 为什么选中或排除某条内容。

## 13. 迁移与兼容策略

1. 新字段只做 expand，不立即删除旧字段；
2. Task payload 增加 version，旧 payload 走明确兼容分支；
3. Knowledge feature flag 关闭时不得改变现有 Turn；
4. 新 Event Consumer 必须以 event_id + consumer_name 幂等；
5. 新 Command 保留旧 delete-memory，但标记 deprecated，erase-memory 使用新语义；
6. ContextBuilder 切换采用 adapter → dual record → unified select 三步，不同时维护两套写路径；
7. 不恢复 fire-and-forget extraction；
8. 不允许从 Knowledge 模块直接操作 Memory Repository。

## 14. 回滚策略

- M1 可停止创建新 extraction Task，但不恢复旧 fire-and-forget；
- M2 出现事件版本问题时暂停对应 Command/Task，不允许降级为吞事件；
- M4/M5 可通过 `knowledge.enabled=false` 停止 Knowledge 注入；
- Hybrid retrieval 可降级 FTS-only，但必须保留降级指标；
- Unified Retrieval 可临时关闭 Knowledge adapter，但 Snapshot schema 保持兼容；
- erase 操作不可自动回滚，必须先完成 Receipt 和幂等设计。
