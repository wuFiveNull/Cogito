# 01 架构边界与领域契约下一步开发计划

> 状态：draft  
> 目标版本：0.2.x 架构收敛阶段  
> 对应目录：`markdown/01_architecture`  
> 主要设计依据：`ARCH-OVERVIEW`、`DOMAIN-CONTRACTS`、`GLOBAL-INVARIANTS`、`SYSTEM-BOUNDARIES`

## 1. 计划目的

当前代码已经具备主要领域实体、状态机、Repository、Service、运行时编排和 SQLite 事实源，但“模块化单体”的边界仍主要依赖开发者自觉。下一阶段不以增加业务功能为重点，而是把系统级设计约束转化为可检查的代码结构、稳定契约和自动化门禁。

本计划完成后，应做到：

1. 每个核心聚合有明确且唯一的写入入口；
2. Domain 不依赖数据库、HTTP、模型或 Channel SDK；
3. Agent Runtime 不直接写 Repository；
4. 跨进程 Envelope 有统一版本、Trace 和安全错误约束；
5. 全局不变量能映射到数据库约束、服务条件更新或测试；
6. 架构违规在 CI 中直接失败，而不是通过代码评审人工猜测。

## 2. 当前基线

### 2.1 已有能力

- `src/cogito/domain/` 已包含 Principal、Conversation、Session、Message、Turn、Task、Delivery、Memory 等领域对象；
- `src/cogito/domain/state_machines.py` 已集中实现主要状态转移；
- `src/cogito/contracts/` 已存在 Envelope 与 TraceContext；
- `src/cogito/service/` 已有 Inbound、Completion、Delivery、Memory、Recovery 等服务；
- `src/cogito/store/` 已有 Repository、Unit of Work 和 Migration；
- Query/Command API 已区分读写入口；
- 主要入站、Turn、Task、Delivery 路径已有幂等、Lease 和版本检查。

### 2.2 主要缺口

- 没有自动化依赖方向测试，跨层导入只能靠 Ruff 和人工发现；
- 多个 Service 直接组合具体 Repository，公开 Facade 与内部实现边界不够清晰；
- 部分状态仍可被多个模块通过 SQL/Repository 方法修改；
- `AgentRequest`、`AgentReply`、Command、Event 等文档契约未全部形成唯一代码模型；
- Envelope 版本兼容策略缺少解码测试；
- ResourceBudget、Patch、SideEffectReceipt 等架构概念只部分落地；
- 全局不变量没有统一的“设计条目 → 执行位置 → 测试 ID”矩阵。

## 3. 范围与非目标

### 3.1 本期范围

- 核心模块公开面和导入方向；
- 聚合状态所有权；
- 跨模块及跨进程契约；
- 全局不变量验证矩阵；
- 兼容性和架构门禁；
- 为后续 Runtime、Capability、Background、Interaction、Infrastructure 计划提供稳定接口。

### 3.2 非目标

- 不重写现有领域实体；
- 不在本计划实现完整 Sandbox、Plugin Runtime 或备份系统；
- 不改变具体状态机语义，字段和状态仍以对应 functional-spec 为准；
- 不为了“纯架构”拆成微服务；个人部署继续允许合并进程。

## 4. 总体实施顺序

```text
架构现状扫描
→ 模块公开面定义
→ 聚合写入入口收敛
→ 契约模型与版本兼容
→ 全局不变量测试矩阵
→ CI 架构门禁
→ 文档与代码双向校验
```

后续模块计划依赖本计划产出的公开接口，因此优先完成 M1～M3，再并行推进其他目录。

## 5. 里程碑

### M1：建立架构基线与违规清单

#### 工作项

1. 生成 `src/cogito` 的模块依赖图，至少覆盖：
   - `domain`
   - `contracts`
   - `runtime`
   - `service`
   - `store`
   - `interaction_web`
   - `channel`
   - `capability`
   - `model`
2. 按 `SYSTEM-BOUNDARIES / 2. 模块依赖方向` 分类所有逆向依赖；
3. 为每个违规记录：调用方、被调用方、原因、目标 Port/Facade、迁移批次；
4. 区分真实违规与装配根中的合法依赖；
5. 固化允许依赖清单，禁止使用不断扩大的全局忽略列表。

#### 交付物

- `docs/architecture/current-dependency-map.md`；
- `tests/architecture/test_dependency_rules.py`；
- 架构违规基线清单和清零计划。

#### 验收

- Domain 导入数据库、HTTP、模型 SDK、Channel SDK 时测试失败；
- Interaction handler 直接导入 SQLite 写接口时测试失败；
- Plugin 代码导入 Core 私有模块时测试失败；
- 装配根允许依赖具体 Adapter，但业务模块不允许。

### M2：定义模块公开面

#### 工作项

1. 为下列状态所有者定义稳定 Facade/Protocol：
   - Conversation/Session → IdentityConversationService；
   - Turn/RunAttempt → TurnService；
   - Task/TaskAttempt → TaskService；
   - MemoryItem → MemoryService；
   - Delivery → DeliveryService；
   - Approval → ApprovalService；
   - Plugin 状态 → PluginRuntime；
2. 将跨模块输入输出放入 `contracts`、`commands`、`queries`、`ports` 或公开 Facade；
3. 禁止公开 SQLite Row、ORM Entity、SDK Event、内部缓存和锁；
4. 为 Facade 添加类型契约与最小示例；
5. 为现有调用建立兼容适配层，分批迁移，避免一次性大重构。

#### 建议代码布局

```text
src/cogito/
  contracts/             跨进程 Envelope 与公共值对象
  application/
    commands/            修改状态的意图
    queries/             只读查询
    ports/               外部依赖接口
  domain/                纯领域对象与规则
  service/               聚合所有者 Facade/实现
  store/                 Repository 具体实现
```

#### 验收

- 每类聚合状态能指出唯一公开写入口；
- Dashboard 写操作只调用 Command API；
- Event Consumer 修改其他聚合时必须提交 Command；
- Runtime 只能通过 Facade 提交 Turn/Attempt 结果。

### M3：收敛领域契约模型

#### 工作项

1. 对照 `DOMAIN-CONTRACTS / 2. 数据契约与接口架构` 建立契约清单；
2. 统一实现并导出：
   - TraceContext；
   - ChannelEnvelope；
   - ContentPart；
   - AgentRequest/AgentReply；
   - ToolRequest/ToolResult；
   - CommandEnvelope；
   - EventEnvelope；
   - ErrorEnvelope；
3. 所有跨进程模型包含 `schema_version`、ID、时间、origin、TraceContext；
4. 大型内容统一使用 inline 或 `payload_ref + sha256 + content_type + size`；
5. 禁止将平台 SDK 对象、Exception 原文和数据库 Row 放入公开契约；
6. 明确字段缺省、可空性、时间格式、枚举未知值策略；
7. 当前版本和前一版本都提供解码器；破坏性变化增加版本而不是修改旧语义。

#### 测试

- JSON round-trip；
- 当前版本与前一版本兼容；
- 未知可选字段忽略或保留策略；
- 缺失必填字段返回安全 ErrorEnvelope；
- Provider 原始错误不会泄露 Secret；
- Channel SDK 对象无法序列化进入 Agent Core。

### M4：实现不可变 Envelope 与受控 Patch

#### 工作项

1. 公共 Envelope 使用 frozen dataclass 或等价不可变模型；
2. 实现文档定义的 Patch 类型：SetField、RemoveField、AppendContentPart、AppendPromptSection、AddMetadata、AddTag、Reject、RequireApproval；
3. 建立受保护字段表：trace_id、principal_id、conversation_id、turn_id、attempt_id、origin、reply_route、schema_version、idempotency_key；
4. Patch 应用器执行 Schema 校验、权限校验和差异记录；
5. Middleware 只能返回 Patch，不能原地修改共享对象；
6. 记录 Patch 来源、执行顺序和拒绝原因。

#### 验收

- 修改受保护字段必定失败并产生 Audit；
- 相同 Patch 序列得到确定结果；
- Patch 失败不留下半修改状态；
- Diff 默认脱敏。

### M5：把全局不变量变成验证矩阵

#### 工作项

为 `GLOBAL-INVARIANTS / 1～6` 的每一条建立记录：

```text
invariant_id
doc_id / heading path
enforcement_type
enforcement_location
test_id
manual_runbook
current_status
```

优先自动化以下高风险不变量：

1. SQLite 是唯一事务事实源；
2. 一个 Context Partition 最多一个可提交 Turn；
3. 外部调用不发生在数据库事务内；
4. 旧 Lease/旧 Attempt 不得提交；
5. 入站、Command、Task、Event、Delivery 均有幂等键；
6. 副作用先持久化意图，后保存 Receipt；
7. unknown 结果先 reconcile；
8. 外部内容不能提升 Trust Label；
9. 等待状态不持有事务、Lane 或 Lease；
10. 控制面默认仅 loopback。

#### 测试类型

- 数据库唯一约束测试；
- 条件更新与并发竞争测试；
- 架构依赖测试；
- 契约 Fixture；
- 崩溃恢复测试；
- 配置跨字段约束测试。

### M6：架构门禁和变更流程

#### 工作项

1. CI 顺序固定为：依赖规则 → 契约 → 单元 → 集成 → 恢复；
2. 架构测试不允许 `xfail` 掩盖新增违规；
3. 新增跨模块依赖必须说明公开面和状态所有者；
4. 破坏性契约变化必须附 ADR、迁移窗口、回滚和双版本测试；
5. 更新 `manifest.json` 之外的设计时检查 doc_id、depends_on 和 related_docs；
6. PR 模板增加不变量影响、契约影响、Migration、回滚、测试 ID。

#### 验收

- 任意新增 Domain→Infrastructure 导入会阻断合并；
- 任意删除契约字段但未提升版本会阻断合并；
- 任意新增状态写路径但未声明所有者会阻断合并；
- 架构例外必须有带到期时间的 ADR。

## 6. 数据与迁移影响

本计划原则上不修改业务表语义。若为契约版本、Patch Audit 或不变量证据增加表，应遵循：

1. 先增加可空字段或新表；
2. 双写或回填；
3. 读取切换；
4. 验证旧版本兼容；
5. 后续版本再收缩旧结构。

禁止借架构整理直接重命名大量已有列。物理 Schema 变更必须回查 `DATABASE-SCHEMA / 8. Migration`。

## 7. 测试矩阵

| 测试层 | 必测内容 |
|---|---|
| architecture | 禁止导入、公开面、Plugin 私有访问 |
| contract | Envelope 版本、序列化、错误脱敏 |
| domain | 聚合不变量、状态转移、不可变语义 |
| integration | Facade→Repository、UoW、Outbox |
| concurrency | 版本冲突、Lease、Lane 串行 |
| recovery | 旧 Attempt 提交、unknown reconcile |
| security | Trust Label、受保护 Patch、loopback |

## 8. 风险与控制

- **风险：过度抽象。** 只有跨模块或需要替换的边界引入 Protocol；模块内部不强制接口化。
- **风险：大规模搬文件造成回归。** 先增加 Facade 和兼容层，再迁移调用方，最后删除旧入口。
- **风险：契约重复。** 每个跨进程对象只保留一个规范模型，Adapter 做转换。
- **风险：架构测试过度脆弱。** 检查依赖方向和公开面，不检查无意义的目录细节。
- **风险：文档和代码再次漂移。** 不变量矩阵和契约 Fixture 纳入发布门禁。

## 9. 完成定义

满足以下条件才可标记完成：

1. `SYSTEM-BOUNDARIES / 2` 的禁止依赖全部有自动检查；
2. `SYSTEM-BOUNDARIES / 4` 的每个状态拥有唯一写入 Facade；
3. `DOMAIN-CONTRACTS / 2` 的核心 Envelope 有代码模型和版本测试；
4. `GLOBAL-INVARIANTS / 1～6` 每条都有执行位置和测试/演练；
5. 新增架构违规能够在 CI 中稳定失败；
6. 当前测试、Ruff、compileall 全绿；
7. 相关文档使用 `doc_id / heading path` 更新，并记录任何设计偏差。

## 10. 建议拆分 PR

1. PR-A1：依赖扫描与 architecture tests；
2. PR-A2：Turn/Task/Delivery/Memory Facade 公开面；
3. PR-A3：统一 Envelope 与版本兼容测试；
4. PR-A4：不可变 Patch 与受保护字段；
5. PR-A5：全局不变量矩阵及高风险并发测试；
6. PR-A6：CI 门禁、PR 模板和文档同步。
