# PLAN-13：记忆系统治理与内容记忆增强开发计划

## 1. 计划结论

Cogito 已具备可运行的短期上下文、长期 `MemoryItem`、候选提取、确认/拒绝、FTS/Embedding 检索、Session Summary、Context Snapshot 和 Markdown 派生视图。本计划不重写现有记忆系统，也不将 `reference/memU` 直接引入生产依赖，而是完成两类工作：

1. **先收敛事实记忆治理**：精确来源、确认策略、强化、衰减、遗忘、删除和文档/代码一致性；
2. **再新增内容型记忆平面**：用于文档、代码、长日志、网页和多模态派生文本的 Resource–Document–Segment 分层检索，同时保持 `MemoryItem` 只负责事实、偏好、经历、目标和约束。

最终形成三层记忆架构：

```text
短期运行时层
  Message + SessionSummary + Checkpoint + ContextSnapshot

事实记忆层
  MemoryItem + MemorySource + MemoryRelation + Embedding/FTS

内容记忆层
  KnowledgeResource + KnowledgeDocument + KnowledgeSegment + Provenance
```

写入权保持分离：Session/Message 服务写短期对象，MemoryService 写事实记忆，KnowledgeService 写内容记忆，Capability/Skill 子系统写 Skill，Context Builder 只读并生成不可变 Snapshot。

执行顺序：

```text
M0 基线、规范勘误与决策冻结
→ M1 精确来源与事实记忆写入闭环
→ M2 强化、衰减、归档与遗忘统一
→ M3 策略化短期→长期晋升
→ M4 Resource–Document–Segment 内容层
→ M5 增量同步、来源变更与删除级联
→ M6 统一检索与 ContextSnapshot 装配
→ M7 Markdown 视图、审计与用户控制面
→ M8 评测、恢复、性能与发布门禁
→ M9 可选 memU 隔离 PoC 与是否采用决策
```

M0～M3 是 P0/P1 治理收敛，不应被内容层开发阻塞；M4～M8 是内容记忆产品化；M9 是可选实验，不属于发布必需路径。

## 2. 权威设计依据

计划实现与评审必须使用 `doc_id / heading path` 引用：

- `DOMAIN-CONTRACTS / 1.3.5 Memory Aggregate`：MemoryItem 是独立事实对象；合并、冲突和替代由关系表达。
- `DOMAIN-CONTRACTS / 1.7 Session`：不同 Session 不共享短期消息、摘要或 Snapshot。
- `DOMAIN-CONTRACTS / 1.13 MemoryItem`：长期记忆必须保留来源。
- `DOMAIN-CONTRACTS / 2.8 Command 契约`：用户确认、拒绝、纠正、删除必须通过意图型 Command，而不是直接改表。
- `RUNTIME-FLOWS / 即时 Turn 流程`：长期提取不得阻塞当前回复。
- `RUNTIME-FLOWS / 事务边界`：源事实提交与派生工作通过事务或幂等 Event 衔接。
- `SESSION-CONTEXT / 3. 短期上下文`：Recent Messages、Conversation Summary、Checkpoint 与临时事实只属于当前 Session。
- `SESSION-CONTEXT / 4. 长期认知共享`：长期召回必须经过 Principal、Scope、Trust 和 Policy。
- `RETRIEVAL-CONTEXT / 1. 两条检索路径`：隐式 Context Builder 与显式 recall 工具共享检索管道。
- `RETRIEVAL-CONTEXT / 4. 检索源`：短期、Memory、Goal、历史 Session 和授权外部上下文是不同来源。
- `RETRIEVAL-CONTEXT / 10. Context Snapshot`：最终选择结果必须不可变、可解释、可回放。
- `MEMORY-LIFECYCLE / 4. 检索权重算法`：权重算法必须版本化并与实现一致。
- `MEMORY-LIFECYCLE / 5.3 Reinforcement 规则`：普通展示与明确正向信号必须分开。
- `MEMORY-LIFECYCLE / 7. 写入流程`：候选去重、冲突分类、确认和 Embedding 分步完成。
- `MEMORY-LIFECYCLE / 10. Consolidation`：后台维护不能静默升级 candidate。
- `MEMORY-LIFECYCLE / 14. 纠正与删除`：删除后不得因索引重建重新出现。
- `MESSAGE-PERSISTENCE / 消息历史与 Summary`：Summary 是短期压缩，不直接等同长期事实。
- `TASK-SCHEDULER / Task 状态与幂等`：跨重启提取、索引、同步和删除必须是持久 Task。
- `EVENT-OUTBOX / Outbox 事务模式`：源提交与派生事件不能形成双写窗口。
- `CONNECTOR-INGESTION / 原始数据与标准化`：外部资源保留 RawItem/来源，不直接写 Memory 表。
- `CAPABILITY-PLUGINS / Skill 所有权`：可执行 Skill 不属于 Memory 聚合。
- `DATABASE-SCHEMA / Migration 策略`：Schema 变更执行 expand → backfill → switch → contract。
- `SECURITY-OBS / 数据最小化与审计`：Secret、敏感信息和跨 Principal 数据不得进入非授权索引。
- `TEST-EVALUATION / 发布门禁`：迁移、恢复、回放、隔离和离线评测必须可重复。

## 4. 冻结的产品与架构决策

### 4.1 MemoryItem 与内容记忆分轨

`MemoryItem` 继续只表达：

```text
fact | preference | episode | goal | constraint
```

内容记忆使用独立聚合，不新增 `knowledge_document` 等 Memory kind。文档主题、代码说明和媒体 caption 属于 `KnowledgeDocument/Segment`；从内容中抽取出的 Owner 事实仍必须走 MemoryService candidate/confirm 流程。

### 4.2 权威数据与派生数据

```text
权威：Message、Event、RawItem、KnowledgeResource、MemoryItem、关系、tombstone
派生：Summary、Document 正文、Segment、Embedding、FTS、Markdown view
```

派生数据必须带生成版本和源版本，可删除重建；不得反向覆盖权威源。

### 4.3 强化语义

统一为：

- `exposed`：进入 Context 或工具结果；只增加 exposure/retrieval count；
- `referenced`：模型在最终结果中引用但用户未确认；记录 usage，不增加 reinforcement；
- `user_affirmed`：用户明确确认有用；reinforcement +2；
- `task_succeeded`：成功 Task 有可验证依赖；reinforcement +1；
- `user_corrected`：纠正并确认新事实；新事实 +2，旧事实不强化；
- `negative_feedback`：可降低 emotional/quality 信号，不产生负 reinforcement 溢出。

主动调用 `recall_memory` 本身不是正向信号。

### 4.4 内容同步与 Skill 边界

- Connector/Workspace Scanner 只产生 RawItem/Knowledge ingest Task；
- KnowledgeService 是 Knowledge 聚合唯一写入者；
- Skill 候选只能交给 Capability/Skill 服务，默认不可执行且必须保留来源；
- MemoryService 不写 Plugin/Skill 表；
- 所有外部内容先经过 Trust/Policy，不能自动成为 Owner confirmed fact。

### 4.5 memU 使用边界

PLAN-13 主路径只借鉴设计，不依赖 `reference/memU`。若执行 M9：

- 通过 Adapter/PoC 进程隔离；
- 使用临时数据库和去敏数据集；
- 产物只作为 `authorized external context`；
- 不允许直接写 `memory_items`、Session、Task 或 Skill；
- PoC 失败不影响 M0～M8 发布。

## 5. 目标数据模型

### 5.1 MemorySource

新增一对多来源表，替代“单个模糊 source_id 足以表达全部来源”的假设：

```text
memory_sources
  memory_source_id
  memory_id
  source_type: message | event | task | connector_item | knowledge_resource | manual
  source_id
  source_revision
  source_sequence
  evidence_ref
  evidence_hash
  trust_label
  extraction_id
  created_at
  deleted_at
```

约束：

- UNIQUE(memory_id, source_type, source_id, source_revision, evidence_hash)；
- `source_id` 必须指向稳定 ID，不写显示名或路径作为唯一身份；
- `evidence_ref` 指向最小必要证据，不能复制 Secret；
- 原 `memory_items.source_type/source_id` 在过渡期保留 dual-read，最终降级为 primary source cache 或废弃字段；
- 一个 consolidation 产物可以关联多个来源。

### 5.2 MemorySignal

强化、展示和反馈使用追加式事件表：

```text
memory_signals
  signal_id
  memory_id
  signal_type
  signal_value
  actor_principal_id
  turn_id
  task_id
  idempotency_key
  algorithm_version
  occurred_at
  metadata_json
```

`memory_items.reinforcement/exposure_count/retrieval_count` 是可重建投影或缓存，不是唯一事实。

### 5.3 Knowledge 聚合

```text
knowledge_resources

  connector_id
  source_uri_hash
  source_kind
  media_type
  payload_ref

knowledge_documents
  document_id
  resource_id
  title
  normalized_text_ref
  summary
  language
  parser_id/parser_version
  content_version
  status
  created_at/updated_at

knowledge_segments
  segment_id
  document_id
  ordinal
  segment_kind
  text_ref_or_inline
  content_hash

  heading_path

  embedding_status

knowledge_embeddings
  segment_id
```

在 `contracts/context.py` 或相邻 contracts 模块定义只读结果：

```text
RetrievalCandidate

  scope
  content_ref
  source_refs

  recency_score
  importance_score
  trust_score
  final_score
  token_estimate
  retrieval_path
```

Store Entity 不得直接作为跨层返回类型。

## 8. M1：精确来源与事实记忆写入闭环

### 8.1 目标

任何 MemoryItem 都能回答“来自什么、哪个版本、哪段证据、由哪个提取任务生成”，并能在来源删除或修订时安全重评。

### 8.2 实现步骤

1. expand migration 新增 `memory_sources`、索引和外键；
2. 新增 `MemorySource` 领域值对象和 Repository；
3. `MemoryExtractor` 输入从 `{role, content}` 升级为含 message_id、revision、sequence、trust 的不可变 DTO；
4. 提取 Schema 要求每个 candidate 返回 `evidence_message_ids` 或 evidence index；
5. 服务端验证 evidence 必须属于当前 principal/session/input window，模型不能伪造来源；
6. `SqliteMemoryService.propose()` 在同一 UnitOfWork 写 MemoryItem、MemorySource、relation 和 Outbox；
7. 手工 `remember_memory` 生成 `source_type=manual`，记录 Command/Turn；
8. Task/Connector 派生必须携带原 Task/Event/RawItem ID；
9. backfill 旧条目：
   - 可解析 source_id 的建立真实来源；
   - `auto_extract` 等不可还原项标记 `legacy_unresolved`，不得伪造 Message；
10. dual-read：新来源表优先，旧字段仅兼容；
11. Query/API/Markdown view 展示去敏来源摘要；
12. 删除源消息时不立即删除 confirmed Memory，而是产生 `MemorySourceInvalidated` 并按策略重评。

### 8.3 测试

- 一条候选关联一条/多条 Message；
- 模型返回窗口外 ID 被拒绝；
- 重复 extraction Event 不重复来源；
- edited Message 新 revision 不覆盖旧 evidence；
- 删除一个来源但仍有其他有效来源时 Memory 保持有效；
- 唯一来源删除后进入 review/stale，而不是静默 confirmed；
- 群聊发送者与 Owner 来源不混淆；
- legacy backfill 可重复执行。

### 8.4 验收

- 新写入 Memory 的可追踪率 100%；
- `source_id=auto_extract` 新增量为 0；
- 任意 Memory 可通过 Query API 返回至少一个安全 SourceRef；
- Migration 升级、回滚限制和备份要求文档齐全。

## 9. M2：强化、衰减、归档与遗忘统一

### 9.1 目标

让检索权重可解释、可重算、可回放，并确保误召回不会自动自我强化。

### 9.2 实现步骤

1. 新增 `memory_signals` 追加表；
2. 所有展示、引用、用户确认、Task 成功和负反馈写幂等 Signal；
3. 实现纯函数 `compute_retrieval_weight(memory, signals, now, policy)`；
4. 固定 `MemoryWeightPolicy` 版本，配置包含：
   - kind decay rate；
   - confirmation score；
   - reinforcement cap；
   - emotional/quality cap；
   - archive/forget threshold；
   - minimum confirmed floor（若保留）；
5. `memory.recompute_weight` Task 批量 checkpoint，使用注入 Clock；
6. `memory_items.retrieval_weight` 作为缓存写回，并记录 `last_weight_update`、algorithm version；
7. 区分：
   - searchable；
   - archived（默认不召回）；
   - forgetting_candidate（等待用户确认）；
   - expired/deleted；
8. candidate TTL 使用独立策略，不复用 confirmed decay；
9. 删除旧 `apply_decay()` 中与规范冲突的隐式乘法路径，或仅保留兼容 shim；
10. 增加 Explain API，返回权重分项而非只返回最终分数。

### 9.3 数值与属性测试

- 同输入、同 now、同 policy 输出稳定；
- 时间单调增加时，无新 Signal 的权重不增加；
- exposure/retrieval Signal 不改变 reinforcement；
- user_affirmed 比单纯 referenced 权重更高；
- active goal 不衰减，completed goal 按策略衰减；
- constraint 不因时间自动失效；
- 边界值 clamp 正确，无 NaN/负数；
- 重算两次结果一致；
- 时区/DST 不影响按 UTC 计算。

### 9.4 验收

- 文档公式、配置和代码使用同一参数名；
- 旧测试与新增固定时钟测试全绿；
- 随机抽样条目可解释到每个权重分项；
- 任何普通 recall 不产生 reinforcement 增量。

## 10. M3：策略化短期记忆晋升

### 10.1 目标

从固定“至少 4 条消息”升级为可配置、幂等、低成本的晋升策略，同时继续保证提取失败不阻塞回复。

### 10.2 Trigger Policy

支持：

```text
explicit_remember       用户明确要求记住，立即排高优先级 Task
turn_completed          达到消息/token/时间增量阈值
session_closed/reset    会话结束或 reset 前收尾提取
decision_detected       当前 Turn 产生重要决策候选
goal_or_constraint      检测到目标/约束候选
idle_consolidation      后台空闲批处理
task_completed          成功 Task 产生可复用 Episode/事实
```

Trigger 只决定是否提交 extraction Task，不直接确认事实。

### 10.3 Watermark

每个 Session/Principal 记录：

```text
memory_extraction_watermark
  session_id
  principal_id
  last_message_sequence
  last_summary_version
  extractor_version
  last_task_id
  updated_at
```

窗口必须：

- 不漏消息；
- 允许有限 overlap；
- 通过 evidence/source 唯一约束去重；
- Session reset 后旧、新 generation 分开；
- Task 成功提交后才推进 watermark。

### 10.4 成本与安全

- 先执行确定性门控，再调用模型；
- explicit remember 可用专用小上下文；
- Secret/凭证模式由 Policy 过滤，但不依赖正则作为唯一判断；
- 群聊只提取当前发送者明确陈述，默认不把他人陈述归为 Owner；
- 模型超时/失败重试有限，超过预算进入 dead-letter/audit；
- 提取模型、prompt、schema 和 policy 版本写 Trace。

### 10.5 验收

- 短会话明确“记住”可立即生成 confirmed/candidate；
- 长会话按 watermark 分窗且无重复；
- reset/close 后完成最后窗口；
- 失败不推进 watermark；
- 同一 Trigger/Event 重放不重复调用或写入；
- 模型调用量相对每 Turn 无条件提取有可测下降。

## 11. M4：Resource–Document–Segment 内容记忆层

### 11.1 目标

为文档、代码、网页、长日志和多模态派生文本建立独立、可追踪、可重建的内容记忆层，不污染 MemoryItem。

### 11.2 首版输入范围

按风险顺序：

1. 已授权本地 text/markdown/code 文件；
2. 已有 Connector RawItem 正文；
3. PDF/Office 的现有解析结果；
4. VisionAnalysis/Audio transcript 等现有派生文本；
5. URL 抓取内容（必须复用 Safe HTTP/Connector Policy）。

首版不扫描未显式授权目录，不新增任意网络抓取器。

### 11.3 服务边界

```python
class KnowledgeService(Protocol):
    async def register_resource(request) -> ResourceRef: ...
    async def ingest(resource_id, expected_version) -> IngestResult: ...
    async def invalidate(resource_id, reason, expected_version) -> None: ...
    async def erase(resource_id, expected_version) -> EraseResult: ...
    async def retrieve(query) -> KnowledgeResult: ...
```

要求：

- 其他模块不能直接 CRUD knowledge 表；
- Payload Store 保存二进制/大文本，数据库保存引用和 hash；
- parser/segmenter/embedding 都是可替换 Port；
- 每个阶段可重试且输出版本化；
- Embedding 不可用时仍可用 FTS。

### 11.4 Segment 策略

- 优先按 heading、段落、代码符号和结构边界切分；
- token 上限和 overlap 配置化；
- Segment 保存 heading path、offset、content hash；
- 同内容 hash + segmenter version 不重复生成；
- 代码与自然语言可使用不同 segment_kind，但共享候选协议；
- 不在 Segment 正文中复制凭证或被 Policy 排除字段。

### 11.5 测试

- Markdown heading 切分；
- 中文长文切分；
- 代码文件符号/行范围来源；
- 空文件、超大文件、二进制误判；
- parser 失败与重试；
- Payload 缺失；
- FTS-only 降级；
- Embedding version 切换隔离；
- Principal/Scope 越权查询返回 0。

### 11.6 验收

- 输入文件可追踪到 Resource→Document→Segment；
- 修改 parser/segmenter 可从 Resource 重建；
- Knowledge 表没有第二份 Message/Memory 状态机；
- 未授权资源不能进入索引。

## 12. M5：增量同步、来源变更与删除级联

### 12.1 目标

借鉴 memU 的增量同步思想，但通过 Cogito Connector + Task + Outbox 实现，避免旁路写库。

### 12.2 同步模型

每个 Connector/source root 使用：

```text
stable_source_id
source_version
content_hash
observed_at
watermark
status
```

Diff 分类：added、modified、unchanged、deleted、renamed/unknown。路径不是稳定 ID 时，使用 Connector 提供 ID；本地文件可组合 file identity + root scope，rename 检测只作优化，不影响正确性。

### 12.3 级联规则

```text
source modified
→ 旧 Resource 标 stale
→ 新版本 ingest
→ 新 Document/Segment active
→ Snapshot 以后只召回新版本
→ 与旧 Resource 关联的 MemorySource 进入 review

source deleted
→ Resource deleted/tombstone
→ Segment 从默认检索撤销
→ Embedding/FTS 清理
→ MemorySourceInvalidated
→ confirmed Memory 按剩余来源与 policy 决定 keep/review/expire
→ Markdown view 重建
```

历史 ContextSnapshot 保持不可变，只保留安全 content_ref/tombstone 解释；不得回写改变过去 Snapshot。

### 12.4 删除等级

- `invalidate`：停止默认召回，保留数据；
- `soft_delete`：撤销正常访问，保留审计 tombstone；
- `erase`：按用户请求清理 Payload/Embedding/正文，保留最小不可逆审计元数据；
- `retention_expire`：按 Retention Policy 自动处理。

### 12.5 验收

- unchanged 不重新 parse/embed；
- modified 只替换受影响版本；
- deleted 后 FTS/vector/Markdown 均不可命中正文；
- rebuild index 不复活 tombstone；
- crash 在任意阶段可通过 Task/Outbox 收敛；
- 同一删除事件重放幂等。

## 13. M6：统一检索与 ContextSnapshot 装配

### 13.1 目标

把 Recent Message、SessionSummary、MemoryItem、Goal、KnowledgeSegment 和 Task state 统一映射为 `RetrievalCandidate`，但保留各源硬过滤和独立召回实现。

### 13.2 检索流水线

```text
ContextQuery
→ Query Plan / optional rewrite
→ per-source authorization filter
→ parallel source recall
   ├── recent messages
   ├── session summary
   ├── confirmed memory/goals
   ├── knowledge segments
   └── task/checkpoint
→ score normalization by source
→ dedupe / supersedes / version filtering
→ diversity and source quotas
→ token budgeting
→ immutable ContextSnapshot
```

### 13.3 评分

禁止直接把不同分布的 BM25、cosine 和规则分数相加。每个源先输出标准化分项，再由版本化 policy 合成：

```text
memory: relevance + retrieval_weight + trust + goal relevance
knowledge: keyword/semantic + source trust + freshness + document diversity
summary: session match + recency + coverage
recent message: sequence/recency + current turn relevance
```

硬过滤永远先于软打分：Principal、Scope、status、validity、Trust、source version、deleted/stale。

### 13.4 Token Budget

初始默认值作为可评测配置，而非永久常量：

```text
recent messages      30%
session summary      10%
fact memory          20%
goals/constraints    10%
knowledge segments   20%
task/tool state      10%
```

当前输入和 System Policy 另行保留。未使用额度可以转移，knowledge 不能挤掉 active constraint 和当前 Task state。

### 13.5 显式工具

- `recall_memory` 只查事实记忆/goal/history；
- 新增或扩展 `search_knowledge` 查 Resource/Document/Segment；
- 不把两者合成一个含糊工具；
- 隐式 Context Builder 可同时调用两路；
- 工具结果必须返回来源、版本、score、retrieval path 和安全摘要。

### 13.6 Snapshot

每个选中项记录：candidate type/id、content ref、source refs、source version、score 分项、tokens、trust、retrieval path、policy version。排除摘要至少统计：unauthorized、stale、superseded、low score、token budget、duplicate。

### 13.7 验收

- 不同 Session 不召回近期消息；
- 同 Principal 可按 Scope 召回长期 Memory/Knowledge；
- 群聊非 Owner 不泄漏 Owner global 内容；
- stale/deleted Segment 不进入 Snapshot；
- Snapshot 可重放解释选中与排除；
- FTS 或 Embedding 任一路失败时降级可用；
- 相同数据版本和 policy 得到稳定排序。

## 14. M7：Markdown 视图、审计与用户控制面

### 14.1 目标

让用户能看见、确认、纠正、删除和理解记忆，同时保持 Markdown 为派生视图。

### 14.2 视图

继续维护：

- `MEMORY.md`：confirmed 且高权重事实/偏好；
- `PENDING.md`：candidate；
- `HISTORY.md`：Episode；
- `RECENT_CONTEXT.md`：短期压缩；

可新增只读：

- `KNOWLEDGE.md`：Resource/Document 索引和 freshness；
- `knowledge/<resource-or-topic>.md`：首版仅在确有人工浏览需求时生成，不作为检索必需依赖。

### 14.3 Command/Query

Command：

```text

RejectMemory
CorrectMemory
ForgetMemory
EraseMemory
RegisterKnowledgeSource
RefreshKnowledgeSource

```

Query：

```text

```

全部通过 service/api；Interaction Web 和 Tool 不直接访问 Repository。

### 14.4 审计

记录：actor、command id、expected version、before/after 状态、source invalidation reason、erase receipt、algorithm/policy version。原始敏感正文不写 Audit。

### 14.5 验收

- 用户确认/拒绝/纠正后下一个新 Snapshot 生效；
- 删除与 erase 有明确差异和 Receipt；
- Markdown 删除后重建不会复活；

### 15.3 恢复场景

- extraction Task 执行前/写一半/commit 后崩溃；
- parse 完成但 segment 未完成；
- segment 完成但 embedding 失败；

- SQLite 查询使用必要复合索引，避免每 Turn 全表扫描；
- Embedding/parse 在后台 Task，不阻塞回复；
- token budget 永不超过模型上下文限制；
- 任何降级路径有 Trace/metric。

## 16. M9：可选 memU 隔离 PoC

### 16.1 目的

只验证 memU 在内容编译/Segment 检索方面是否比 Cogito M4～M6 的原生实现带来净收益，不验证替换事实记忆层。

### 16.2 数据集

使用去敏、可提交或合成数据：

- 中文多轮对话；
- Markdown/PDF 项目文档；
- Python 代码；
- 长日志；
- 矛盾偏好和删除场景。

不得使用真实 Secret、私聊全文或未经授权的用户文件。

### 16.3 对比

同一 query 集比较：

- recall@k/MRR；
- source traceability；
- stale/delete correctness；
- ingest latency/cost；
- retrieval latency；
- token output；
- crash recovery；
- 依赖体积和升级风险。

### 16.4 决策门槛

只有同时满足以下条件才进入正式 Adapter 设计：

- 关键检索指标有显著、稳定提升；
- 不绕过 Principal/Scope/Policy；
- 能以派生索引身份重建；
- 删除和来源版本语义可映射；
- 依赖与 API 稳定性风险可接受；
- 本地原生实现维护成本确实更高。

否则记录“不采用”决策并结束 PoC，不维持双系统。

## 17. 数据库 Migration 计划

### 17.1 建议切片

```text
Migration A  memory_sources + indexes
Migration B  memory_signals + algorithm_version caches
Migration C  extraction_watermarks
Migration D  knowledge_resources/documents/segments
Migration E  knowledge_embeddings + FTS
Migration F  optional legacy source fields contract cleanup
```

### 17.2 规则

- 每个 Migration 独立 meta，声明 online_safe、requires_backup、rollback；
- 大表 backfill 使用 checkpoint Task，不在启动事务中长时间锁库；
- 先 expand 和 dual-write，再切读，最后才 contract；
- FTS/Embedding 为派生数据，Migration 失败可重建；
- 权威来源和 tombstone 不允许通过 down migration 静默丢失；
- Schema version 与 Event/Contract version 分离。

### 17.3 启动兼容

升级期间支持：

- 旧行无 `memory_sources` 时读取 legacy source；
- 新写入强制 dual-write；
- backfill 完成后切换新读路径；
- contract migration 前至少保留一个发布周期；
- 恢复旧备份后能够重新运行 backfill/index Task。

## 22. 风险与缓解

| 风险 | 缓解 |
|---|---|
| Schema 扩张导致 SQLite 写锁 | 后台批处理、短事务、WAL、分批 checkpoint、索引评审 |
| 新内容层增加检索延迟 | 并行召回、per-source timeout、缓存 query embedding、严格 top_k |
| LLM 提取错误进入 confirmed | 明确陈述仍做 evidence/Policy 校验；敏感/高风险保持 candidate |
| 来源删除误删有效事实 | 多来源计数；先 invalid/review，明确 erase 才物理清理 |
| 双路检索分数不可比 | per-source normalization + 离线校准，不直接混加 raw score |
| Embedding 成本和版本漂移 | 异步增量索引、内容 hash、版本隔离、FTS 降级 |
| Markdown 与 DB 不一致 | DB 权威、view generation/version/freshness、可重建 |
| Skill 学习越权 | Skill 归 Capability，默认不可执行、需要审批和来源 |
| memU 重构造成依赖风险 | 主路径不依赖；仅 M9 隔离 PoC |
| 删除后历史 Snapshot 暴露 | Snapshot 保存安全引用/tombstone；访问时继续做授权与 retention 检查 |

## 23. 回滚策略

### 23.1 M1～M3

- Feature flag 关闭新 trigger/signal 算法；
- dual-read 回退 legacy source/weight；
- 不删除新表；
- 已写新 Source/Signal 保留，等待修复后重放。

### 23.2 M4～M7

- `knowledge.enabled=false` 停止新 ingest 和上下文召回；
- 保留 Resource/tombstone，取消 Knowledge 在 Query Plan 中的 source；
- FTS/Embedding 可删除重建；
- 不影响现有 Memory/Session 路径。

### 23.3 Migration

- 发布前备份；
- 权威数据 Migration 不依赖破坏性 down；
- contract migration 单独发布且需人工门禁；
- 恢复后运行 reconcile/backfill/rebuild，而不是复制派生文件覆盖 DB。

## 24. 完成定义

PLAN-13 只有同时满足以下条件才能标记 completed：

1. 新 Memory 写入 100% 有可验证 SourceRef；
2. reinforcement/decay 文档、代码、配置和测试只有一套语义；
3. 普通隐式或显式 recall 不会自动强化；
4. extraction 使用持久 watermark/Task，失败不漏消息、不阻塞回复；
5. Knowledge Resource–Document–Segment 可从源重建并有唯一服务写入口；
6. source add/modify/delete 全链路幂等，删除后索引重建不复活；
7. ContextSnapshot 能解释 Memory、Knowledge、Summary 等所有选中项；
8. Principal/Session/Scope 隔离测试无泄漏；
9. FTS/Embedding/Parser 任一路故障有受测降级；
10. Migration、备份恢复、Worker crash、SQLite busy 演练通过；
11. 离线评测和性能指标达到 M0 冻结的阈值；
12. Markdown/API/工具均不绕过数据库服务；
13. manifest、权威文档、Schema、代码和测试状态一致；
14. 没有未登记的架构例外或 TODO 占位；
15. M9 如执行，产生明确 adopt/reject 决策，不留下未治理的双系统。
