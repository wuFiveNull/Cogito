# PLAN-17：PLAN-14 Proactive 与 Drift 生产真实性收口修复计划

## 1. 结论

PLAN-14 当前不能按 `completed` 验收。代码已创建大量 Proactive/Drift 组件和测试，但生产纵向链存在关键断点：

```text
Scheduler admission
  → drift.run Task
  → DriftRun(skill_name="(selected-at-run)")
  → Runner 查不到 manifest
  → completed / unknown skill
```

因此，当前状态应描述为：

| 口径 | 估算完成度 | 说明 |
|---|---:|---|
| 文件和组件存在度 | 约 70% | Domain、Repo、Service、Migration、Query、测试文件大多存在 |
| Proactive 生产闭环 | 约 55%～60% | dry-run、Presence、Cadence 已部分接通；Context、Alert immediate、Enrichment 未完成 |
| Drift 组件实现 | 约 60% | Admission、Catalog、Selector、Runner、Preemption、Projection 均有代码 |
| Drift 生产可用度 | 约 15%～25% | Skill 选择、Task 生命周期、Checkpoint、Resume、抢占、投影均未真实接通 |
| PLAN-14 总体验收 | 约 35%～40% | 聚焦测试绿，但真实生产路径和发布门禁不成立 |

本计划不重写现有架构，而是删除占位和假闭环，完成一条可实际运行、可崩溃恢复、可抢占、可审计的最小生产纵切面。

目标执行顺序：

```text
R0 事实状态与架构边界纠正
→ R1 Skill 选择与 Admission 原子化
→ R2 Task Handler 结构化结果与生命周期收口
→ R3 Task Checkpoint 持久化与真实 Resume
→ R4 入站抢占、Lease、Attempt 与预算接线
→ R5 Skill 执行、结果持久化与 Candidate 自动投影
→ R6 Proactive M1/M2 遗留收口
→ R7 Admission、Policy、Dashboard 和安全指标做实
→ R8 真实 E2E、并发、崩溃、覆盖率和灰度门禁
```

R1～R5 是 P0，不完成不得启用 Drift real mode；R6～R7 是 P1；R8 是发布必需门禁。

## 3. 问题清单

## 3.1 P0：阻断生产运行

### DR-P0-01：Skill Selector 未接入生产路径

证据：

- `Scheduler.tick_drift_admit()` 写入 `skill_name="(selected-at-run)"`；
- `DriftRunner._resolve_manifest()` 立即按该字段查内置 Catalog；
- `DriftSkillSelector.select_skill()` 只有测试调用，没有生产调用；
- 结果是所有真实 admission 都被标记为 `completed / unknown skill`。

根因：Catalog、Selector、Scheduler 和 Runner 被分别实现，但没有冻结“何时选择、由谁持久化、如何保证原子性”的契约。

### DR-P0-02：paused/waiting 被 TaskWorker 当作 completed

证据：

- Drift Handler 通过字符串返回 `paused`；
- `TaskWorker.run_once()` 对所有正常返回无条件调用 `dispatcher.complete()`；
- Task 进入 completed 后不会创建新 Attempt 恢复。

根因：TaskHandler 返回类型仍是 `str`，不能表达 completed、paused、waiting、retry、failed、lost 等生命周期语义。

### DR-P0-03：Checkpoint JSON 构造后被丢弃

证据：

- `write_checkpoint()` 构造 `DriftCheckpointV1` JSON；
- 数据库只保存 `drift-check:<run>:<step>` 字符串；
- 返回的 JSON 没有调用方保存；
- `TaskAttempt.checkpoint_ref` 未更新；
- `attempt_id` 实际传空字符串。

根因：Task 虽有 `checkpoint_ref` 字段，但没有完成通用 Task Checkpoint Store 和 Handler 执行上下文接线。

### DR-P0-04：Resume 仍为占位

证据：

- `_resume_drift_run()` 使用 `validate_checkpoint_for_resume("{}")`；
- `_start_drift_run()` 每次从 step 0、空 cursor、零预算开始；
- `resume_from_step`、`resume_completed_actions`、`resume_budget`、`resume_cursor` 未消费；
- 测试允许 `needs_review` 或任意非空字符串作为成功。

根因：为了让测试先绿，恢复路径用引用字符串和弱断言替代真实状态重建。

### DR-P0-05：生产抢占和 Lease 校验未接通

证据：

- `request_preemption()` 只有测试调用，`InboundService` 没有生产调用；
- `TaskHandlerContext` 不包含当前 TaskAttempt 和 Lease checker；
- Runner 默认 `lease_valid=True`；
- step 检查的 `active_normal_turns` 与 `priority_backlog` 使用默认 0。

根因：进程级静态 `TaskHandlerContext` 与每次 claim 的动态执行上下文没有分层。

### DR-P0-06：Drift → Candidate 仅为孤立 Service

证据：

- `DriftProjectionService.project()` 仅在测试中调用；
- Runner 不生成 `DriftCandidateDraft`；
- `_finish_drift()` 接收 `internal_items` 但不保存、不投影；
- E2E 测试手工插入 completed DriftRun 后直接调用 ProjectionService。

根因：结果协议、结果存储与投影触发没有纳入 Task 完成事务或 Outbox Consumer。

## 3.2 P1：行为、安全和治理不完整

### PA-P1-01：Presence 失败反而提高主动性

`PresenceReader` 失败返回 `None`，而 `compute_energy(None)=0`，最终进入 low energy 的 ×1.5 urgency 路径，与“fail-safe 不提高主动性”的注释相反。

### PA-P1-02：Alert immediate trigger 只有注释

Scheduler 注释提到 `schedule_immediate_evaluate`，但源码不存在该函数及 Event Consumer 调用链。Alert 仍可能等待普通 cadence。

### PA-P1-03：Context fallback 未实现

未发现 `ContextSignal`、概率阀、日配额、仅在 alert/content 为空时注入的生产实现。

### PA-P1-04：模型 Enrichment 未实现

PLAN-14 已明确标记 M2 Task C 未完成。当前没有结构化 summary/topic/novelty/relevance/urgency/evidence/confidence 模型增强服务。

### PA-P1-05：语义近重复未实现

目前只有事务幂等，没有 Candidate 与近期主动消息的语义近重复信号。

### PA-P1-06：反馈 ACK 只返回窗口，没有消费闭环

`record_feedback()` 计算 `ack_until` 并写 Signal payload，但 Connector/Event/Candidate 查询没有使用 ACK 窗口抑制或重新评估。

### DR-P1-01：Admission 不是事务快照

Admission 连续执行多条 SELECT，没有显式一致性快照；期间 Turn、Task、Delivery、Outbox 状态可能变化。

### DR-P1-02：并发唯一性不成立

- `max_concurrent` 参数没有用于判定；
- `drift_runs` 没有 active partial unique index；
- Scheduler 的“分钟幂等键”只能减少重复，不能阻止不同进程竞态；
- 所谓并发测试实际是单连接串行 20 次，而且 admission 不创建 DriftRun。

### DR-P1-03：Admission 缺资源压力与真实恢复状态

resource pressure 未进入 Snapshot；recovery 只用 running Turn 近似；失败 Run 不计日预算，可能反复消耗资源。

### DR-P1-04：Skill 与宣称不一致

唯一内置 Skill `proactive-policy-view-audit` 只读取 DB Policy，没有读取或比较 Markdown 派生视图；计划中的 candidate quality、embedding scan、payload GC scan 均未实现。

### DR-P1-05：安全门禁只检查测试常量

安全测试没有调用真实 Policy/Registry/Executor；声明 `shell` 的 manifest 不会被 validator 拒绝，测试只确认字符串出现在测试内定义的禁止集合。

### DR-P1-06：Dashboard 指标包含占位值

- `admission_rate=None`；
- unauthorized count 固定 0；
- duplicate side effect 固定 0；
- model cost 固定 0；
- proactive feedback 的 ignored/dismissed/not_useful 等多项固定 0。

## 4. 冻结的修复原则

### 4.1 不再接受“helper 存在”作为完成证据

每个能力必须通过生产组合根和真实 Worker 路径证明：

```text
Config
→ Application wiring
→ Scheduler / Inbound
→ TaskDispatcher.claim_next
→ TaskWorker
→ Handler execution context
→ Repository / Payload / Outbox
→ Recovery / Projection / Delivery
```

直接调用内部 Service 的单元测试只能证明组件行为，不能关闭生产闭环 Issue。

### 4.2 Task 生命周期只有一个权威来源

`tasks/task_attempts` 是生命周期权威；`drift_runs.status` 是查询投影。禁止：

- DriftRun=paused 而 Task=completed；
- DriftRun=completed 而 Skill 未执行；
- 通过重新调用 completed Task 模拟 resume；
- 在 Drift 内创建第二套 Lease/Retry 状态机。

### 4.3 Checkpoint 必须可寻址、可校验、可恢复

Checkpoint Reference 必须指向真实持久 Payload，至少包含：

```text
schema_version
task_id
task_attempt_id
drift_run_id
skill_name / skill_version
step_index
cursor
completed_actions
pending_action
budget_used
config_version_id
capability_snapshot_version
created_at
```

缺失 Payload、版本不兼容或校验失败必须进入 `needs_review`，不能从 step 0 静默重跑。

### 4.4 用户可见结果仍走 Candidate

Drift Runner 不直接调用 DeliveryService。结果通过事务/Outbox 形成：

```text
DriftResultCommitted
→ DriftCandidateProjectionConsumer
→ ProactiveCandidate(origin=drift)
→ ProactivePolicy
→ Delivery / Digest / Silent
```

### 4.5 默认安全不依赖测试约定

安全约束必须在生产 Registry/Policy/Executor 强制执行；测试中的 forbidden set 不是安全边界。

## 6. R1：Skill 选择与 Admission 原子化

### 目标

确保每个 admitted DriftRun 在创建 Task 前已经选择并持久化一个真实 Skill。

### 修复方案

1. Scheduler 调用 `DriftSkillCatalog.resolve_catalog()`。
2. 从 `drift_skill_state` 读取状态。
3. 调用 `DriftSkillSelector.select_skill()`。
4. 校验：
   - Skill enabled；
   - manifest version；
   - config allow_workspace_skills；
   - risk level；
   - allowed tool scopes；
   - circuit/backoff；
   - daily/per-skill budget。
5. 在同一 `BEGIN IMMEDIATE` 事务内：
   - 读取 idle snapshot；
   - 检查 active unique guard；
   - 选择 Skill；
   - 插入 `tasks`；
   - 插入 `drift_runs`，保存真实 skill_name/version；
   - 保存 selection score/weights version；
   - commit。
6. 无 Skill 时持久化 admission decision=`deny/no_skill`，不创建 Task。
7. 删除 `"(selected-at-run)"` 占位。

## 7. R2：Task Handler 结构化结果与生命周期收口

### 新契约

```python
TaskHandlerOutcome(
    status: completed | paused | waiting | retry | failed,
    summary: str,
    checkpoint_ref: str | None,
    retry_at: datetime | None,
    waiting_condition: dict | None,
    result_ref: str | None,

```

### 任务

1. 将 `TaskHandler = Callable[..., str]` 演进为结构化 Outcome。
2. 为旧 Handler 提供临时兼容适配器：字符串只映射 completed，记录 deprecation warning。
3. TaskWorker 根据 Outcome 调用：

   - paused → `dispatcher.pause/requeue_with_checkpoint()`；

5. paused/waiting 必须有 checkpoint_ref，否则 Handler 结果非法并 fail-safe。
6. Drift status `needs_review` 映射 Task waiting，不映射 completed。

## 8. R3：Task Checkpoint Store 与真实 Resume

### 目标

持久保存真实 Checkpoint，并从它重建执行状态。

### Schema 方案

新增通用 Task Checkpoint 表；不复用只属于 Turn 的 `turn_checkpoints`：

```sql
CREATE TABLE task_checkpoints (
  checkpoint_id TEXT PRIMARY KEY,
  task_id TEXT NOT NULL REFERENCES tasks(task_id),
  task_attempt_id TEXT NOT NULL REFERENCES task_attempts(task_attempt_id),
  checkpoint_type TEXT NOT NULL,
  schema_version INTEGER NOT NULL,
  payload_ref TEXT NOT NULL,
  payload_hash TEXT NOT NULL,
  config_version_id TEXT,
  capability_snapshot_version TEXT,
  created_at INTEGER NOT NULL
);

CREATE INDEX idx_task_checkpoints_latest
ON task_checkpoints(task_id, created_at DESC);
```

Checkpoint JSON 写 Payload Store；小 Payload 是否允许数据库内联由 `DATABASE-SCHEMA` 统一决定。

### 任务

## 9. R4：入站抢占、Lease、Attempt 与预算生产接线

### 目标

让 Runner 每一步看到真实的运行状态，而不是默认值。

### 修复方案

```text

lease_expires_at

```

3. lease checker 使用 `task_id + attempt_id + worker_id + lease_version` 条件查询。
4. heartbeat 失败设置本地 lost flag；Runner 下一安全点立即停止。

## 11. R6：Proactive M1/M2 遗留收口

### R6.1 Presence fail-safe

区分三种状态：

```text
known_activity(timestamp)
never_seen
read_failed
```

- never_seen 可按低 energy 处理；
- read_failed 必须采用保守策略：medium/high energy、提高阈值或 digest-only；
- Decision trace 保存 presence_status。

### R6.2 Alert immediate evaluation

1. Alert SourceEvent 提交时通过 Outbox Consumer 创建幂等 `proactive.evaluate` Task。
2. 不等待 cadence，但仍受 deny、Quiet Hours、alert budget 和 Delivery Policy。
3. 与普通 cadence Task 使用不同幂等键。

### R6.3 Context fallback

1. 定义 `ContextSignal`，不是 SourceEvent，不要求 ACK event_id。
2. 仅在当前 batch 无 alert/content Candidate 时考虑。
3. 使用可复现概率阀和日配额。
4. 保存 trust、freshness、source_ref 和 reason trace。
5. Context 只生成 Candidate，不直接发送。

### R6.4 Model Enrichment

新增 `ProactiveEnrichmentService`，严格输出：

```text
summary
topic
novelty
relevance
urgency
confidence
evidence_refs
rationale
model_version
prompt_version
```

约束：

- 模型输出只提供信号；
- allow/deny、Quiet Hours、budget、Endpoint 仍由 Policy 决定；
- 非法输出/超时/低置信度回退规则评分；
- 外部内容保持 `external_untrusted` 边界；
- enrichment 默认关闭。

### R6.5 语义去重与 ACK 消费

1. Candidate 与近期主动消息产生 semantic_duplicate_score。
2. 分数进入 Decision trace，不代替唯一键。
3. ACK 窗口持久化为结构化记录。
4. Connector/Event/Candidate 查询在 ACK 有效期内抑制或降低重新评估优先级。
