### 3.2 工作项

#### 3.2.1 DST 确定策略

文件：`src/cogito/domain/schedule.py`

1. 在 `next_fire_at()` 中引入 `fold` 参数（Python 3.9+ `datetime` 支持）
2. 对 non-UTC timezone：
   - 先将 `after` 用 `ZoneInfo` 本地化
   - 若结果处于 DST gap（spring forward），按 `Policy.forward` 跳到 gap 后
   - 若结果处于 DST overlap（fall back），用 `fold=0/1` 显式选择并记录策略
3. 新增模块级常量：
   ```python
   DST_POLICY = "post"          # spring-forward: 跳到 gap 之后
   FOLD_POLICY = "earlier"      # fall-back: fold=0
   ```
4. 在 `Schedule` 中暴露 `dst_policy` 字段（默认 `"post"`），可在 `config.toml` 中覆盖

#### 3.2.2 Misfire 执行器

文件：`src/cogito/service/scheduler.py`

新增 `_handle_misfire(schedule, now,due_cutoff)` 方法：

```
输入：schedule（含 misfire_policy、max_catch_up、last_fired_at）
输出：list[Task]

逻辑：
  last = schedule.last_fired_at or schedule.created_at
  gap = now - last
  expected_interval = schedule.normalized_interval  # 解析 expression 后的最小周期
  missed_count = gap // expected_interval

  if missed_count <= 1:
      return [single_fire(schedule, now)]     # 正常触发

  match schedule.misfire_policy:
      skip:
          return [single_fire(schedule, now)]  # 只补一次当前
      run_once:
          return [single_fire(schedule, now)]  # 同 skip
      catch_up_limited:
          n = min(missed_count, schedule.max_catch_up)
          return [single_fire(schedule, last + i*interval) for i in range(1, n+1)]
      merge:
          return [merged_fire(schedule, now, count=missed_count)]
```

- `merged_fire` 创建一个 Task，payload 携带 `merged_count`、`first_missed_at`、`last_missed_at`
- 在 `connector_handler` / `rss_fetcher` 中解析 `merged_count`，决定单轮拉取窗口

#### 3.2.3 接入主 tick

修改 `Scheduler.tick()`：

```python
for schedule in due:
    if self._is_misfired(schedule, now):
        tasks = self._handle_misfire(schedule, now, due_cutoff=now)
    else:
        tasks = [self._process_schedule(schedule, now)]
```

`_is_misfired`：当 `now - last_fired_at > 1.5 * interval` 时判定为 misfire。

#### 3.2.4 幂等键保持

所有 misfire 产生的 fire 记录保持原幂等键 `schedule_id:epoch_ms(fire_at)`，确保崩溃后重 tick 不会重复。

### 4.2 目标契约

#### InboundMessage DTO（对齐 `LANGBOT-BRIDGE / 2`）

```python
@dataclass(frozen=True)
class InboundMessage:
    schema_version: str                  # 当前 "1"，前一版本 "0"
    event_id: str                        # 平台稳定 ID
    channel_name: str                    # "onebot" / "telegram" / ...
    instance_id: str                     # 具体 Instance
    conversation_ref: str                # 不透明稳定字符串
    thread_ref: str | None
    sender_ref: str                      # 不透明，Core 不解析内部
    content_parts: list[ContentPart]
    reply_route: ReplyRoute              # 快照
    trust_label: str                     # external_untrusted
    capability_ref: str                  # 引用，不内联
    raw_payload_ref: str                 # sha256 引用
    trace: TraceContext
    received_at: str                     # RFC3339
```

#### DeliveryOperation DTO

```python
@dataclass(frozen=True)
class DeliveryOperation:
    operation_id: str
    delivery_id: str
    attempt_id: str
    operation_seq: int
    idempotency_key: str
    target_snapshot: TargetSnapshot
    action: str                          # send / start_placeholder / append_or_replace / finish / delete / reconcile
    content: list[ContentPart]
    schema_version: str
```

- 热更新没有运行时激活入口
- Attempt / Task / Decision 未保存 `config_version`

### 5.2 工作项

#### 5.2.1 接入配置加载器

文件：`src/cogito/config.py`

1. 在执行 `read_toml()` 后调用 `normalize_config()`，计算：
   - `config_version`（语义版本，当顶层结构破坏性变化时递增）
   - `schema_version`（当前 configuration schema 版本，初始 "1"）
   - `content_hash`（排序 JSON 后 sha256 前 16 位）
   - `source_layers`（来源层级列表）
2. 将四个值写入 `RuntimeApplication` 的公开属性
3. 启动日志只输出 `config.content_hash`，不输出 Secret

#### 5.2.2 Secret 引用解析

1. `config.py` 新增 `resolve_secrets(config, env=None)`：
   - 扫描 `SENSITIVE_KEYS = {"api_key", "token", "secret", "access_token", "password"}`
   - 若值为 `"env://REF_NAME"` 形式，从 `os.environ` 读取
   - 若为纯字符串且长度 > 8，保留在内存但禁止进入 dump/trace/dashboard
2. `config dump` 命令输出时敏感字段显示为 `"***secret_ref:REF_NAME***"`
3. `__repr__` 和 `__str__` 自动脱敏

#### 5.2.3 热更新运行时激活

文件：`src/cogito/interaction_web/command_service.py`（或新建 `config_command.py`）

1. `POST /commands/config.update`：
   - 接受新 TOML payload
   - 调用 `hot_reload_dry_run` 校验
   - 通过则写入临时文件 → 原子替换 → 重载
   - 失败则保留旧版本 + Audit 记录
2. Command 执行时检查 `expected_version`（配置内容的条件更新）
3. 权限要求：仅 Owner Principal 可触发
4. 放宽安全约束只影响后续 Attempt，当前 Attempt 继续使用旧快照

#### 5.2.4 config_versions 审计表


- 每次启动自动插入一条
- `run_attempts.config_version_id`、`task_attempts.config_version_id`、`proactive_decisions.config_version_id` 外键引用

- `RunAttempt` 表新增 `config_version_id TEXT`

## 7. T5：Plan 06 M6 — Migration Framework 2.0

### 7.1 当前基线

已有：
- `store/migration.py`：发现 NNNN_*.sql 文件、顺序应用、记录 checksum
- 28 个编号 migration（0001-0028）
- 启动时自动运行未应用的 migration
- 迁移后 `PRAGMA foreign_key_check`

缺口：
- 无 `online_safe` / `requires_backup` / `estimated_space` / `preconditions` / `post_checks` 元数据
- 启动时**所有** migration 都自动运行，无维护模式门禁
- 无大型/破坏性迁移的 maintenance profile 要求
- 无分批回填 Checkpoint
- 无中断恢复测试

### 7.2 工作项

#### 7.2.1 Migration 元数据文件

每个 migration 增加同名 `.meta.toml`：

```toml
# migrations/0031_commands.meta.toml
online_safe = true
requires_backup = false
estimated_space = "10MB"
preconditions = []
post_checks = ["foreign_key_check", "integrity_check"]
rollback = "DROP TABLE commands"
checksum = "sha256:abc..."
```

修改 `migration.py`：

1. 解析 `.meta.toml`，缺失时按保守默认（`online_safe=false`）
2. 启动只自动运行 `online_safe=true` 的 migration
3. `online_safe=false` 的 migration 需要 `cogito run --profile maintenance` 显式触发
4. 执行前若 `requires_backup=true`，调用 `infrastructure/backup.py` 创建快照
5. 执行后运行 `post_checks` 列表中的检查

#### 7.2.2 Maintenance Profile

文件：`src/cogito/infrastructure/profile.py`

1. `maintenance` profile 默认只读，真实副作用需显式解锁
2. CLI：`cogito maintenance migrate` — 运行所有未应用的 non-online-safe migration
3. CLI：`cogito maintenance migrate --to 0035` — 运行到指定版本
4. 执行前自动 backup，失败自动回滚

#### 7.2.3 回填框架

文件：`src/cogito/store/backfill.py`

```python
class Backfill:
    """分批、可重入、有进度 Checkpoint 的回填器。"""
    def __init__(self, conn, batch_size=1000):
        ...

    def run(self, migration_id, transform_fn):
        """按主键分页执行 transform_fn(row)，每批写 Checkpoint。"""
        ...
```

- 中断后重新运行从最后 Checkpoint 继续
- 每批独立事务
- 进度写入 `_schema_version.backfill_progress`

#### 7.2.4 中断恢复

1. 每个 migration 执行前记录 `started_at`
2. 执行成功记录 `completed_at`
3. 执行失败记录 `error` 并回滚该 migration 的事务
4. 重启时：
   - 若 `started_at` 存在但 `completed_at` 不存在 → 标记为 `failed`，跳过
   - 若 `completed_at` 存在 → 跳过
5. 提供 `cogito migration status` 查看每个 migration 的状态

## 9. 风险与控制

| 风险 | 控制 |
|---|---|
| Schema 变更引入回归 | 严格 expand → backfill → switch → contract；每步有独立测试 |
| DST 策略在 Windows 表现不同 | 使用 `zoneinfo` 库（Python 3.9+），不依赖系统时区 |
| Bridge DTO 版本兼容破坏现有 OneBot 路径 | 保留旧 Adapter 并行运行，Bridge 默认 shadow 模式 |
| Config 热更新导致 Attempt 不一致 | 放宽只影响后续 Attempt，当前 Attempt 继续使用旧快照 |
| Migration 中断导致 Schema 不一致 | 每个 migration 独立事务 + started/completed 状态记录 |
| 回填耗时阻塞启动 | 回填在 maintenance profile 执行，默认启动只做 online-safe |
