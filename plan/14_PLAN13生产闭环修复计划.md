# PLAN-14：PLAN-13 生产闭环修复计划

## 1. 计划结论

PLAN-13 的 Schema、领域对象、Repository 和独立 Service 已基本落地，但生产运行时仍存在组件未接线、Task 占位、类型不匹配、事件缺失和控制面不可达等问题。本计划不重做 PLAN-13，而是完成从“组件存在”到“正常入口可用、跨重启可恢复、可解释可运维”的最后闭环。

执行顺序：

```text
R0 冻结缺口与修复边界
→ R1 durable memory.extract 唯一路径
→ R2 Signal / Weight / Consolidation / Memory Event
→ R3 Knowledge 生产装配、Task、Embedding 与删除级联
→ R4 Unified Retrieval + Context Budget + Snapshot provenance
→ R5 Config / Tool / Command / Query / Markdown / Audit
→ R6 Metrics / PoC / 文档状态收口
→ R7 完整验证、提交与发布
```

## 2. 冻结决策

1. `memory.extract` 只有 durable Task 路径；AgentRunner 不再 fire-and-forget 直接调用模型。
2. Turn 完成事务通过 Outbox/Task enqueue 触发提取；Task 成功后才 CAS 推进 watermark。
3. `MemoryExtractor` 只接收 `ExtractMessage`，生产边界负责 DTO 转换，禁止用未类型化 dict 静默兼容。
4. `MemoryService` 是 MemoryItem/MemorySource/MemoryRelation 唯一写入口；SignalWriter 是 MemorySignal 唯一写入口。
5. `KnowledgeService` 是 Knowledge 聚合唯一写入口；sync、invalidate、embedding 不直接在外部模块改表。
6. Knowledge 默认 feature flag 关闭；关闭时不得改变现有 Turn 行为。
7. Knowledge Embedding 复用现有 Embedding Port/Provider，不新增网络 Provider。
8. ContextBuilder 统一消费 `RetrievalCandidate`，但保留消息、摘要、Memory、Knowledge 的独立授权与召回器。
9. ContextSnapshot 保存选择结果，不复制无界正文；历史 Snapshot 不因源删除而回写。
10. memU PoC 保持拒绝集成结论，不引入 `reference/memU` 生产依赖。

## 4. R1：durable memory.extract 唯一路径

### 4.1 Turn 完成触发

- 在完成回复的同一事务或紧随其后的幂等服务中计算当前 Session extraction window。
- 使用 `processing_watermarks` 获取 `from_sequence`，以当前 `receive_sequence` 为 `to_sequence`。
- 调用 `ExtractionTriggerPolicy.should_trigger()`；未满足阈值不创建 Task。
- 创建 `memory.extract` Task，幂等键：`conversation_id + session_id + from + to + extractor_version`。
- explicit remember 仍由 Tool 直接写 Memory，但可附加高优先级 extraction Task 捕获上下文证据。
- Session reset/close 生成收尾 Task；不能直接跨 Session 拼消息。

### 4.2 Handler

- 删除候选计数 Stub。
- 从 DB 按稳定序列读取消息并构造 `ExtractMessage`。
- 构造 `SqliteMemoryService`、`WatermarkRepository`、`MemoryExtractor`。
- await 模型提取并写 MemoryItem/MemorySource。
- 只有整个写入成功后推进 watermark。
- 模型/DB 错误必须抛出，让 TaskWorker retry；禁止返回 `extracted (error)` 伪成功。
- CAS 失败视为幂等竞争：重新读取 watermark，已覆盖目标则成功，否则重试。

### 4.3 移除旧路径

- AgentRunner 删除 `_memory_extractor` 和 `_run_extraction()` fire-and-forget。
- Application 只向 TaskHandlerContext 注入模型、连接工厂与 MemoryService 工厂。
- 不保留第二条兼容写路径。

### 4.4 验收

- 正常 Turn 创建至多一个 extraction Task。
- Worker 重启后可继续处理。
- Handler 写入精确 Message Source。
- 失败不推进 watermark。
- explicit、turn_completed、session_closed 至少三类触发生效。

## 5. R2：Signal、权重、Consolidation 与 Memory Event

### 5.1 Signal 接线

- confirm-memory → `user_affirmed`。
- correct-memory → 新 Memory `user_corrected`。
- 成功 Task 显式声明依赖 Memory 时 → `task_succeeded`。
- ContextBuilder 选中 Memory → `exposed`，异步/批量写入，不阻塞 Turn。
- `recall_memory` 命中 → `exposed`，不强化。
- 用户负反馈入口 → `negative_feedback`。

### 5.2 权重维护

- 注册 `memory.recompute_weight` Task Handler。
- Scheduler 定期创建唯一重算 Task。
- Handler 调用版本化纯函数和 Signal 聚合，分批 checkpoint。
- `last_weight_update/algorithm_version` 必须更新。

### 5.3 Consolidation

- 删除旧 `_compute_retention_score` 决策路径。
- Consolidation 调用统一权重维护和候选过期策略。
- 低权重 confirmed 先退出默认检索/进入 review，不直接物理删除。
- erase 仅由显式 Command 执行并生成 Receipt。

### 5.4 Event/Outbox

至少发布：

```text
MemoryExtractionRequested
MemoryCandidateCreated
MemoryConfirmed
MemoryRejected
MemorySuperseded
MemorySourceInvalidated
MemorySignalRecorded
MemoryWeightRecomputed
```

事件写入与聚合变更在同一 UnitOfWork。

## 6. R3：Knowledge 生产闭环

### 6.1 配置与装配

- 类型化 `KnowledgeConfig`、`KnowledgeRetrievalConfig`。
- RuntimeApplication 根据 flag 构造 KnowledgeService/KnowledgeReader。
- TaskHandlerContext 注入 KnowledgeService 工厂和 Embedding Port。

### 6.2 Durable Task

注册：

```text
knowledge.ingest
knowledge.embed
knowledge.invalidate
knowledge.rebuild_index
knowledge.sync_source
```

首版把 parse/segment 合并在 ingest 内，避免无必要的细碎 Task；每步仍写版本和 checkpoint。

### 6.3 唯一写入口

- `knowledge.sync` 改为调用 KnowledgeService 的 sync/delete 方法。
- `embedding.invalidate_resource_segments` 下沉为 KnowledgeService 内部操作。
- 外部模块不直接 UPDATE knowledge 表。
- register + ingest 使用明确事务边界，崩溃后可由 Task 重试收敛。

### 6.4 Embedding

- `KnowledgeService.embed_pending()` 调用 Embedding Port。
- 写入 model/version/vector；按 content hash 幂等。
- query embedding + cosine 召回，与 FTS 合并。
- Provider 不可用时 FTS-only，记录 retrieval path。

### 6.5 删除传播

- invalid/delete KnowledgeResource 时撤销 Segment/FTS/Embedding。
- 查找关联 `MemorySource(source_type=knowledge_resource)`。
- 发布 `MemorySourceInvalidated`，由 MemoryService 根据剩余来源 keep/review/expire。
- erase 生成 Receipt 并清理 Payload；保留最小 tombstone。

### 6.6 Event

发布：ResourceDiscovered/Changed/Deleted、DocumentParsed、SegmentsIndexed、ResourceInvalidated。

## 7. R4：统一检索与 Snapshot

### 7.1 Source adapters

- RecentMessageRetriever
- SessionSummaryRetriever
- MemoryRetriever
- KnowledgeRetriever
- TaskStateRetriever

每个适配器输出 `RetrievalCandidate`，先执行 Principal/Scope/Trust/status/version 硬过滤。

### 7.2 ContextBuilder

- 使用 Query Plan 并行/顺序调用可用 source adapter。
- 每源标准化 score，不直接混加 raw BM25/cosine。
- 使用 `allocate_budget()` 分配 token。
- active constraint/goal、当前输入和 system policy 受保护。
- Knowledge flag 关闭或服务不可用时保持原行为。

### 7.3 Snapshot

- 为每条选择项写 candidate type/id、source refs/version、score 分项、tokens、path、policy version。
- 填充 `per_source_tokens`、`exclusion_stats`。
- ContextSnapshotRepository 持久化 provenance，不只保存在内存 DTO。

## 8. R5：配置和控制面

### 8.1 Config

实现并校验：

```toml
[memory.extraction]
[memory.weight]
[knowledge]
[knowledge.retrieval]
```

同步 `config.example.toml`；配置快照进入 Attempt/Task。

### 8.2 Tool

- 新增 `search_knowledge`，只读 KnowledgeReader Port。
- recall_memory 保持事实记忆职责。
- 工具返回安全来源摘要，不返回本地真实路径或 Secret。

### 8.3 Command

- reject-memory
- correct-memory
- erase-memory
- register/refresh/invalidate/erase-knowledge
- 所有变更支持 expected_version、Audit 和幂等 Command。

### 8.4 Query

- memory detail/sources/weight explain/pending list
- knowledge resource/document list
- retrieval snapshot explain
- QueryService 调用只读 Service/Port，不复制写逻辑。

### 8.5 Markdown

- Knowledge 变更后通过 Task/事件刷新 KNOWLEDGE.md。
- View 失败不回滚事实事务。
- freshness/version 可见。
