# PLAN-10：残余违规清零与可发布闭环开发计划

## 1. 计划结论

PLAN-09 已把 10 条架构例外降到 2 条（V7 `store→model`、C1 `channel→inbound→service` cycle），并清除了 CLI、runtime→store、interaction_web→store、tools↔service 等主要越界。本计划承接 PLAN-09，处理三块剩余工作：

1. **必须人工商酌**（本计划主体，完成后搁置等待人工确认）：残余架构例外清零 (V7/C1) + ADR 冻结 + Delivery/LangBot 真实闭环 + Plugin Runtime 产品化
2. **可并行机械推进**（可在 PLAN-10 启动后独立排期）：Legacy Channel Adapter 逐平台迁移 (M8)
3. **依赖外部资源**（需真实账号 / 浏览器 / 第三方服务）：QQ 人工验收、Web Playwright smoke、真实 Provider 冒烟

执行顺序：

```text
M0 冻结 ADR + 建立 Port/Facade 契约
→ M1 清零 V7 (store→model) 单端口注入
→ M2 清零 C1 (channel↔inbound↔service) Port 破环
→ M3 ADR-001/002 落到 docs/adr/，补 ADR-GUIDE 注册
→ M4 实现 SqliteDeliveryService + GatewayClient + Bridge E2E
→ M5 产品化 Plugin Runtime（持久化 / subprocess / 多源发现）
→ M6 Web CI + browser smoke + Provider/QQ 人工验收
→ M7+ 逐平台迁移 Legacy Channel Adapter（独立切片）
```

M0～M3 是 PLAN-09 的收尾；M4 是真正的可发布闭环前置；M5～M6 是 beta 发布门禁；M7+ 是扩展。

## 10. M4：SqliteDeliveryService + GatewayClient

目标：将 `service/delivery_service.py::DeliveryService` 从纯 Protocol 落地为 SQLite 完整实现；Gateway 侧有可替换的客户端协议。

### 10.1 SqliteDeliveryService

需要满足：

```python
class SqliteDeliveryService(DeliveryService):
    async def enqueue(request) -> DeliveryRef
    def get(delivery_id) -> DeliveryView | None
    async def cancel(delivery_id, expected_version) -> ...
    async def retry(delivery_id, expected_version) -> ...
    async def reconcile(delivery_id, evidence) -> ReconcileResult
```

约束：

- `idempotency_key` 唯一（UNIQUE 索引或前置检查）
- `TargetSnapshot` 创建后不可静默改变（copy-on-write 后 freeze）
- `retry` 只允许 `retry_scheduled` 状态（状态机守护）
- `unknown` 只允许 `reconcile` / `manual_review`
- 每次平台操作有 `Attempt` + `operation_seq`（单调递增）
- 所有平台结果形成 `Receipt`
- 旧 Lease/Attempt 不能提交（final state guard）
- 跨重启持久化；crash 后未确认 Attempt 恢复

### 10.2 GatewayClient Protocol

```python
class GatewayClient(Protocol):
    async def send(request) -> PlatformResult
    async def send_streaming(request, on_update) -> PlatformResult
    async def edit(request) -> PlatformResult
    async def finish(request) -> PlatformResult
    async def delete(request) -> PlatformResult
    async def reconcile(request) -> PlatformResult
    async def health() -> GatewayHealth
```

两种部署实现：
- `LoopbackGatewayClient`：合并进程部署，直接通过 `ChannelManager`+`ChannelAdapter` 调用（不走网络）
- `HttpGatewayClient`：独立 Gateway 进程，走 `core/gateway` 出站 HTTP

错误映射：timeout→unknown_result；rate_limited+retry_after；auth_error/route_expired/unsupported/too_large/temporary/permanent 明确定义。请求和日志不得含 Secret 或无限期平台 URL。

### 10.3 Bridge 升级

`channel/bridge_server.py::_handle_delivery()` 接入真实 `SqliteDeliveryService`；删除 "校验 DTO 后仅返回 accepted" 占位逻辑。幂等：重复 operation_seq 不产生重复平台副作用。

### 10.4 Schema 策略

优先复用 `delivery_attempts`、`delivery_receipts` 现有 operation_seq 字段。确需新字段走：

```text
expand migration → backfill → dual read/write → switch → contract migration
```

任何 Migration 必须标记 `online_safe`、`requires_backup`、rollback 限制。

## 15. 风险与控制

### 15.1 DTO/Protocol 过度抽象

控制：每个 use case 只注入实际依赖；Protocol 字段由调用方需求反向驱动；组合只发生在 application/deployment；禁止全局可变容器。

### 15.2 Delivery 重构产生重复副作用

控制：先持久化 intent，operation_seq + idempotency_key，平台结果写 Receipt；未知结果只 reconcile。

### 15.3 Gateway 分进程增加运维复杂度

控制：先提供 in-process 和 loopback HTTP 两个同 Protocol 实现；个人部署可合并进程，但不合并状态所有权。

### 15.4 Plugin Runtime 持久化引入 Schema 风险

控制：Plugin 状态表独立于 Core Schema；Plugin 数据迁移走独立 migration 目录（`store/migrations_plugins/` 已存在但空）。

### 15.5 Web CI 引入第三方工具链（Node / Playwright）

控制：CI 阶段 5+6 失败不阻塞阶段 1-4；本地 `npm run build` 仍可在无网络环境跳过。

### 15.6 Legacy Adapter 依赖污染

控制：冻结、可选依赖、逐平台迁移；未验收平台不进入 supported 列表。
